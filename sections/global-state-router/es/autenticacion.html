Autenticación y el Inicio del Prop Drilling

El inicio del problema

Imaginá que estás construyendo una web de empleo:

🏠 Home
🔍 Buscar trabajos
📄 Detalle de trabajo
👤 Usuario

Ahora aparece una nueva necesidad:

❓ ¿Cómo sabe la aplicación si el usuario ha iniciado sesión?

Objetivo real de esta clase

No estamos aprendiendo autenticación real todavía. Estamos simulando esto:

👤 Usuario NO logueado
        ↓
   [Iniciar sesión]
        ↓
👤 Usuario logueado

Y queremos que diferentes partes de la aplicación reaccionen automáticamente.

¿Dónde guardamos el estado de login?

Como toda la aplicación necesita saberlo, lo guardamos en el componente raíz:

// App.jsx
const [isLoggedIn, setIsLoggedIn] = useState(false)
isLoggedIn = false
     ↓
🚫 Usuario NO autenticado

isLoggedIn = true
     ↓
✅ Usuario autenticado

¿Quién necesita esta información?

Componente Necesita saber
Header¿Mostrar "Iniciar sesión" o "Cerrar sesión"?
JobDetailPage¿Puede el usuario aplicar al trabajo?
 
          App
           │
           │ isLoggedIn
           ▼
 ┌─────────────────┐
 │     Header      │
 └─────────────────┘

           │
           │ isLoggedIn
           ▼
 ┌─────────────────┐
 │ JobDetailPage   │
 └─────────────────┘

Simulación de Login y Logout

const login  = () => setIsLoggedIn(true)
const logout = () => setIsLoggedIn(false)
[Iniciar sesión]
       ↓
login()
       ↓
isLoggedIn = true
       ↓
✅ Usuario logueado

[Cerrar sesión]
       ↓
logout()
       ↓
isLoggedIn = false
       ↓
🚫 Usuario desconectado

Header funcionando

{isLoggedIn ? (
  <button onClick={onLogout}>
    Cerrar sesión
  </button>
) : (
  <button onClick={onLogin}>
    Iniciar sesión
  </button>
)}
NO logueado:
┌─────────────┐
│ Iniciar     │
│ sesión      │
└─────────────┘

Logueado:
┌─────────────┐
│ Cerrar      │
│ sesión      │
└─────────────┘

Página de detalle

<button disabled={!isLoggedIn}>
  {isLoggedIn
    ? 'Aplicar ahora'
    : 'Inicia sesión para aplicar'}
</button>
NO logueado:
Frontend Developer
🔒 Inicia sesión para aplicar

Logueado:
Frontend Developer
✅ Aplicar ahora

La aplicación crece — aparece el problema

Cuando organizás mejor el código, JobDetailPage se divide en componentes más pequeños:

JobDetailPage
│
├─ JobDetailHeader
├─ JobDetailInfo
├─ JobDetailSalary
└─ JobDetailApplyButton  ← este necesita isLoggedIn

Pero isLoggedIn está en App. La prop tiene que viajar por todos los niveles:

<JobDetailPage isLoggedIn={isLoggedIn} />
  ↓
<JobDetailHeader isLoggedIn={isLoggedIn} />
  ↓
<JobDetailApplyButton isLoggedIn={isLoggedIn} />
  ↓
if (isLoggedIn) ...

Esto es Prop Drilling

Prop Drilling = pasar props por componentes
que realmente no las necesitan

App
 │
 ▼
JobDetailPage     ❌ no la usa
 │
 ▼
JobDetailHeader   ❌ no la usa
 │
 ▼
JobDetailApplyButton  ✅ la usa

Los de arriba actúan como "tuberías".

Problema 1: Código difícil de mantener

// Si mañana añadís más datos globales...
<JobDetailPage
  user={user}
  isLoggedIn={isLoggedIn}
  theme={theme}
  language={language}
  permissions={permissions}
/>

😵 Empieza el caos

Problema 2: Componentes acoplados

function JobDetailHeader({ isLoggedIn }) {
  return (
    <JobDetailApplyButton isLoggedIn={isLoggedIn} />
  )
}

// ¿JobDetailHeader necesita isLoggedIn?
// ❌ No — solo la está reenviando

Problema 3: Renders innecesarios

Cuando cambia isLoggedIn, React vuelve a renderizar todos los componentes que reciben esa prop, aunque no la usen:

Al hacer login:

App
 ├─ Header      🔄
 ├─ Home        🔄
 ├─ Search      🔄
 ├─ JobDetail   🔄
 └─ Otros       🔄

💥 Toda la pantalla parpadea
= Muchos renders innecesarios

La lección real de esta clase

Tamaño Estructura Estado
Pequeña App → Header, JobDetail ✅ Props normales
Mediana App → Page → Header → Button ⚠️ Empieza el Prop Drilling
Grande App → … → … → … → Button 😱 Prop Drilling severo

La solución que aprenderás después

La solución moderna es usar un estado global con Zustand:

              Zustand
                 │
      ┌──────────┼──────────┐
      │          │          │
      ▼          ▼          ▼

   Header    JobDetail   Profile
// Cualquier componente puede leer directamente
const isLoggedIn = useAuthStore(
  state => state.isLoggedIn
)
❌ Prop Drilling
❌ Props infinitas
❌ Renders innecesarios

✅ Estado global
✅ Código más limpio
✅ Mejor rendimiento
La verdadera misión de esta clase no es la autenticación — es que veas por qué el Prop Drilling se vuelve un problema cuando una aplicación React empieza a crecer.
