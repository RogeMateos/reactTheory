React Context

Prop Drilling: el problema real

Imaginá que tenés un usuario logueado:

const [isLogin, setIsLogin] = useState(false)

Y el botón que lo necesita está muy abajo en el árbol:

App
 └── Header
      └── Navbar
           └── UserMenu
                └── UserButton  ← necesita isLogin

Pero el estado está en App. Sin Context tenés que hacer esto:

<App isLogin={isLogin}>
  ↓
<Header isLogin={isLogin}>
  ↓
<Navbar isLogin={isLogin}>
  ↓
<UserMenu isLogin={isLogin}>
  ↓
<UserButton isLogin={isLogin}>
Header    ❌ no lo usa
Navbar    ❌ no lo usa
UserMenu  ❌ no lo usa
UserButton ✅ lo usa

Todos tienen que recibirlo igual.

Esto se llama Prop Drilling — estás "taladrando" las props a través de componentes que no las necesitan.

Es como la diferencia entre un cable de red y el WiFi en tu casa.

createContext()

const AuthContext = createContext()

No guarda datos. No tiene estado. Solo crea un canal.

createContext()
      ↓
AuthContext
      ↓
Canal creado

Es como comprar un router.
Todavía no hay internet — necesitamos el Provider.

El Provider

El Provider es quien emite la información. Todo lo que esté dentro podrá acceder:

function AuthProvider({ children }) {
  const [isLogin, setIsLogin] = useState(false)

  function login()  { setIsLogin(true)  }
  function logout() { setIsLogin(false) }

  const value = { isLogin, login, logout }

  return (
    <AuthContext value={value}>
      {children}
    </AuthContext>
  )
}
┌───────────────────────┐
│ AuthProvider          │
│                       │
│ isLogin = false       │
│ login()               │
│ logout()              │
└───────────┬───────────┘
            │
            ▼
          value
            │
            ▼
    Disponible para toda la app

Envolver la aplicación

Normalmente en main.jsx:

<AuthProvider>
  <App />
</AuthProvider>
AuthProvider
      │
      ▼
     App
   /  |  \
Header Home Detail

Todos tienen acceso al contexto.

useContext()

Ahora cualquier componente puede conectarse directamente:

const { isLogin } = useContext(AuthContext)
Antes:

<App isLogin={isLogin} login={login} logout={logout} />
  ↓
<Header isLogin={isLogin} login={login} logout={logout} />
  ↓
<Nav isLogin={isLogin} login={login} logout={logout} />
  ↓
<UserButton isLogin={isLogin} login={login} logout={logout} />

Después:

<App />
<Header />
<Nav />
<UserButton />

// Y dentro de UserButton:
const { isLogin } = useContext(AuthContext)

⚠️ Optimización importante

Cuando cambia el Context, React re-renderiza todos los componentes que lo consumen. Por eso importa dónde usás useContext:

// ❌ Malo — Header se re-renderiza en cada cambio de login
function Header() {
  const { isLogin } = useContext(AuthContext)
  return (...)
}

// ✅ Mejor — solo UserButton se re-renderiza
function Header() {
  return (
    <>
      <Logo />
      <Nav />
      <UserButton />
    </>
  )
}

function UserButton() {
  const { isLogin } = useContext(AuthContext)
}
Cuando cambia login:

Regla para entrevistas

Usá props cuando Usá Context cuando
La información viaja pocos niveles (padre → hijo) Muchos componentes necesitan la misma información
El dato es específico de esa relación Usuario autenticado, tema, idioma, carrito, config global

😎 Acceso directo sin intermediarios

Los tres pasos

Paso Qué hace Código
1Crear el canalcreateContext()
2Emitir los datos<AuthContext value={value}>
3Consumir los datosuseContext(AuthContext)

Referencia rápida

// 1. Crear
const AuthContext = createContext()

// 2. Provider
function AuthProvider({ children }) {
  const [isLogin, setIsLogin] = useState(false)
  const value = {
    isLogin,
    login:  () => setIsLogin(true),
    logout: () => setIsLogin(false),
  }
  return (
    <AuthContext value={value}>{children}</AuthContext>
  )
}

// 3. Consumir
function UserButton() {
  const { isLogin, login, logout } = useContext(AuthContext)
}
React Context es una forma de crear un estado global para que cualquier componente pueda acceder a él sin tener que pasar props por toda la aplicación.
