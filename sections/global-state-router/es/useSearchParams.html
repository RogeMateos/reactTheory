useSearchParams

El hook que reemplaza todo lo manual

Antes hacías esto a mano:

window.location.search
new URLSearchParams(...)
navigate(`?${params.toString()}`)

Ahora React Router te da un hook específico:

const [searchParams, setSearchParams] = useSearchParams()

El flujo completo

Input / filtros
      ↓
Estado de React
      ↓
useSearchParams
      ↓
URL actualizada
      ↓
?text=react&technology=javascript

Leer valores de la URL

Dada esta URL:

/jobs?text=react&technology=javascript&remote=true

Podés leer cada valor así:

searchParams.get('text')        // "react"
searchParams.get('technology')  // "javascript"
searchParams.get('remote')      // "true"

Es como si la URL fuera un objeto:

{
  text: 'react',
  technology: 'javascript',
  remote: 'true'
}
useSearchParams es como un useState, pero para los parámetros de la URL.

Antes: manual y repetitivo

// ❌ Antes
const params = new URLSearchParams(window.location.search)
const text = params.get('text') ?? ''

navigate(`?${params.toString()}`)
Problema — vos hacés todo:

leer URL
  ↓
crear params
  ↓
modificar params
  ↓
navegar manualmente

Después: con React Router

// ✅ Después
const [searchParams, setSearchParams] = useSearchParams()

const text = searchParams.get('text') ?? ''
Mejor — React Router hace el trabajo pesado:

leer URL
  ↓
actualizar URL
  ↓
mantener navegación sincronizada

Lazy initialization explicado fácil

Esto funciona, pero la expresión se evalúa en cada render:

// ⚠️ Se evalúa en cada render
const [textToFilter, setTextToFilter] = useState(
  searchParams.get('text') ?? ''
)

Mejor con función:

// ✅ Solo se evalúa una vez
const [textToFilter, setTextToFilter] = useState(() => {
  return searchParams.get('text') ?? ''
})
Sin función:
render 1 → lee URL
render 2 → lee URL otra vez
render 3 → lee URL otra vez

Con función:
render 1 → lee URL
render 2 → NO vuelve a inicializar
render 3 → NO vuelve a inicializar

Actualizar filtros en la URL

const updateFiltersInURL = ({ text, technology, remote }) => {
  setSearchParams((prevParams) => {
    const params = new URLSearchParams(prevParams)

    if (text) params.set('text', text)
    else params.delete('text')

    if (technology) params.set('technology', technology)
    else params.delete('technology')

    if (remote) params.set('remote', 'true')
    else params.delete('remote')

    return params
  })
}
La lógica:

Si hay valor   →  params.set()
Si no hay valor →  params.delete()
text = "react"
technology = "javascript"
remote = true

Resultado:
/jobs?text=react&technology=javascript&remote=true

Si borrás text:
/jobs?technology=javascript&remote=true

⚠️ Error importante: append vs set

No uses append para filtros normales:

// ❌ Genera duplicados
params.append('text', text)
// ?text=react&text=react&text=react

// ✅ Reemplaza el valor anterior
params.set('text', text)
// ?text=react
Método Cuándo usarlo
params.set()Filtros normales — reemplaza el valor
params.append()Múltiples valores con la misma clave
params.delete()Quitar filtros vacíos de la URL

Lo que logramos con useSearchParams

useSearchParams te permite sincronizar los filtros de tu aplicación con la URL sin escribir código manual con window.location.search ni navigate.

Sirve para que esto funcione bien:

/jobs?text=react&technology=javascript

Y que al recargar la página, los filtros sigan activos.

Beneficio Resultado
Más limpioSin window.location.search manual
Más fácil de mantenerUn hook en lugar de lógica dispersa
Más profesionalURL como fuente de verdad de los filtros
Menos repetitivoSin boilerplate de URLSearchParams

import { useSearchParams } from 'react-router'

const [searchParams, setSearchParams] = useSearchParams()

// Leer
const text = searchParams.get('text') ?? ''

// Escribir
setSearchParams((prev) => {
  const params = new URLSearchParams(prev)
  if (text) params.set('text', text)
  else params.delete('text')
  return params
})
