1. useParams() → Obtener el ID

Cuando tenés una ruta como /jobs/:jobId, React Router captura el parámetro:

const { jobId } = useParams()

Si visitás /jobs/25, obtenés jobId = "25".

URL
│
└── /jobs/25
        │
        ▼
 useParams()
        │
        ▼
 jobId = 25

Es exactamente igual que leer ?id=25 desde una query string — pero usando parámetros de ruta limpios.

2. Los tres estados fundamentales

Esto es muy importante. Todo fetch async en React sigue el mismo patrón:

const [job, setJob] = useState(null)
const [loading, setLoading] = useState(true)
const [error, setError] = useState(null)
Estado Valor Qué significa Mostrar
Loading loading = true La API todavía no respondió <p>Cargando...</p>
Error error = "not found" La API respondió con error Mensaje de error
Success job = { title: ... } Datos recibidos correctamente Contenido del empleo

3. useEffect() para llamar a la API

Cuando cambia el jobId, React ejecuta el efecto:

useEffect(() => {
  // lógica del fetch
}, [jobId])
/jobs/1  →  /jobs/2  →  nuevo fetch disparado

4. AbortController — muy importante

Muchos juniors no entienden esto. Imaginá:

Usuario entra en /jobs/1

La petición tarda 3 segundos

Antes de terminar, el usuario navega a /jobs/2

❌ La petición vieja sigue viva y podría
   sobrescribir los datos de /jobs/2

La solución:

useEffect(() => {
  const controller = new AbortController()

  fetch(`/api/jobs/${jobId}`, { signal: controller.signal })
    .then(...)
    .catch((err) => {
      if (err.name === 'AbortError') return  // ignorar peticiones canceladas
    })

  return () => controller.abort()  // cleanup al desmontar o cambiar jobId
}, [jobId])
/jobs/1

 fetch
   │
   │
   ├──────────── X  cancelada (AbortController)
   │
Usuario navega a /jobs/2  →  nuevo fetch limpio

5. El error más común con fetch()

Muchísima gente piensa que fetch() lanza error automáticamente con 404 o 500. No lo hace.

// ❌ Esto NO lanza error con 404
fetch('/api/jobs/999')
  .then(res => res.json())  // ¡sigue ejecutándose aunque sea 404!

// ✅ Correcto
fetch('/api/jobs/999')
  .then(res => {
    if (!response.ok) {
      throw new Error('Job not found')
    }
    return res.json()
  })
Respuesta 404

response.ok = false

        ↓

throw Error()

        ↓

.catch()    ← recién acá lo manejamos

6. Guard Clauses

En vez de if/else anidado, React moderno usa guard clauses:

// ❌ Anidado (sucio)
if (loading) {
  return <p>Cargando...</p>
} else if (error) {
  return <p>Error: {error}</p>
} else {
  return <JobContent job={job} />
}

// ✅ Guard clauses (limpio)
if (loading) return <p>Cargando...</p>
if (error) return <p>Error: {error}</p>

return <JobContent job={job} />
¿Loading?

   ├─ Sí → retornar UI de loading
   │
   ▼ No

¿Error?
   │
   ├─ Sí → retornar UI de error
   │
   ▼ No

7. JobSection — componente reutilizable

Sin componente repetís la misma estructura para cada sección:

// ❌ Repetitivo
<section><h2>Descripción</h2> ... </section>
<section><h2>Requisitos</h2> ... </section>
<section><h2>Empresa</h2> ... </section>

// ✅ Reutilizable
<JobSection title="Descripción" content={job.content} />
<JobSection title="Requisitos" content={job.requirements} />
<JobSection title="Empresa" content={job.company} />

8. El problema del Markdown

Las APIs suelen devolver texto formateado en Markdown — no en HTML:

# React Developer

## Responsibilities

- Build apps
- Create tests

Si renderizás esto directamente en React, el usuario ve los símbolos # y guiones como texto plano. React no sabe qué es Markdown.

9. snarkdown

snarkdown es una librería tiny que convierte Markdown a HTML:

import snarkdown from 'snarkdown'

const html = snarkdown('# React Developer\n\n- React\n- TypeScript')

Resultado:

<h1>React Developer</h1>
<ul>
  <li>React</li>
  <li>TypeScript</li>
</ul>

     ▼
snarkdown()
     │
     ▼

10. dangerouslySetInnerHTML

Este es probablemente el concepto más importante de toda la clase.

Normalmente React escapa todo:

const html = '<h1>Hola</h1>'

// ❌ Renderiza como texto: <h1>Hola</h1>
return <div>{html}</div>

// ✅ Renderiza como un H1 real
return <div dangerouslySetInnerHTML={{ __html: html }} />
String HTML

      ↓

dangerouslySetInnerHTML={{ __html: html }}

      ↓

Elemento DOM real

¿Por qué se llama "dangerously"?

Por seguridad. Si contenido de usuarios llega a dangerouslySetInnerHTML sin filtrar:

// ❌ NUNCA hacer esto con input de usuarios
const comentario = '<script>stealPasswords()</script>'
return <div dangerouslySetInnerHTML={{ __html: comentario }} />

Eso es un ataque XSS (Cross-Site Scripting). El script se ejecutaría en el navegador de cada visitante.

Seguro ✅ No seguro ❌
Datos de tu propia API Comentarios de usuarios de un formulario público
Contenido creado por tu empresa Cualquier texto que cualquiera puede escribir
CMS controlado Fuentes externas no confiables
