Route Parameters with useParams

What's missing from our app

We have a working job listings page with React Router, but there's one critical piece missing: we can't navigate to the detail of a specific offer.

✅ We see a list of jobs
✅ We can navigate between some routes
❌ There is no detail page for each individual offer

We have the id of each job, but we're not using it in the URL to load the specific data for that offer.

In this lesson we will:

  • Create a detail page for an offer
  • Define a dynamic route that receives a parameter (the id)
  • Use the useParams hook from React Router to read that id from the URL

What are route parameters?

In a URL like /jobs/1234:

  • /jobs is the base route
  • 1234 is a dynamic parameter: the offer's id

React Router lets us define routes that aren't fixed but include variable segments we can read later:

/jobs/:id
URL visited Value of id
/jobs/abc "abc"
/jobs/42 "42"
/jobs/frontend-dev "frontend-dev"

The goal is to capture that value in order to:

  • Request the details from the API
  • Display the specific content for that offer

Why this matters

Imagine you find an offer you like. If the URL is /jobs/1234, you can:

  • Copy it and send it on WhatsApp
  • Bookmark it
  • Reload the page without losing context

If the information were only in React state (not in the URL), reloading the page would lose everything. This is what deep linking means — and it's a core feature of any professional web app.

Creating the detail page

First, we create a new component for the job detail.

// src/pages/JobDetail.jsx

export function JobDetail() {
  return <h1>Job detail</h1>
}

For now it just shows generic text. We'll improve it with the id and real data next.

Defining a dynamic route with :id

Now we create a route that receives a dynamic id. In the file where we define routes (App.jsx or router.jsx):

import { Route, Routes } from 'react-router'
import { JobDetail } from './pages/JobDetail'

export function App() {
  return (
    <Routes>
      {/* Other routes */}
      <Route path="/jobs/:id" element={<JobDetail />} />
    </Routes>
  )
}
Key point Explanation
path="/jobs/:id" :id signals that this URL segment is a dynamic parameter
element={<JobDetail />} When the URL matches /jobs/anything, JobDetail is rendered

Reading parameters with useParams

React Router gives us the useParams hook to read the id from the URL:

// src/pages/JobDetail.jsx
import { useParams } from 'react-router'

export function JobDetail() {
  const params = useParams()
  console.log(params)

  return (
    <>
      <h1>Job detail</h1>
      <pre>{JSON.stringify(params, null, 2)}</pre>
    </>
  )
}

If you visit /jobs/jajajata, you'll see in the console:

{ "id": "jajajata" }

Destructuring the parameter

In practice, we always destructure directly:

import { useParams } from 'react-router'

export function JobDetail() {
  const { id } = useParams()

  return (
    <>
      <h1>Job detail</h1>
      <h2>ID: {id}</h2>
    </>
  )
}

⚠️ The parameter name is what YOU define in the route

This is a very common mistake. The parameter name is NOT always id — it depends on what you put in the route path.

// Route defined with :jobId
<Route path="/jobs/:jobId" element={<JobDetail />} />
// Component MUST use jobId, not id
export function JobDetail() {
  const { jobId } = useParams()  // ✅ matches the route
  // const { id } = useParams()  // ❌ would be undefined
}

The parameter name must match exactly between the path and useParams().

Using the id to fetch real data

The natural next step is using the id to request real data from an API:

import { useEffect, useState } from 'react'
import { useParams } from 'react-router'

export function JobDetail() {
  const { id } = useParams()
  const [job, setJob] = useState(null)

  useEffect(() => {
    if (!id) return

    fetch(`/api/jobs/${id}`)
      .then((res) => res.json())
      .then((data) => setJob(data))
      .catch((error) => console.error('Error loading job', error))
  }, [id])

  if (!job) return <p>Loading offer...</p>

  return (
    <section>
      <h1>{job.title}</h1>
      <p>{job.description}</p>
      <p>Location: {job.location}</p>
    </section>
  )
}
Key concept: The id comes from the URL, not from props. This makes the component independent — if the user reloads the page, the id is still in the URL.

Multiple parameters in the URL

React Router supports more than one parameter in a route:

<Route path="/jobs/:id/page/:page" element={<JobDetail />} />
import { useParams } from 'react-router'

export function JobDetail() {
  const { id, page } = useParams()

  return (
    <>
      <h1>Job detail</h1>
      <p>ID: {id}</p>
      <p>Page: {page}</p>
    </>
  )
}

Visiting /jobs/1234/page/2 gives:

{ "id": "1234", "page": "2" }

Route params vs Query strings

Use route params when… Use query strings when…
The value identifies a specific resource They are optional filters
The page doesn't make sense without it The page works without them
You want clean, semantic URLs You have many optional parameters
/jobs/1234 /jobs?keyword=react&page=2

Converting the id to a number

useParams() always returns strings. If you need a number:

export function JobDetail() {
  const { id } = useParams()
  const numericId = Number(id)
  const jobId = parseInt(id, 10)  // alternative

  console.log(typeof id)        // "string"
  console.log(typeof numericId) // "number"
}

Validating parameters

It's good practice to validate that the parameter has the expected format:

import { useParams, Navigate } from 'react-router'

export function JobDetail() {
  const { id } = useParams()

  // If id is not a number, redirect to 404
  if (isNaN(Number(id))) {
    return <Navigate to="/404" replace />
  }

  // Rest of the component...
}

Or show an error message inline:

export function JobDetail() {
  const { id } = useParams()

  if (!id || isNaN(Number(id))) {
    return (
      <div>
        <h1>Error</h1>
        <p>Invalid offer ID</p>
      </div>
    )
  }

  // Rest of the component...
}

What we covered

Concept Key point
Dynamic routes Use :paramName in the path to define a variable segment
useParams() Returns an object with all parameters from the current URL
Parameter names Must match exactly between the route path and useParams()
Values are always strings Convert with Number() or parseInt() if needed
Deep linking URL params make apps shareable and reload-safe
Multiple params /jobs/:id/page/:page → { id, page }

The mental model

Route defined as:   /jobs/:id
URL visited:        /jobs/1234
useParams() gives:  { id: "1234" }

Route defined as:   /jobs/:jobId
URL visited:        /jobs/1234
useParams() gives:  { jobId: "1234" }
