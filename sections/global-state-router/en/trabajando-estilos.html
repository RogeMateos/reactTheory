1. useParams() — Get the ID

When you have a route like /jobs/:jobId, React Router captures the parameter:

const { jobId } = useParams()

If you visit /jobs/25, you get jobId = "25".

URL
│
└── /jobs/25
        │
        ▼
 useParams()
        │
        ▼
 jobId = 25

It's exactly the same idea as reading ?id=25 from a query string — but using clean route params instead.

2. The three fundamental states

This is very important. Every async fetch in React follows the same pattern:

const [job, setJob] = useState(null)
const [loading, setLoading] = useState(true)
const [error, setError] = useState(null)
State Value What it means Show
Loading loading = true API hasn't responded yet <p>Loading...</p>
Error error = "not found" API responded with an error Error message
Success job = { title: ... } Data received correctly Job content

3. useEffect() to call the API

When jobId changes, React runs the effect:

useEffect(() => {
  // fetch logic here
}, [jobId])
/jobs/1  →  /jobs/2  →  new fetch triggered

4. AbortController — very important

Many junior devs skip this. Imagine:

User visits /jobs/1

fetch takes 3 seconds

Before it finishes, user navigates to /jobs/2

❌ The old fetch is still alive and could
   overwrite the data for /jobs/2

The fix:

useEffect(() => {
  const controller = new AbortController()

  fetch(`/api/jobs/${jobId}`, { signal: controller.signal })
    .then(...)
    .catch((err) => {
      if (err.name === 'AbortError') return  // ignore cancelled requests
    })

  return () => controller.abort()  // cleanup on unmount or jobId change
}, [jobId])
/jobs/1

 fetch
   │
   │
   ├──────────── X  cancelled (AbortController)
   │
User navigates to /jobs/2  → clean new fetch

5. The most common fetch() mistake

A lot of people think fetch() throws automatically on 404 or 500. It does not.

// ❌ This will NOT throw on 404
fetch('/api/jobs/999')
  .then(res => res.json())  // still runs even on 404!

// ✅ Correct
fetch('/api/jobs/999')
  .then(res => {
    if (!response.ok) {
      throw new Error('Job not found')
    }
    return res.json()
  })
404 response

response.ok = false

        ↓

throw Error()

        ↓

.catch()    ← now we handle it

6. Guard Clauses

Instead of nested if/else, modern React uses guard clauses:

// ❌ Nested (messy)
if (loading) {
  return <p>Loading...</p>
} else if (error) {
  return <p>Error: {error}</p>
} else {
  return <JobContent job={job} />
}

// ✅ Guard clauses (clean)
if (loading) return <p>Loading...</p>
if (error) return <p>Error: {error}</p>

return <JobContent job={job} />
Is loading?
   │
   ├─ Yes → return loading UI
   │
   ▼ No

Is error?
   │
   ├─ Yes → return error UI
   │
   ▼ No

Return content

7. JobSection — reusable component

Without a component you repeat the same structure for every section:

// ❌ Repetitive
<section><h2>Description</h2> ... </section>
<section><h2>Requirements</h2> ... </section>
<section><h2>Company</h2> ... </section>

// ✅ Reusable
<JobSection title="Description" content={job.content} />
<JobSection title="Requirements" content={job.requirements} />
<JobSection title="Company" content={job.company} />

8. The Markdown problem

APIs often return text formatted in Markdown — not HTML:

# React Developer

## Responsibilities

- Build apps
- Create tests

If you render this directly in React, the user sees the raw # symbols and dashes as plain text. React doesn't know what Markdown is.

9. snarkdown

snarkdown is a tiny library that converts Markdown to HTML:

import snarkdown from 'snarkdown'

const html = snarkdown('# React Developer\n\n- React\n- TypeScript')

Output:

<h1>React Developer</h1>
<ul>
  <li>React</li>
  <li>TypeScript</li>
</ul>
Markdown
     │
     ▼
snarkdown()
     │
     ▼
HTML string

10. dangerouslySetInnerHTML

This is probably the most important concept in this lesson.

Normally React escapes everything:

const html = '<h1>Hello</h1>'

// ❌ This renders as text: <h1>Hello</h1>
return <div>{html}</div>

// ✅ This renders as an actual H1
return <div dangerouslySetInnerHTML={{ __html: html }} />
HTML string

      ↓

dangerouslySetInnerHTML={{ __html: html }}

      ↓

Real DOM element

Why is it called "dangerously"?

Because of security. If user-provided content reaches dangerouslySetInnerHTML unfiltered:

// ❌ NEVER do this with user input
const userComment = '<script>stealPasswords()</script>'
return <div dangerouslySetInnerHTML={{ __html: userComment }} />

That's an XSS (Cross-Site Scripting) attack. The script would execute in every visitor's browser.

Safe ✅ Not safe ❌
Data from your own API User comments from a public form
Content created by your company Any text that anyone can write
Controlled CMS content Untrusted external sources
