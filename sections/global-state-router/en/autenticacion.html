Authentication and the Start of Prop Drilling

The start of the problem

Imagine you're building a job board web app:

🏠 Home
🔍 Search jobs
📄 Job detail
👤 User

A new requirement appears:

❓ How does the app know if the user is logged in?

The real goal of this lesson

We're not learning real authentication yet. We're simulating this:

👤 User NOT logged in
        ↓
   [Sign in]
        ↓
👤 User logged in

And we want different parts of the app to react automatically.

Where do we store the login state?

Since the whole app needs to know, we store it at the root component:

// App.jsx
const [isLoggedIn, setIsLoggedIn] = useState(false)
isLoggedIn = false
     ↓
🚫 User NOT authenticated

isLoggedIn = true
     ↓
✅ User authenticated

Who needs this information?

Component Needs to know
HeaderShow "Sign in" or "Sign out"?
JobDetailPageCan the user apply to the job?
 
          App
           │
           │ isLoggedIn
           ▼
 ┌─────────────────┐
 │     Header      │
 └─────────────────┘

           │
           │ isLoggedIn
           ▼
 ┌─────────────────┐
 │ JobDetailPage   │
 └─────────────────┘

Simulating Login and Logout

const login  = () => setIsLoggedIn(true)
const logout = () => setIsLoggedIn(false)
[Sign in]
       ↓
login()
       ↓
isLoggedIn = true
       ↓
✅ User logged in

[Sign out]
       ↓
logout()
       ↓
isLoggedIn = false
       ↓
🚫 User signed out

Header in action

{isLoggedIn ? (
  <button onClick={onLogout}>
    Sign out
  </button>
) : (
  <button onClick={onLogin}>
    Sign in
  </button>
)}
NOT logged in:
┌─────────────┐
│ Sign in     │
└─────────────┘

Logged in:
┌─────────────┐
│ Sign out    │
└─────────────┘

Detail page

<button disabled={!isLoggedIn}>
  {isLoggedIn
    ? 'Apply now'
    : 'Sign in to apply'}
</button>
NOT logged in:
Frontend Developer
🔒 Sign in to apply

Logged in:
Frontend Developer
✅ Apply now

The app grows — the problem appears

When you organise the code better, JobDetailPage splits into smaller components:

JobDetailPage
│
├─ JobDetailHeader
├─ JobDetailInfo
├─ JobDetailSalary
└─ JobDetailApplyButton  ← this one needs isLoggedIn

But isLoggedIn lives in App. The prop has to travel down every level:

<JobDetailPage isLoggedIn={isLoggedIn} />
  ↓
<JobDetailHeader isLoggedIn={isLoggedIn} />
  ↓
<JobDetailApplyButton isLoggedIn={isLoggedIn} />
  ↓
if (isLoggedIn) ...

This is Prop Drilling

Prop Drilling = passing props through components
that don't actually need them

App
 │
 ▼
JobDetailPage     ❌ doesn't use it
 │
 ▼
JobDetailHeader   ❌ doesn't use it
 │
 ▼
JobDetailApplyButton  ✅ uses it

The ones above act as "pipes".

Problem 1: Hard to maintain

// If tomorrow you add more global data...
<JobDetailPage
  user={user}
  isLoggedIn={isLoggedIn}
  theme={theme}
  language={language}
  permissions={permissions}
/>

😵 Chaos begins

Problem 2: Coupled components

function JobDetailHeader({ isLoggedIn }) {
  return (
    <JobDetailApplyButton isLoggedIn={isLoggedIn} />
  )
}

// Does JobDetailHeader need isLoggedIn?
// ❌ No — it's just forwarding it

Problem 3: Unnecessary renders

When isLoggedIn changes, React re-renders every component that receives that prop, even if they don't use it:

On login:

App
 ├─ Header      🔄
 ├─ Home        🔄
 ├─ Search      🔄
 ├─ JobDetail   🔄
 └─ Others      🔄

💥 The whole screen flickers
= Many unnecessary renders

The real lesson of this class

App size Structure Status
Small App → Header, JobDetail ✅ Normal props
Medium App → Page → Header → Button ⚠️ Prop Drilling starts
Large App → … → … → … → Button 😱 Severe Prop Drilling

The solution you'll learn next

The modern solution is global state with Zustand:

              Zustand
                 │
      ┌──────────┼──────────┐
      │          │          │
      ▼          ▼          ▼

   Header    JobDetail   Profile
// Any component can read directly
const isLoggedIn = useAuthStore(
  state => state.isLoggedIn
)
❌ Prop Drilling
❌ Infinite props
❌ Unnecessary renders

✅ Global state
✅ Cleaner code
✅ Better performance
The real mission of this lesson is not authentication — it's showing you why Prop Drilling becomes a problem as a React app grows.
