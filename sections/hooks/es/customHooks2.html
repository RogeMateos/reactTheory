Custom Hooks II: Creando un hook para un formulario

😤 El problema: un componente con demasiadas responsabilidades

Tenemos un componente principal que gestiona toda la lógica de búsqueda, y eso lo hace demasiado complejo. El componente tiene mezclado:

  • handleSubmit
  • handleChange / handleTextChange
  • La lógica de búsqueda (search)
  • IDs como technology, location, experience, etc.

Todo junto hace que el componente sea difícil de leer y mantener. Vamos a separar responsabilidades usando un custom hook.

Señal de alerta: cuando un componente tiene demasiadas responsabilidades, es señal de que necesitamos refactorizar.

✨ La solución: useSearchForm

Vamos a crear un custom hook llamado useSearchForm que encapsule toda la lógica del formulario. A diferencia de hooks como useRouter (visto antes), los custom hooks pueden recibir parámetros igual que funciones normales.

Parámetros que necesita el hook

  • onSearch: función que se ejecuta al buscar
  • onTextFilter: función para filtrar por texto
  • IDs necesarios para los filtros:
    • IDTechnology
    • IDLocation
    • IDExperienceLevel

🎯 Ventajas de la refactorización

  • ✅ Código más limpio: el componente se enfoca en renderizar
  • ✅ Lógica reutilizable: el hook puede usarse en otros componentes
  • ✅ Más fácil de testear: podemos probar el hook independientemente
  • ✅ Mejor organización: separación clara de responsabilidades

Paso 1 — Esqueleto del hook con parámetros

function useSearchForm({
  onSearch,
  onTextFilter,
  IDTechnology,
  IDLocation,
  IDExperienceLevel
}) {
  // Lógica del hook aquí
}

Paso 2 — Añadir estado interno

El texto de búsqueda vive dentro del hook:

function useSearchForm({ onSearch, onTextFilter, IDTechnology, IDLocation, IDExperienceLevel }) {
  const [searchText, setSearchText] = useState('')

  // Este estado:
  // - guarda el texto introducido por el usuario
  // - se actualiza en handleChange
  // - se devuelve al componente para mostrarlo en el input
}

Paso 3 — Implementar los handlers

function useSearchForm({ onSearch, onTextFilter, IDTechnology, IDLocation, IDExperienceLevel }) {
  const [searchText, setSearchText] = useState('')

  const handleSubmit = (event) => {
    event.preventDefault()

    const formData = new FormData(event.target)

    const filters = {
      text: formData.get('search'),
      technology: formData.get('technology'),
      location: formData.get('location'),
      experience: formData.get('experience'),
    }

    onSearch(filters)
  }

  const handleChange = (event) => {
    const text = event.target.value
    setSearchText(text)
    onTextFilter(text)
  }

  // Devolvemos lo que el componente necesita
  return {
    searchText,
    handleSubmit,
    handleChange,
  }
}

Paso 4 — Usar el hook en el componente

El componente queda muchísimo más simple:

function SearchComponent() {
  const { searchText, handleSubmit, handleChange } = useSearchForm({
    onSearch: performSearch,
    onTextFilter: filterByText,
    IDTechnology: 'tech',
    IDLocation: 'loc',
    IDExperienceLevel: 'exp',
  })

  return (
    <form onSubmit={handleSubmit}>
      <input value={searchText} onChange={handleChange} name="search" />
      {/* Resto del formulario */}
    </form>
  )
}

Resultado: el componente ya no implementa la lógica, solo consume las funciones y valores del hook.

📏 Reglas importantes de los custom hooks

1. Llamar solo en el nivel superior

Los custom hooks deben llamarse solo en el nivel superior del componente o de otro custom hook.

❌ NO dentro de:

  • Callbacks (handleClick, handleSubmit, etc.)
  • Condicionales (if)
  • Loops (for, while)
  • useEffect ni otros hooks
// ❌ MAL - dentro de un callback
function Component() {
  const handleClick = () => {
    const { data } = useSearchForm() // ERROR
  }
}

// ❌ MAL - dentro de un condicional
function Component() {
  if (condition) {
    const { data } = useSearchForm() // ERROR
  }
}

// ✅ BIEN - nivel superior
function Component() {
  const { searchText, handleSubmit } = useSearchForm()

  const handleClick = () => {
    // Usar los valores del hook aquí está bien
    console.log(searchText)
  }
}

2. Deben empezar con el prefijo use

Como useState, useEffect, etc. No es solo convención: React confía en ese nombre para validar reglas internas. 

// ✅ BIEN
function useSearchForm() {}
function useUserData() {}
function useAuthentication() {}

// ❌ MAL
function searchFormHook() {}  // No empieza con "use"
function getSearchForm() {}   // No es un hook

3. Solo se pueden usar en componentes o custom hooks

// ❌ MAL - función normal
function calculateTotal() {
  const { data } = useData() // ERROR
  return data.total
}

// ✅ BIEN - componente
function Component() {
  const { data } = useData()
  return <div>{data.total}</div>
}

// ✅ BIEN - otro custom hook
function useCalculateTotal() {
  const { data } = useData()
  return data.total
}

🧪 Ejercicios

Ejercicio 1

Agregá al hook un handleReset que limpie searchText y llame a onTextFilter(''). Exponelo en el return.

Ejercicio 2

Creá otro hook useDebouncedText(value, delay) que devuelva el valor retrasado. Luego usalo dentro de useSearchForm para disparar onTextFilter solo cuando el usuario deja de tipear por 300ms.

Ejercicio 3 — Desafío

Refactorizá useSearchForm para que no dependa de callbacks (onSearch, onTextFilter). En su lugar, que exponga filters y text y que sea el componente quien decida qué hacer. ¿Qué ventajas e inconvenientes tiene cada enfoque?

🔑 Conclusión

Extraer la lógica a custom hooks es una práctica esencial en React moderno:

  • El componente se enfoca en renderizar
  • La lógica vive en un sitio reutilizable
  • El desarrollo se vuelve más escalable

Esta técnica mantiene tu código ✅ limpio, ✅ modular, ✅ fácil de testear y ✅ mantenible a largo plazo.
