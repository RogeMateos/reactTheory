Hook useEffect() — Parte III

🚫 Cuándo NO usar useEffect

La regla central: si la operación no toca nada fuera del componente, no necesitás useEffect. Tres anti-patrones muy comunes: 

¿Necesito useEffect?  Preguntate:
  ¿Estoy transformando datos que ya tengo?   → NO, hacelo directo
  ¿Estoy calculando un valor derivado?       → NO, calculalo en el render
  ¿Solo quiero inicializar un estado?        → NO, usá el valor inicial de useState
  ¿Necesito comunicarme con algo externo?    → SÍ, useEffect es correcto

💡 La regla de oro: si podés hacerlo sin useEffect, no uses useEffect. Cada useEffect innecesario agrega complejidad y posibles bugs.

✅ Cuándo SÍ usar useEffect

useEffect es correcto cuando:
  🌐 Hacés fetching a APIs externas
  📝 Modificás el DOM directamente (document.title, focus)
  🔔 Suscribís a eventos del navegador (resize, scroll, click)
  ⏱️ Creás timers (setInterval, setTimeout)
  📡 Abrís conexiones persistentes (WebSockets)

🧹 La función de limpieza (cleanup)

El problema: si un componente se suscribe a un evento o inicia un timer, y después el componente se desmonta (desaparece de la pantalla), la suscripción o el timer siguen corriendo en segundo plano. Eso es un memory leak — recursos que consumen memoria sin hacer nada útil.

La función de limpieza (return () => { ... } dentro de useEffect) se ejecuta en dos momentos:

  1. Cuando el componente se desmonta
  2. Antes de volver a ejecutar el efecto (cuando cambian las dependencias)
Montar componente
  → useEffect corre → addEventListener / setInterval / WebSocket

Desmontar componente (o dep cambia)
  → cleanup corre → removeEventListener / clearInterval / ws.close()
  → (si dep cambió) → useEffect corre con nuevos valores

⚠️ Atención: si creás un event listener, interval o WebSocket dentro de useEffect y NO limpiás, estás creando un memory leak. El navegador sigue ejecutando código para un componente que ya no existe.

📦 Un efecto, una responsabilidad

No metas todo en un único useEffect gigante. Cada efecto debe hacer una sola cosa. ¿Por qué?

  • Es más fácil de leer y entender
  • Cada efecto tiene sus propias dependencias — sin interacciones raras
  • Las limpiezas no se mezclan
  • Podés habilitar o deshabilitar efectos de forma independiente
// Un componente puede tener múltiples useEffect:
useEffect(() => { /* cargar datos del usuario */ }, [userId])
useEffect(() => { /* suscribirse al resize */ }, [])
useEffect(() => { /* actualizar el título */ }, [user])

🔮 Alternativas modernas al fetching con useEffect

Mencionamos brevemente las alternativas que existen en proyectos reales:

  • React Server Components (Next.js 13+): podés hacer fetch directo en el componente, sin useEffect.
  • React 19 use hook: lee promesas directamente durante el render.
  • TanStack Query / SWR: manejan caché, estados de carga, manejo de errores y refetch — casi no necesitás useEffect para fetching de datos.

💡 Dato profesional: para fetching de datos en apps reales, TanStack Query o SWR son la elección profesional. useEffect para fetching es educativo — te enseña los fundamentos, pero en producción hay mejores herramientas.

❌ Anti-patrón: transformar datos con useEffect

// ❌ MAL: useEffect innecesario para transformar datos
function ProductList({ products }) {
  const [expensiveProducts, setExpensiveProducts] = useState([])

  useEffect(() => {
    const filtered = products.filter((p) => p.price > 100)
    setExpensiveProducts(filtered)
  }, [products])

  return <div>{expensiveProducts.map((p) => p.name)}</div>
}

// ✅ BIEN: cálculo directo en el render
function ProductList({ products }) {
  const expensiveProducts = products.filter((p) => p.price > 100)

  return <div>{expensiveProducts.map((p) => p.name)}</div>
}

La primera versión hace 2 renders (inicial + después del efecto). La segunda hace 1. Más simple y más rápido.

✅ Caso 1: Fetching de datos

import { useState, useEffect } from 'react'

function UserProfile({ userId }) {
  const [user, setUser] = useState(null)
  const [loading, setLoading] = useState(true)
  const [error, setError] = useState(null)

  useEffect(() => {
    setLoading(true)
    setError(null)

    fetch(`https://api.example.com/users/${userId}`)
      .then((res) => {
        if (!res.ok) throw new Error('Error al cargar usuario')
        return res.json()
      })
      .then((data) => {
        setUser(data)
        setLoading(false)
      })
      .catch((err) => {
        setError(err.message)
        setLoading(false)
      })
  }, [userId])

  if (loading) return <div>Cargando...</div>
  if (error) return <div>Error: {error}</div>

  return (
    <div>
      <h2>{user?.name}</h2>
      <p>{user?.email}</p>
    </div>
  )
}

✅ Caso 2: Suscripción a eventos con limpieza

import { useState, useEffect } from 'react'

function WindowSize() {
  const [windowWidth, setWindowWidth] = useState(window.innerWidth)

  useEffect(() => {
    const handleResize = () => {
      setWindowWidth(window.innerWidth)
    }

    // Suscribirse
    window.addEventListener('resize', handleResize)

    // Limpieza: desuscribirse al desmontar
    return () => {
      window.removeEventListener('resize', handleResize)
    }
  }, []) // [] = solo al montar / desmontar

  return <div>Ancho: {windowWidth}px</div>
}

✅ Caso 3: Timer con limpieza

function LiveCounter() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    const interval = setInterval(() => {
      setCount((prev) => prev + 1)
    }, 1000)

    // Limpieza: detener el interval al desmontar
    return () => clearInterval(interval)
  }, [])

  return <div>Contador: {count}</div>
}

✅ Caso 4: Cleanup cuando cambian dependencias

function Timer({ delay }) {
  const [count, setCount] = useState(0)

  useEffect(() => {
    console.log(`Configurando timer: ${delay}ms`)
    const interval = setInterval(() => {
      setCount((prev) => prev + 1)
    }, delay)

    return () => {
      console.log(`Limpiando timer: ${delay}ms`)
      clearInterval(interval) // limpia el anterior antes de crear el nuevo
    }
  }, [delay]) // cuando delay cambia: cleanup → nuevo efecto

  return <div>Contador cada {delay}ms: {count}</div>
}

💡 Tip: cuando delay cambia: (1) corre el cleanup del efecto anterior → (2) corre el efecto nuevo. El componente nunca tiene dos intervals al mismo tiempo.

✅ Caso 5: Múltiples efectos — un efecto, una responsabilidad

function UserDashboard({ userId }) {
  const [user, setUser] = useState(null)
  const [windowWidth, setWindowWidth] = useState(window.innerWidth)

  // Efecto 1: cargar datos del usuario
  useEffect(() => {
    fetch(`/api/users/${userId}`)
      .then((res) => res.json())
      .then(setUser)
  }, [userId])

  // Efecto 2: suscribirse al resize
  useEffect(() => {
    const handleResize = () => setWindowWidth(window.innerWidth)
    window.addEventListener('resize', handleResize)
    return () => window.removeEventListener('resize', handleResize)
  }, [])

  // Efecto 3: actualizar el título
  useEffect(() => {
    if (user) document.title = `Dashboard de ${user.name}`
  }, [user])

  return (
    <div>
      <h1>{user?.name}</h1>
      <p>Ancho: {windowWidth}px</p>
    </div>
  )
}

✅ Caso 6: Debounce con cleanup

function SearchWithDebounce() {
  const [query, setQuery] = useState('')
  const [results, setResults] = useState([])

  useEffect(() => {
    if (query === '') {
      setResults([])
      return
    }

    // Esperar 300ms antes de hacer el fetch
    const timeoutId = setTimeout(() => {
      fetch(`/api/search?q=${query}`)
        .then((res) => res.json())
        .then(setResults)
    }, 300)

    // Si el usuario sigue escribiendo, cancelar el timeout anterior
    return () => clearTimeout(timeoutId)
  }, [query])

  return (
    <div>
      <input
        value={query}
        onChange={(e) => setQuery(e.target.value)}
        placeholder="Buscar..."
      />
      <ul>
        {results.map((r) => (
          <li key={r.id}>{r.title}</li>
        ))}
      </ul>
    </div>
  )
}

Cada tecla cancela el timeout anterior y arranca uno nuevo de 300ms. Solo cuando el usuario deja de escribir por 300ms el fetch se dispara. Eso es debounce.

🐛 Error 1: useEffect para datos derivados (render doble)

// ❌ Dos renders, más código, mismo resultado
function Cart({ items }) {
  const [total, setTotal] = useState(0)

  useEffect(() => {
    setTotal(items.reduce((sum, i) => sum + i.price, 0))
  }, [items])

  return <div>Total: ${total}</div>
}

// ✅ Un render, menos código, mismo resultado
function Cart({ items }) {
  const total = items.reduce((sum, i) => sum + i.price, 0)
  return <div>Total: ${total}</div>
}

🐛 Error 2: memory leak por no limpiar

// ❌ Memory leak: el interval sigue corriendo después de desmontar
function Ticker() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    setInterval(() => {
      setCount((prev) => prev + 1)
    }, 1000)
    // ← sin return → sin cleanup → memory leak
  }, [])

  return <div>{count}</div>
}

// ✅ Sin memory leak
function Ticker() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    const id = setInterval(() => {
      setCount((prev) => prev + 1)
    }, 1000)
    return () => clearInterval(id) // ✅
  }, [])

  return <div>{count}</div>
}

✅ Resumen

NO necesitás useEffect para:
  → Transformar datos                  (calculalo en el render)
  → Valores derivados del estado       (calculalo en el render)
  → Inicializar estado                 (usá useState(valorInicial))

SÍ necesitás useEffect para:
  → Fetching a APIs externas
  → document.title, focus, DOM directo
  → addEventListener / removeEventListener
  → setInterval / clearInterval
  → WebSockets / conexiones persistentes

Limpieza (return () => ...):
  → Siempre que crees suscripciones, timers o conexiones
  → Se ejecuta al desmontar y antes de re-ejecutar el efecto

Buenas prácticas:
  → Un efecto, una responsabilidad
  → Si podés hacerlo sin useEffect, no uses useEffect
  → En producción: TanStack Query o SWR para fetching
