useId() Hook

🆔 The Problem: Duplicate IDs in HTML

In HTML, IDs must be unique across the entire page. They're how a <label> knows which field it belongs to, and how screen readers associate elements with each other.

When you write IDs manually in a React component, everything seems fine... until you use that component twice on the same page. 

First SearchForm:
  <label for="filter-location">Location</label>
  <select id="filter-location">

Second SearchForm:
  <label for="filter-location">Location</label>
  <select id="filter-location">   ← duplicate ID ❌

The consequences are subtle but real:

❌ Technically invalid HTML
❌ Clicking a label may focus the wrong element
❌ Screen readers get confused (broken accessibility)
❌ Testing tools report errors

💡 The Solution: Let React Generate the IDs

useId() is a React 18 hook that automatically generates a unique ID every time a new instance of the component is created.

You don't need to think about names, you don't need to worry about duplicates. React ensures that every call to useId() returns a value that's unique across the entire application. 

First SearchForm:
  label for=":r0:"  →  select id=":r0:"   ✅

Second SearchForm:
  label for=":r2:"  →  select id=":r2:"   ✅ different and unique

📋 What Does useId() Generate?

The format React generates is :r0:, :r1:, :r2:, etc. The colons are valid in HTML5 and the number increments automatically with each new instance. 

Characteristics of the generated ID:

  🔢 Incremental    — :r0:, :r1:, :r2:...
  🔒 Unique         — never repeated on the same page
  📌 Stable         — doesn't change if the component re-renders
  🌐 Works with SSR — compatible with Server-Side Rendering
  📏 Short          — doesn't bloat the HTML

📝 The Submit Event and preventDefault()

Before focusing only on IDs, there's something equally important in forms: the submit event.

When an HTML form is submitted, the browser reloads the page by default. That destroys React's state. With e.preventDefault() we cancel that behavior and handle the submission entirely with JavaScript. 

Without preventDefault():
  1. User clicks "Search"
  2. The browser reloads the page ❌
  3. All React state is lost

With preventDefault():
  1. User clicks "Search"
  2. handleSubmit runs ✅
  3. React keeps its state

🔘 Button Types in Forms

A detail that often surprises people: inside a <form>, a <button> without a type is automatically type="submit". 

<button type="submit">  →  submits the form (triggers onSubmit)
<button type="button">  →  does not submit (only runs onClick)
<button type="reset">   →  resets fields to their initial values
<button>               →  ⚠️ equivalent to type="submit" inside a form

It's always good practice to declare type explicitly so the behavior is predictable.

✅ Form with useId() + handleSubmit

import { useId } from 'react'

export function SearchForm() {
  // useId() generates a unique ID per component instance
  const locationId = useId()
  const experienceId = useId()

  const handleSubmit = (e) => {
    e.preventDefault()        // cancels page reload
    console.log('Form submitted')
  }

  return (
    <form className="search-form" onSubmit={handleSubmit}>

      <div className="form-group">
        <label htmlFor={locationId}>Location</label>
        <select id={locationId} name={locationId}>
          <option value="">Location</option>
          <option value="remote">Remote</option>
          <option value="cdmx">Mexico City</option>
          <option value="barcelona">Barcelona</option>
        </select>
      </div>

      <div className="form-group">
        <label htmlFor={experienceId}>Experience level</label>
        <select id={experienceId} name={experienceId}>
          <option value="">Experience level</option>
          <option value="junior">Junior</option>
          <option value="mid">Mid-level</option>
          <option value="senior">Senior</option>
        </select>
      </div>

      <button type="submit">Search</button>

    </form>
  )
}

🎯 Use Cases for useId()

Labels and inputs

function TextField({ label }) {
  const id = useId()
  return (
    <>
      <label htmlFor={id}>{label}</label>
      <input id={id} type="text" />
    </>
  )
}

ARIA attributes

function Tooltip({ children, tip }) {
  const id = useId()
  return (
    <>
      <div aria-describedby={id}>{children}</div>
      <div role="tooltip" id={id}>{tip}</div>
    </>
  )
}

Related IDs with suffixes

function FormField({ label }) {
  const baseId = useId()              // ":r0:"

  const inputId = `${baseId}-input`  // ":r0:-input"
  const helpId  = `${baseId}-help`   // ":r0:-help"

  return (
    <>
      <label htmlFor={inputId}>{label}</label>
      <input id={inputId} aria-describedby={helpId} />
      <small id={helpId}>Help text</small>
    </>
  )
}

⚖️ useId() vs Alternatives

// ❌ Math.random() — changes every render, guaranteed bugs
const id = `input-${Math.random()}`

// ❌ useState with a string — unnecessarily complex
const [id] = useState('my-static-id')  // still duplicable

// ✅ useId() — simple, unique, stable
const id = useId()

📏 Rules for useId() — Like All Hooks

useId() follows the same rules as all React hooks: it must be called at the top level of the component. 

// ✅ Top level — correct
function MyComponent() {
  const id1 = useId()
  const id2 = useId()
  const id3 = useId()
  return <div id={id1}>...</div>
}

// ✅ Concatenating to the ID — correct
const base = useId()
const inputId = `${base}-input`
const errorId = `${base}-error`
// ❌ Inside a condition
if (isVisible) {
  const id = useId()  // Error — violates hook rules
}

// ❌ Inside a loop
for (let i = 0; i < 3; i++) {
  const id = useId()  // Error
}

// ❌ Inside a callback
const handleClick = () => {
  const id = useId()  // Error
}

// ❌ As a key in .map() — not what it's for
items.map((item) => <div key={useId()}>{item}</div>)
// ✅ For keys, use item.id
