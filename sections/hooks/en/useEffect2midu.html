Hook useEffect() — Part II

🌊 What is a side effect?

A component's render must be a pure function: same input → same output, no surprises. But apps need to interact with the outside world. useEffect is the escape hatch for those operations. 

Common side effects:
  🌐 API calls
  📝 Changing document.title or localStorage
  🔔 Subscribing to browser events
  ⏱️ Timers and intervals
  📡 WebSockets

⏰ When does useEffect run?

The key is the second parameter — the dependency array. Three scenarios: 

No second parameter    →  on EVERY render
[], empty array        →  only once (on mount)
[dep1, dep2]           →  on mount + every time dep1 or dep2 change

Important: the effect always runs at least once on the first render, regardless of dependencies.

🔗 The dependency array

Think of the dependency array as a "watch list". React watches those values, and if any of them changes between renders, React runs the effect again.

If the list is empty [], React watches nothing → effect runs only once.
If there's no list at all, React runs it on every render.

💡 The golden rule: if you use a variable inside the effect, it must be in the dependency array. ESLint (with the React plugin) will warn you if you forget one.

♾️ The infinite loop: the most common mistake

When does it happen? When the effect modifies one of its own dependencies. This creates a cycle: 

1. component mounts → useEffect runs
2. setCount() → state changes
3. count changed → useEffect runs
4. setCount() → state changes
5. ... ♾️ infinite

The fix: never modify a dependency inside the effect that has it as a dependency.

⚠️ Objects and arrays as dependencies

Objects and arrays are recreated on every render. React uses === to compare dependencies: a new object is never equal to the previous one, even if the contents are the same: 

// On every render:
const filters = { tech: 'react' }   ← new object, new reference

// React compares: previous filters === new filters?
// {} === {} → false ❌ → the effect runs on every render

// Solution: use the primitive properties as dependencies
[filters.tech, filters.location]   ← strings, compared by value ✅

1️⃣ No dependencies: on every render

function App() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    console.log('The component rendered')
    // ⚠️ Warning: runs on EVERY render
  })

  return (
    <div>
      <p>Counter: {count}</p>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  )
}

2️⃣ Empty array: only on mount

function App() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    console.log('Only runs once on mount')
    // Useful for: initial fetch, subscriptions, etc.
  }, [])

  return (
    <div>
      <p>Counter: {count}</p>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  )
}

Now even if you click 100 times, the effect only runs once.

3️⃣ With dependencies: when something specific changes

function App() {
  const [count, setCount] = useState(0)
  const [name, setName] = useState('Midu')

  useEffect(() => {
    console.log('The counter changed:', count)
    // Only runs when `count` changes
    // Does NOT run when `name` changes
  }, [count])

  return (
    <div>
      <p>Counter: {count}</p>
      <button onClick={() => setCount(count + 1)}>Increment</button>
      <input value={name} onChange={(e) => setName(e.target.value)} />
    </div>
  )
}

4️⃣ Real example: updating the document title

function App() {
  const [currentPage, setCurrentPage] = useState(1)
  const [textToFilter, setTextToFilter] = useState('')
  const [jobsData] = useState([
    { id: 1, title: 'React Developer' },
    { id: 2, title: 'Vue Developer' },
    { id: 3, title: 'Angular Developer' },
  ])

  const jobsFiltered =
    textToFilter === ''
      ? jobsData
      : jobsData.filter((job) =>
          job.title.toLowerCase().includes(textToFilter.toLowerCase())
        )

  useEffect(() => {
    const jobCount = jobsFiltered.length

    if (textToFilter === '') {
      document.title = `DevJobs - Page ${currentPage}`
    } else {
      document.title = `${jobCount} jobs for "${textToFilter}" - Page ${currentPage}`
    }
  }, [jobsFiltered.length, currentPage, textToFilter])

  return (
    <div>
      <input
        value={textToFilter}
        onChange={(e) => setTextToFilter(e.target.value)}
        placeholder="Filter jobs..."
      />

      <ul>
        {jobsFiltered.map((job) => (
          <li key={job.id}>{job.title}</li>
        ))}
      </ul>

      <button
        onClick={() => setCurrentPage(currentPage - 1)}
        disabled={currentPage === 1}
      >
        Previous
      </button>
      <span> Page {currentPage} </span>
      <button onClick={() => setCurrentPage(currentPage + 1)}>
        Next
      </button>
    </div>
  )
}

document.title is a side effect — it lives outside the component. Perfect use case for useEffect.

5️⃣ Multiple dependencies

function JobListings() {
  const [currentPage, setCurrentPage] = useState(1)
  const [textToFilter, setTextToFilter] = useState('')

  useEffect(() => {
    console.log('Page or filter changed')
    console.log('Page:', currentPage, '| Filter:', textToFilter)
    // Runs if EITHER dependency changes
  }, [currentPage, textToFilter])

  return (
    <div>
      <input
        value={textToFilter}
        onChange={(e) => setTextToFilter(e.target.value)}
        placeholder="Filter..."
      />
      <button onClick={() => setCurrentPage(currentPage + 1)}>
        Next page
      </button>
    </div>
  )
}

📊 Dependency summary table

Dependencies            When does the effect run?
──────────────────────────────────────────────────────
No second parameter     On every render
[]                      Only once (on mount)
[dep1]                  On mount + when dep1 changes
[dep1, dep2]            On mount + when dep1 or dep2 change

🐛 Bug 1: infinite loop

// ❌ INFINITE LOOP — the effect modifies its own dependency
function App() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    setCount(count + 1)  // ← modifies count
  }, [count])             // ← count is a dependency → loop ♾️

  return <div>{count}</div>
}

// ✅ Fix 1: empty array if you don't need it to react to the change
useEffect(() => {
  setCount(1)
}, [])

// ✅ Fix 2: functional form of setState (doesn't need count as dep)
useEffect(() => {
  setCount((prev) => prev + 1)
}, [])

// ✅ Fix 3: only read, don't modify
useEffect(() => {
  document.title = `Counter: ${count}`
}, [count])

🐛 Bug 2: missing dependency

// ❌ textToFilter is used but not in the dependencies
useEffect(() => {
  console.log('Filter:', textToFilter)  // uses textToFilter...
}, [])                                  // ...but it's not here ❌
// Bug: the effect only runs on mount, uses the initial value of textToFilter forever

// ✅ Include every variable that is used
useEffect(() => {
  console.log('Filter:', textToFilter)
}, [textToFilter])  // ✅

🐛 Bug 3: object as dependency

// ❌ The object is recreated on every render → effect runs on every render
const filters = { technology: 'react', location: 'remote' }

useEffect(() => {
  console.log('Filters:', filters)
}, [filters])  // filters is new on every render → always different ❌

// ✅ Use primitive properties as dependencies
useEffect(() => {
  console.log('Filters:', filters.technology, filters.location)
}, [filters.technology, filters.location])  // strings compared by value ✅

✅ Summary

No deps          →  runs on every render (watch out for performance)
[]               →  runs once on mount
[dep1, dep2]     →  runs on mount + when dep1 or dep2 change
Infinite loop    →  happens when the effect modifies its own dependency
Objects/arrays   →  never use directly as dependencies, use their properties
Missing deps     →  cause "frozen value" bugs
Golden rule      →  everything you use inside the effect goes in the dependencies
