useEffect() Hook — Part III

🚫 When NOT to use useEffect

The core rule: if the operation doesn't touch anything outside the component, don't use useEffect. Three very common anti-patterns: 

Do I need useEffect?  Ask yourself:
  Am I transforming data I already have?    → NO, do it directly
  Am I calculating a derived value?         → NO, calculate it in the render
  Do I just want to initialize some state?  → NO, use the initial value of useState
  Do I need to communicate with something external? → YES, useEffect is correct

💡 The golden rule: if you can do it without useEffect, don't use useEffect. Every unnecessary useEffect adds complexity and potential bugs.

✅ When you SHOULD use useEffect

useEffect is correct when:
  🌐 You're fetching from external APIs
  📝 You're directly modifying the DOM (document.title, focus)
  🔔 You're subscribing to browser events (resize, scroll, click)
  ⏱️ You're creating timers (setInterval, setTimeout)
  📡 You're opening persistent connections (WebSockets)

🧹 The cleanup function

The problem: if a component subscribes to an event or starts a timer, and then the component unmounts (disappears from the screen), the subscription or timer keeps running in the background. That's a memory leak — resources wasting memory doing nothing useful.

The cleanup function (return () => { ... } inside useEffect) runs at two moments:

  1. When the component unmounts
  2. Before running the effect again (when dependencies change)
Mount component
  → useEffect runs → addEventListener / setInterval / WebSocket

Unmount component (or dep changes)
  → cleanup runs → removeEventListener / clearInterval / ws.close()
  → (if dep changed) → useEffect runs with new values

⚠️ Warning: if you create an event listener, interval, or WebSocket inside useEffect and do NOT clean up, you're creating a memory leak. The browser keeps running code for a component that no longer exists.

📦 One effect, one responsibility

Don't put everything in one giant useEffect. Each effect should do one thing. Why?

  • Easier to read and understand
  • Each effect has its own dependencies — no weird interactions
  • Cleanups don't get mixed up
  • You can enable or disable effects independently
// A component can have multiple useEffect calls:
useEffect(() => { /* load user data */ }, [userId])
useEffect(() => { /* subscribe to resize */ }, [])
useEffect(() => { /* update the title */ }, [user])

🔮 Modern alternatives to fetching with useEffect

A brief overview of the alternatives available in real-world projects:

  • React Server Components (Next.js 13+): fetch data directly in the component, no useEffect needed.
  • React 19 use hook: reads promises directly during render.
  • TanStack Query / SWR: handle cache, loading states, error handling, and refetch — you barely need useEffect for data fetching.

💡 Pro tip: for data fetching in real apps, TanStack Query or SWR are the professional choice. useEffect for fetching is educational — it teaches you the fundamentals, but in production there are better tools.

❌ Anti-pattern: transforming data with useEffect

// ❌ BAD: unnecessary useEffect to transform data
function ProductList({ products }) {
  const [expensiveProducts, setExpensiveProducts] = useState([])

  useEffect(() => {
    const filtered = products.filter((p) => p.price > 100)
    setExpensiveProducts(filtered)
  }, [products])

  return <div>{expensiveProducts.map((p) => p.name)}</div>
}

// ✅ GOOD: direct calculation in the render
function ProductList({ products }) {
  const expensiveProducts = products.filter((p) => p.price > 100)

  return <div>{expensiveProducts.map((p) => p.name)}</div>
}

The first version does 2 renders (initial + after the effect). The second does 1. Simpler and faster.

✅ Case 1: Data fetching

import { useState, useEffect } from 'react'

function UserProfile({ userId }) {
  const [user, setUser] = useState(null)
  const [loading, setLoading] = useState(true)
  const [error, setError] = useState(null)

  useEffect(() => {
    setLoading(true)
    setError(null)

    fetch(`https://api.example.com/users/${userId}`)
      .then((res) => {
        if (!res.ok) throw new Error('Failed to load user')
        return res.json()
      })
      .then((data) => {
        setUser(data)
        setLoading(false)
      })
      .catch((err) => {
        setError(err.message)
        setLoading(false)
      })
  }, [userId])

  if (loading) return <div>Loading...</div>
  if (error) return <div>Error: {error}</div>

  return (
    <div>
      <h2>{user?.name}</h2>
      <p>{user?.email}</p>
    </div>
  )
}

✅ Case 2: Subscribing to events with cleanup

import { useState, useEffect } from 'react'

function WindowSize() {
  const [windowWidth, setWindowWidth] = useState(window.innerWidth)

  useEffect(() => {
    const handleResize = () => {
      setWindowWidth(window.innerWidth)
    }

    // Subscribe
    window.addEventListener('resize', handleResize)

    // Cleanup: unsubscribe on unmount
    return () => {
      window.removeEventListener('resize', handleResize)
    }
  }, []) // [] = only on mount / unmount

  return <div>Width: {windowWidth}px</div>
}

✅ Case 3: Timer with cleanup

function LiveCounter() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    const interval = setInterval(() => {
      setCount((prev) => prev + 1)
    }, 1000)

    // Cleanup: stop the interval on unmount
    return () => clearInterval(interval)
  }, [])

  return <div>Counter: {count}</div>
}

✅ Case 4: Cleanup when dependencies change

function Timer({ delay }) {
  const [count, setCount] = useState(0)

  useEffect(() => {
    console.log(`Setting up timer: ${delay}ms`)
    const interval = setInterval(() => {
      setCount((prev) => prev + 1)
    }, delay)

    return () => {
      console.log(`Cleaning up timer: ${delay}ms`)
      clearInterval(interval) // cleans up the old one before creating a new one
    }
  }, [delay]) // when delay changes: cleanup → new effect

  return <div>Counter every {delay}ms: {count}</div>
}

💡 Tip: when delay changes: (1) the previous effect's cleanup runs → (2) the new effect runs. The component never has two intervals at the same time.

✅ Case 5: Multiple effects — one effect, one responsibility

function UserDashboard({ userId }) {
  const [user, setUser] = useState(null)
  const [windowWidth, setWindowWidth] = useState(window.innerWidth)

  // Effect 1: load user data
  useEffect(() => {
    fetch(`/api/users/${userId}`)
      .then((res) => res.json())
      .then(setUser)
  }, [userId])

  // Effect 2: subscribe to resize
  useEffect(() => {
    const handleResize = () => setWindowWidth(window.innerWidth)
    window.addEventListener('resize', handleResize)
    return () => window.removeEventListener('resize', handleResize)
  }, [])

  // Effect 3: update the title
  useEffect(() => {
    if (user) document.title = `${user.name}'s Dashboard`
  }, [user])

  return (
    <div>
      <h1>{user?.name}</h1>
      <p>Width: {windowWidth}px</p>
    </div>
  )
}

✅ Case 6: Debounce with cleanup

function SearchWithDebounce() {
  const [query, setQuery] = useState('')
  const [results, setResults] = useState([])

  useEffect(() => {
    if (query === '') {
      setResults([])
      return
    }

    // Wait 300ms before fetching
    const timeoutId = setTimeout(() => {
      fetch(`/api/search?q=${query}`)
        .then((res) => res.json())
        .then(setResults)
    }, 300)

    // If the user keeps typing, cancel the previous timeout
    return () => clearTimeout(timeoutId)
  }, [query])

  return (
    <div>
      <input
        value={query}
        onChange={(e) => setQuery(e.target.value)}
        placeholder="Search..."
      />
      <ul>
        {results.map((r) => (
          <li key={r.id}>{r.title}</li>
        ))}
      </ul>
    </div>
  )
}

Every keystroke cancels the previous timeout and starts a new 300ms countdown. Only when the user stops typing for 300ms does the fetch actually fire. That's debounce.

📊 Summary table

Use case                       useEffect?   Cleanup needed?
──────────────────────────────────────────────────────────────────
Transforming data              ❌ No        —
Calculating derived values     ❌ No        —
Initializing state             ❌ No        —
Data fetching                  ✅ Yes       No (except abort)
Modifying document.title       ✅ Yes       No
Event listeners (resize, etc.) ✅ Yes       ✅ Yes — removeEventListener
setInterval / setTimeout       ✅ Yes       ✅ Yes — clearInterval/clearTimeout
WebSocket / connection         ✅ Yes       ✅ Yes — ws.close()

🐛 Error 1: useEffect for derived data (double render)

// ❌ Two renders, more code, same result
function Cart({ items }) {
  const [total, setTotal] = useState(0)

  useEffect(() => {
    setTotal(items.reduce((sum, i) => sum + i.price, 0))
  }, [items])

  return <div>Total: ${total}</div>
}

// ✅ One render, less code, same result
function Cart({ items }) {
  const total = items.reduce((sum, i) => sum + i.price, 0)
  return <div>Total: ${total}</div>
}

🐛 Error 2: memory leak from missing cleanup

// ❌ Memory leak: the interval keeps running after unmount
function Ticker() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    setInterval(() => {
      setCount((prev) => prev + 1)
    }, 1000)
    // ← no return → no cleanup → memory leak
  }, [])

  return <div>{count}</div>
}

// ✅ No memory leak
function Ticker() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    const id = setInterval(() => {
      setCount((prev) => prev + 1)
    }, 1000)
    return () => clearInterval(id) // ✅
  }, [])

  return <div>{count}</div>
}

✅ Summary

You DON'T need useEffect for:
  → Transforming data              (calculate it in the render)
  → Derived values from state      (calculate it in the render)
  → Initializing state             (use useState(initialValue))

You DO need useEffect for:
  → Fetching from external APIs
  → document.title, focus, direct DOM manipulation
  → addEventListener / removeEventListener
  → setInterval / clearInterval
  → WebSockets / persistent connections

Cleanup (return () => ...):
  → Whenever you create subscriptions, timers, or connections
  → Runs on unmount and before re-running the effect

Best practices:
  → One effect, one responsibility
  → If you can do it without useEffect, don't use useEffect
  → In production: TanStack Query or SWR for fetching
