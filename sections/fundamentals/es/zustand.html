Zustand: Estado Global Hecho Simple

🏙️ La Analogía de la Ciudad: Del Prop Drilling al Estado Global

Imaginá una ciudad con casas (componentes) que necesitan información del municipio (estado). A medida que la ciudad crece, la información tiene que pasar por muchas casas para llegar a la que la necesita. 

Municipio (Estado Global)
   │
   ├─ La información baja por tuberías
   │
   Casa 1
   │
   Casa 2
   │
   Casa 3 ✅ Finalmente recibe la info, ¡pero pasó por 3 casas!

Esto es PROP DRILLING — doloroso e ineficiente.

🚨 El Problema del Prop Drilling

Sin estado global, pasás props a través de componentes intermedios que ni siquiera los usan: 

// ❌ Prop Drilling: App → Page → Card → Button
function App() {
  const [user, setUser] = useState({ name: 'Alex', isLoggedIn: true })
  return <Page user={user} />
}

function Page({ user }) {
  return <Card user={user} />  // ¡no usa user!
}

function Card({ user }) {
  return <Button user={user} />  // ¡no usa user!
}

function Button({ user }) {
  return <button>Welcome, {user.name}</button>  // al fin lo usa
}

⚠️ El problema: Si necesitás cambiar de dónde viene user, tenés que actualizar App, Page, Card Y Button. ¿Agregar 10 niveles de anidamiento? Ahora es una pesadilla.

🏪 Context API: Un Supermercado Central

La solución built-in de React es Context. Creás un Provider en la raíz que pone los datos disponibles para todos los hijos sin pasar props: 

// Crear el context
const UserContext = createContext()

// Provider envuelve la app
function App() {
  const [user, setUser] = useState({ name: 'Alex' })
  return (
    <UserContext.Provider value={{ user, setUser }}>
      <Page />
    </UserContext.Provider>
  )
}

// Cualquier hijo puede consumirlo
function Button() {
  const { user } = useContext(UserContext)
  return <button>Welcome, {user.name}</button>  // ✅ ¡Acceso directo!
}

Context resuelve el prop drilling, PERO...

⚡ El Problema de Context: Re-renders Innecesarios

Context tiene un costo oculto. Cuando CUALQUIER valor en el Provider cambia, TODOS los componentes que usan ese Context se re-renderizan, aunque solo usen una pequeña parte de los datos. 

Lógica de Context:
  1. El usuario se loguea
  2. UserContext se actualiza
  3. MyComponent observa UserContext
  4. MyComponent se re-renderea ✅
  5. ThemeComponent observa UserContext
  6. ThemeComponent se re-renderea ❌ (¡no lo necesitaba!)
  7. NotificationComponent observa UserContext
  8. NotificationComponent se re-renderea ❌ (¡tampoco lo necesitaba!)

❌ El problema: Si 100 componentes usan el mismo Context, y un pequeño valor cambia, los 100 se re-renderizan. Esto mata el rendimiento en apps grandes.

🔑 Zustand: La Llave Inteligente

Acá viene Zustand. Es como reemplazar al recepcionista (Context) con llaves inteligentes. Cada componente solo se suscribe a los datos que realmente usa. 

Context = Recepcionista
  "¿Quién necesita información?" pregunta
  Todos la escuchan
  Aunque no la necesites, la escuchás

Zustand = Llave Inteligente
  Vos elegís exactamente qué información querés
  Solo vos recibís notificaciones cuando TU dato cambia
  A los demás no les importa

Zustand es una librería de gestión de estado que te da estado global con mínimos re-renders y cero boilerplate.

📦 Diferencias Clave de un Vistazo

PROPS
└─ Flujo unidireccional (parent → child)
└─ Bien para: Apps simples, anidamiento único
└─ Problema: Prop drilling con anidamiento profundo

CONTEXT API
└─ Estado global con patrón Provider/Consumer
└─ Bien para: Temas, idiomas, configuración simple
└─ Problema: Re-renders innecesarios de todos los suscriptores

ZUSTAND ✅
└─ Estado global con suscripciones
└─ Bien para: Cualquier app, cualquier tamaño, estado complejo
└─ Ventaja: Actualizaciones granulares (solo re-renderizan los afectados)
└─ Ventaja: Sin Provider necesario (opcional)
└─ Ventaja: Bundle diminuto (2.3 kB)

⚙️ Cómo Funciona Zustand Bajo el Capó

1. Creás una store con create()
2. La store es un hook (useAuthStore, useCartStore, etc.)
3. Tu componente llama ese hook: const user = useAuthStore(...)
4. El componente se suscribe a estado específico
5. Cuando el estado cambia, Zustand notifica SOLO a los suscriptores de esa parte
6. El componente se re-renderiiza SOLO si su estado suscrito cambió

Esto es mucho más eficiente que Context.

✅ Por Qué los Desarrolladores Eligen Zustand

✨ Boilerplate mínimo — menos líneas que Redux o Context
⚡ Suscripciones granulares — solo re-renderiiza cuando lo necesitás
🔒 Type-safe — funciona hermosamente con TypeScript
🪶 Bundle diminuto — 2.3 kB (mucho más pequeño que Redux)
💾 Persistente — fácil de guardar en localStorage
🧪 Fácil de testear — stores son solo funciones
🎯 Sin Provider hell — opcional, pero podés envolver si querés

🚀 Instalación

npm install zustand

Eso es. Sin configuración, sin Providers requeridos (a menos que quieras).

📖 Crear Tu Primera Store (Paso a Paso)

Acá está la store de Zustand más simple posible. Vamos a hacer una store de autenticación de usuario: 

import { create } from 'zustand'

// Paso 1: Usa create() para definir tu store
const useAuthStore = create((set) => ({
  // Paso 2: Define tu estado inicial
  user: null,
  isLoggedIn: false,

  // Paso 3: Define acciones (funciones que cambian el estado)
  login: (userData) => set({ user: userData, isLoggedIn: true }),
  logout: () => set({ user: null, isLoggedIn: false }),
}))

// ¡Eso es! useAuthStore ahora es un React hook

Desglosándolo:

create()           → Función de Zustand para crear una store
(set) => ({})      → Arrow function que recibe la función "set"
set                → Cómo actualizás el estado (actualiza toda la store)
user, isLoggedIn   → Tu estado
login, logout      → Tus acciones

🎣 Usar la Store en un Componente

// ✅ ¡Úsalo como un hook! Sin Provider necesario
function LoginButton() {
  const { user, login, logout, isLoggedIn } = useAuthStore()

  return (
    <div>
      {isLoggedIn ? (
        <>
          <p>Welcome, {user.name}</p>
          <button onClick={logout}>Logout</button>
        </>
      ) : (
        <button onClick={() => login({ name: 'Alex', email: 'alex@example.com' })}>
          Login
        </button>
      )}
    </div>
  )
}

🎯 Insight clave: Notá que LoginButton es un componente regular. Sin Context.Provider, sin setup complejo. Solo usa el hook.

🎯 Suscripciones Selectivas (La Parte Inteligente)

Acá es donde Zustand brilla. Podés suscribirte SOLO a las partes del estado que necesitás. Si solo necesitás isLoggedIn, solo te re-renderiizás cuando ESO cambia: 

// ❌ Suscribirse a la store completa (se re-renderiiza en CUALQUIER cambio)
function MyComponent() {
  const state = useAuthStore()  // Toda la store
  return <div>{state.isLoggedIn}</div>
}

// ✅ Suscribirse a un slice específico (se re-renderiiza SOLO en este cambio)
function MyComponent() {
  const isLoggedIn = useAuthStore((state) => state.isLoggedIn)
  return <div>{isLoggedIn}</div>
}

// ✅ Suscribirse a múltiples slices específicos
function MyComponent() {
  const { isLoggedIn, user } = useAuthStore(
    (state) => ({ isLoggedIn: state.isLoggedIn, user: state.user })
  )
  return <div>{user.name} — {isLoggedIn}</div>
}

💡 Esto es magia de rendimiento: El componente A solo se re-renderiiza cuando isLoggedIn cambia. El componente B solo se re-renderiiza cuando user cambia. No interfieren el uno con el otro.

🛒 Ejemplo del Mundo Real: Store de Carrito de Compras

import { create } from 'zustand'

const useCartStore = create((set) => ({
  // Estado
  items: [],
  totalPrice: 0,

  // Acciones
  addItem: (product) => set((state) => ({
    items: [...state.items, product],
    totalPrice: state.totalPrice + product.price,
  })),

  removeItem: (productId) => set((state) => {
    const removedProduct = state.items.find(item => item.id === productId)
    return {
      items: state.items.filter(item => item.id !== productId),
      totalPrice: state.totalPrice - (removedProduct?.price || 0),
    }
  }),

  clearCart: () => set({ items: [], totalPrice: 0 }),

  getItemCount: () => set(
    (state) => state.items.length
  ),
}))

// Uso en componentes
function CartWidget() {
  const itemCount = useCartStore((state) => state.items.length)
  return <div>Carrito: {itemCount} items</div>
}

function CartTotal() {
  const totalPrice = useCartStore((state) => state.totalPrice)
  return <div>Total: €{totalPrice}</div>
}

🔄 Zustand vs Context Lado a Lado

Misma funcionalidad, ambas soluciones:

// ============ VERSIÓN CON CONTEXT ============
const UserContext = createContext()

function App() {
  const [user, setUser] = useState(null)
  return (
    <UserContext.Provider value={{ user, setUser }}>
      <Page />
    </UserContext.Provider>
  )
}

function MyComponent() {
  const { user, setUser } = useContext(UserContext)
  return <div>{user?.name}</div>
}

// ============ VERSIÓN CON ZUSTAND ============
const useUserStore = create((set) => ({
  user: null,
  setUser: (userData) => set({ user: userData }),
}))

function App() {
  return <Page />  // ¡Sin Provider!
}

function MyComponent() {
  const { user } = useUserStore()
  return <div>{user?.name}</div>
}

La versión con Zustand es más limpia, simple y eficiente.

💾 Persistir el Estado en localStorage

Zustand tiene middleware para persistencia. Podés guardar tu store en localStorage y restaurarla cuando la app recargue: 

import { create } from 'zustand'
import { persist } from 'zustand/middleware'

const useAuthStore = create(
  persist(
    (set) => ({
      user: null,
      login: (userData) => set({ user: userData }),
      logout: () => set({ user: null }),
    }),
    {
      name: 'auth-storage',  // localStorage key
    }
  )
)

// ¡Ahora login/logout persiste entre recargas de página!

🧪 Las Stores de Zustand son Fáciles de Testear

import { useAuthStore } from './stores/auth'

describe('AuthStore', () => {
  // Las stores de Zustand son funciones simples, así que testear es directo
  it('should login and update user', () => {
    const store = useAuthStore.getState()
    store.login({ name: 'Alex' })

    expect(store.user.name).toBe('Alex')
    expect(store.isLoggedIn).toBe(true)
  })

  it('should logout', () => {
    const store = useAuthStore.getState()
    store.logout()

    expect(store.user).toBe(null)
    expect(store.isLoggedIn).toBe(false)
  })
})

🎯 Árbol de Decisión: Cuál Solución de Estado Usar

¿Tu estado vive en UN SOLO componente?
├─ SÍ → useState ✅
└─ NO, múltiples componentes lo necesitan
   │
   ├─ ¿Estado simple (tema, idioma)?
   │  └─ Context API ✅
   │
   └─ ¿Estado complejo (usuario, carrito, notificaciones)?
      ├─ ¿Son muchos re-renders un problema?
      │  └─ Zustand ✅✅
      └─ ¿Solo unos pocos consumidores?
         └─ Context API ⚠️ (funciona, pero Zustand es más simple)

✅ Usa useState Cuando:

• El estado vive en un solo componente
• Solo ese componente e hijos inmediatos lo usan

Ejemplo: estado de botón toggle, estado de input de form,
         estado de modal abierto/cerrado

✅ Usa Context API Cuando:

• Configuración simple que raramente cambia
• Tema (modo oscuro / modo claro)
• Selección de idioma (EN / ES)
• Preferencias de accesibilidad del usuario
• Pocos suscriptores (5-10 componentes máximo)

Por qué: Context está built-in en React, sin dependencias,
         y el estado es simple, así que no es problema.

✅✅ Usa Zustand Cuando:

• Estado complejo de aplicación
  - Info del usuario autenticado
  - Carrito de compras
  - Notificaciones
  - Favoritos
  - Filtros y búsqueda

• El rendimiento importa
  - Muchos componentes se suscriben al estado
  - Actualizaciones frecuentes
  - Necesitás re-renders granulares

• Querés código limpio
  - Sin wrapper de Provider
  - Sin useContext(...) en todos lados
  - Acciones simples

Por qué: Zustand es más pequeño, más rápido, y mucho más limpio
         para cualquier cosa más allá de estado trivial.

🚀 La Regla del Desarrollador Senior

Empezá con useState.

Si necesitás pasar estado por 3+ niveles → Context.
Si Context causa re-renders → Zustand.

En la práctica:
• 80% de los proyectos usan useState + Zustand (sin Context)
• Context solo para estilos/configuración
• Zustand para todo lo demás

El anti-patrón: Usar Context para lógica de negocio.
Eso es donde la mayoría de desarrolladores se equivocan.
