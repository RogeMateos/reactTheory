Creando un componente <Route> declarativo

🎯 El objetivo

Transformar un router con lógica imperativa (basado en if y switch) en un sistema declarativo usando un componente <Route>. Esto mejora drásticamente la legibilidad, escalabilidad y mantenibilidad.

😤 El problema: lógica imperativa en App

Actualmente el router funciona con condicionales en el componente principal:

// App.jsx — versión con if
import { useRouter } from './hooks/useRouter'
import { Home } from './pages/Home'
import { Search } from './pages/Search'
import { About } from './pages/About'

function App() {
  const { currentPath } = useRouter()

  // ❌ Lógica imperativa
  if (currentPath === '/') return <Home />
  if (currentPath === '/search') return <Search />
  if (currentPath === '/about') return <About />

  return <NotFound />
}

Alternativa con switch (tampoco es ideal)

switch (currentPath) {
  case '/': return <Home />
  case '/search': return <Search />
  case '/about': return <About />
  default: return <NotFound />
}

Aunque switch está mejor que múltiples if, ambos tienen problemas:

  • ❌ Imperativo: describís cómo funciona el routing, no qué debe hacer
  • ❌ No escala bien: añadir rutas = modificar lógica del componente
  • ❌ Difícil de leer: no es evidente qué rutas existen de un vistazo
  • ❌ Lógica acoplada: App conoce los detalles del routing

✨ La solución: componente <Route> declarativo

En lugar de condicionales, creamos un componente que encapsule la lógica de matching. Cada <Route> decide por sí mismo si debe renderizarse.

Declarativo vs Imperativo: declarativo dice "qué quiero"; imperativo dice "cómo lograrlo paso a paso". React favorece lo declarativo.

Ventajas del enfoque declarativo

  • ✅ Expresa intención: cada ruta es clara y autoexplicativa
  • ✅ Fácil de leer: ves todas las rutas de un vistazo
  • ✅ Escalable: añadir rutas es solo añadir más <Route>
  • ✅ Separación de responsabilidades: cada ruta gestiona su propia lógica
  • ✅ Sin condicionales en el componente principal

🧩 Patrón de renderizado con null

Retornar null es un patrón muy común en React para indicar "no renderizar nada":

  • ✅ Limpio: no ensucia el DOM con elementos vacíos
  • ✅ Performante: React optimiza el renderizado de null
  • ✅ Idiomático: es el estándar en React

🎨 Props con renaming (component: Component)

En JSX los componentes deben empezar con mayúscula. Por eso al recibir una prop llamada component, la renombramos: 

export function Route({ path, component: Component }) {
  // 'component' se renombra a 'Component' (con mayúscula)
  return <Component />
}

Sin el rename, JSX interpretaría <component /> como un tag HTML nativo.

Paso 1 — Crear el componente Route

// components/Route.jsx
import { useRouter } from '../hooks/useRouter'

export function Route({ path, component: Component }) {
  // Ruta actual del router
  const { currentPath } = useRouter()

  // Si no coincide, no renderiza nada
  if (currentPath !== path) {
    return null
  }

  // Si coincide, renderiza el componente
  return <Component />
}

Desglose

  • useRouter() reutiliza el hook de la clase anterior
  • if (currentPath !== path) return null es renderizado condicional
  • component: Component renombra la prop para que JSX la acepte

Paso 2 — Usar Route en App (versión declarativa)

// App.jsx — DESPUÉS
import { Route } from './components/Route'
import { Home } from './pages/Home'
import { Search } from './pages/Search'
import { About } from './pages/About'
import { NotFound } from './pages/NotFound'

function App() {
  return (
    <>
      <Route path="/" component={Home} />
      <Route path="/search" component={Search} />
      <Route path="/about" component={About} />
      <Route path="/404" component={NotFound} />
    </>
  )
}

export default App

🔍 Cómo funciona internamente

Si estamos en /

<Route path="/" component={Home}/>       → "/" === "/"       ✅ renderiza <Home />
<Route path="/search" component={Search}/> → "/" !== "/search" ❌ null
<Route path="/about" component={About}/>   → "/" !== "/about"  ❌ null

Resultado: solo se renderiza <Home />

Si estamos en /search

<Route path="/" component={Home}/>       → "/search" !== "/"       ❌ null
<Route path="/search" component={Search}/> → "/search" === "/search" ✅ renderiza <Search />
<Route path="/about" component={About}/>   → "/search" !== "/about"  ❌ null

Resultado: solo se renderiza <Search />

Paso 3 — Añadiendo layout compartido

Una vez declarativo, es trivial añadir elementos comunes que deben aparecer en todas las rutas: 

function App() {
  return (
    <>
      <Header />

      <main>
        <Route path="/" component={Home} />
        <Route path="/search" component={Search} />
        <Route path="/about" component={About} />
        <Route path="/contact" component={Contact} />
      </main>

      <Footer />
    </>
  )
}

Lo que pasa:

  • <Header /> siempre se renderiza
  • Solo una de las rutas se renderiza (la que coincide)
  • <Footer /> siempre se renderiza

⚖️ Imperativo vs Declarativo

Imperativo (if/switch) Declarativo (<Route>)
Describe CÓMO decidir Declara QUÉ debe pasar
App conoce todos los detalles Cada ruta es autónoma
Difícil añadir/quitar rutas Trivial añadir/quitar rutas
Difícil testear rutas aisladas Fácil testear cada ruta

🧪 Ejercicios

Ejercicio 1 — Refactorizar a declarativo

Dado este router imperativo, refactorizalo a la versión declarativa con <Route>:

function App() {
  const { currentPath } = useRouter()
  if (currentPath === '/') return <Home />
  if (currentPath === '/products') return <Products />
  if (currentPath === '/cart') return <Cart />
  return <NotFound />
}

Ejercicio 2 — Agregar layout

Al router del ejercicio 1, añadile un <Navbar /> que aparezca en todas las rutas y un <Footer /> al final.

Ejercicio 3 — Desafío

Modificá el componente <Route> para que acepte children en lugar de la prop component. Así podrías usarlo: 

<Route path="/">
  <Home />
</Route>
