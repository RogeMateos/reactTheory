Sincronización de la URL con el Buscador

🧠 ¿Qué resuelve este patrón?

Hacer que el estado del buscador (texto, filtros y página actual) se sincronice con la URL para que la búsqueda sea:

Beneficio Qué significa
Compartible Podés enviar la URL y el otro ve exactamente lo mismo
Persistente tras F5 Recargar la página no borra los filtros
Compatible con pestañas Cada pestaña puede tener su propio estado de búsqueda
Historial del navegador El botón atrás funciona como se espera

🎯 El objetivo

Tenés dos fuentes de información que siempre deben tener los mismos datos:

Estado React          URL
──────────────        ────────────────────────────────
texto = React         /search?text=React
filtro = JavaScript   &technology=javascript
pagina = 2            &page=2

⚠️ El problema inicial

El buscador solo actualizaba los resultados, pero nunca tocaba la URL:

useEffect(() => {
  fetchJobs({ filters, textToFilter, currentPage })
}, [filters, textToFilter, currentPage])

Esto funciona para cargar datos, pero:

❌ No actualiza la URL
❌ Si hacés F5, perdés todo
❌ No podés compartir la búsqueda

🔄 Dos sincronizaciones distintas

El patrón en realidad implementa dos flujos separados:

1. React → URL
   Usuario escribe "React"
   → State cambia
   → URL cambia

2. URL → React
   URL contiene ?text=React&page=2
   → React lee la URL
   → Inicializa el estado

🔧 Bloque 1: escribir en la URL (React → URL)

Cada vez que cambian filtros, texto o página, un useEffect actualiza la URL:

useEffect(() => {
  const params = new URLSearchParams()

  if (textToFilter) params.set('text', textToFilter)
  if (filters.technology) params.set('technology', filters.technology)
  if (currentPage > 1) params.set('page', String(currentPage))

  const newURL = `${window.location.pathname}?${params.toString()}`
  window.history.pushState({}, '', newURL)
}, [filters, textToFilter, currentPage])

Visualmente:

Usuario cambia filtro
        ↓
   filters cambia
        ↓
  useEffect se ejecuta
        ↓
  URLSearchParams construye
        ↓
  /search?technology=javascript

✅ Por qué usar URLSearchParams y no concatenación

// ✅ Correcto — URLSearchParams codifica automáticamente
const params = new URLSearchParams()
params.set('text', 'React & TypeScript')
// Resultado: text=React+%26+TypeScript

// ❌ Peligroso — concatenación manual rompe con caracteres especiales
const url = '/search?text=' + text + '&technology=' + technology

🔧 Bloque 4: input sincronizado con el estado

Este es el detalle más fácil de olvidar. Si React tiene textToFilter = 'React' pero el input arranca vacío:

// ❌ Input desincronizado — muestra resultados de "React" pero el campo está vacío
const [text, setText] = useState('')

// ✅ Input sincronizado — el campo muestra el mismo valor que el estado
const [text, setText] = useState(initialTextToFilter)

Así el flujo completo queda:

URL → State → Input

Todo muestra el mismo valor.

🔧 Bloque 2: leer la URL al arrancar (URL → React)

Si siempre arrancás el estado en 1, ignorás lo que dice la URL: 

// ❌ Ignora la URL — siempre arranca en página 1
const [currentPage, setCurrentPage] = useState(1)

// ✅ Lee la URL — arranca en la página correcta
const [currentPage, setCurrentPage] = useState(() => {
  const params = new URLSearchParams(window.location.search)
  const page = Number(params.get('page'))
  return Number.isNaN(page) || page < 1 ? 1 : page
})

⚡ Concepto React importante: Lazy Initialization

Mucha gente no entiende la diferencia entre estas dos formas:

// Forma 1 — se ejecuta en CADA render (aunque el valor no cambie)
const [page, setPage] = useState(getPageFromURL())

// Forma 2 — se ejecuta UNA SOLA VEZ (lazy initialization)
const [page, setPage] = useState(() => getPageFromURL())

Cuando pasás una función a useState, React la ejecuta solo en el primer render. Por eso se llama lazy initialization.

🛡️ Bloque 3: validación de la URL

Muy buena práctica. Alguien podría escribir manualmente en la URL:

?page=hola   → Number('hola') = NaN → rompe la lógica
?page=-500   → Página inválida → comportamiento inesperado
function getPageFromURL() {
  const params = new URLSearchParams(window.location.search)
  const page = Number(params.get('page'))

  if (Number.isNaN(page) || page < 1) return 1

  return page
}

💡 Mejora: helper reutilizable

El tutorial repite varias veces new URLSearchParams(window.location.search). Podría extraerse:

// Antes — repetido en varios lugares
const params = new URLSearchParams(window.location.search)

// Después — un helper limpio
function getSearchParams() {
  return new URLSearchParams(window.location.search)
}

// Uso
const params = getSearchParams()

🎯 La respuesta de entrevista

Si en una entrevista te preguntan: "¿Cómo sincronizarías filtros y paginación con la URL?"

  1. Guardo filtros, texto y página en React State.
  2. Uso un useEffect para actualizar la URL cuando cambie el estado.
  3. Uso URLSearchParams para construir los query params sin concatenación manual.
  4. Inicializo los useState leyendo los parámetros de la URL.
  5. Valido los valores que vienen de la URL con Number.isNaN y rangos.
  6. Mantengo inputs y filtros sincronizados con el estado inicial correcto.

📌 Resumen de conceptos clave

Concepto Para qué sirve
URLSearchParams Construir y leer query params de forma segura
window.history.pushState Actualizar la URL sin recargar la página
Lazy initialization Inicializar estado desde la URL solo una vez
Validación de URL Proteger contra valores inválidos escritos manualmente
Input sincronizado Consistencia visual entre URL, estado e input
