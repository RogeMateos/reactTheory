Keys y Renderizado de Listas

😱 El Problema: Componentes Repetidos a Mano

Cuando los datos son dinámicos, repetir JSX manualmente no escala:

// ❌ Repetitivo, no escala, datos mezclados con UI
function App() {
  return (
    <div>
      <article><h3>Frontend Developer</h3><p>TechCorp</p></article>
      <article><h3>Backend Developer</h3><p>DataStack</p></article>
      <article><h3>Full Stack Developer</h3><p>StartupX</p></article>
    </div>
  )
}
📝 Repetitivo       — copiar y pegar el mismo HTML
🔢 No escala        — ¿y si tenés 100 trabajos?
📊 Datos hardcodeados — la UI y los datos están mezclados
🔄 Difícil actualizar — cambiar algo implica tocar múltiples lugares

La solución: separar datos en un archivo JSON y renderizar con .map().

🔄 ¿Qué hace .map()?

.map() recorre un array y devuelve un nuevo array transformado. En React, transformamos datos en elementos JSX: 

const jobs = [
  { id: 1, title: 'Frontend Developer' },
  { id: 2, title: 'Backend Developer' },
  { id: 3, title: 'Full Stack Developer' },
]

// Datos → Componentes
jobs.map((job) => <JobCard key={job.id} job={job} />)

// Resultado: array de componentes React
// [
//   <JobCard key={1} job={{...}} />,
//   <JobCard key={2} job={{...}} />,
//   <JobCard key={3} job={{...}} />
// ]

🔑 ¿Por qué es obligatoria la key?

La key es un identificador único que le permite a React saber qué elementos cambiaron, se agregaron o se eliminaron cuando la lista se actualiza.

Sin keys — React no sabe qué cambió

// Agregás un elemento al inicio
Antes:  [Frontend, Backend, FullStack]
Después: [DevOps, Frontend, Backend, FullStack]

// Sin key, React compara posición por posición
// posición 0: Frontend → DevOps  ← "cambió"
// posición 1: Backend  → Frontend ← "cambió"
// posición 2: FullStack → Backend  ← "cambió"
// posición 3: (nada)  → FullStack  ← "nuevo"
// React re-renderiza TODO 🐌

Con keys — React sabe exactamente qué pasó

// Con key={id}, React compara por identidad
Antes:  [key=1 Frontend, key=2 Backend, key=3 FullStack]
Después: [key=4 DevOps, key=1 Frontend, key=2 Backend, key=3 FullStack]

// key=4 es nuevo → solo renderiza ese ⚡
// key=1, 2, 3 no cambiaron → React los reutiliza

📋 Reglas de las Keys

1. Deben ser únicas entre hermanos

// ✅ IDs únicos
{jobs.map((job) => <JobCard key={job.id} job={job} />)}

// ❌ Todas iguales — React no puede diferenciarlas
{jobs.map((job) => <JobCard key="job" job={job} />)}

2. Deben ser estables (no cambiar entre renders)

// ❌ Math.random() genera una key diferente en cada render
{jobs.map((job) => <JobCard key={Math.random()} job={job} />)}

// ✅ El ID no cambia entre renders
{jobs.map((job) => <JobCard key={job.id} job={job} />)}

3. Índices: solo en listas estáticas

// ✅ OK — lista estática que nunca cambia
const DAYS = ['Lunes', 'Martes', 'Miércoles']
{DAYS.map((day, index) => <li key={index}>{day}</li>)}

// ❌ Peligroso — lista dinámica con operaciones de reordenar/eliminar
{tasks.map((task, index) => <Task key={index} task={task} />)}

// ✅ Correcto — lista dinámica con ID único
{tasks.map((task) => <Task key={task.id} task={task} />)}

🧩 Componente JobCard

// src/components/JobCard.jsx
export function JobCard({ job }) {
  return (
    <article
      className="job-listing-card"
      data-modalidad={job.data.modalidad}
      data-nivel={job.data.nivel}
    >
      <div>
        <h3>{job.titulo}</h3>
        <small>{job.empresa} | {job.ubicacion}</small>
        <p>{job.descripcion}</p>
      </div>
      <button>Aplicar</button>
    </article>
  )
}

📋 Componente JobListings con .map()

// src/components/JobListings.jsx
import { JobCard } from './JobCard'
import jobsData from '../data/jobs.json'

export function JobListings() {
  return (
    <div className="jobs-grid">
      {jobsData.map((job) => (
        <JobCard key={job.id} job={job} />
      ))}
    </div>
  )
}
Flujo de datos:

jobs.json
  │
  ▼
jobsData (array de objetos)
  │
  .map((job) => <JobCard key={job.id} job={job} />)
  │
  ▼
Array de componentes React renderizados

🎭 Lista Vacía: Renderizado Condicional

// Con early return
export function JobListings({ jobs }) {
  if (jobs.length === 0) {
    return (
      <div className="no-jobs">
        <p>No hay trabajos disponibles en este momento.</p>
      </div>
    )
  }

  return (
    <div className="jobs-grid">
      {jobs.map((job) => (
        <JobCard key={job.id} job={job} />
      ))}
    </div>
  )
}
// Con ternaria inline
export function JobListings({ jobs }) {
  return jobs.length === 0 ? (
    <p>No hay trabajos disponibles.</p>
  ) : (
    <div className="jobs-grid">
      {jobs.map((job) => (
        <JobCard key={job.id} job={job} />
      ))}
    </div>
  )
}

⚠️ El Bug con Índices: Ejemplo Visual

Por qué usar index como key puede causar bugs en listas dinámicas:

// Estado inicial
<Task key={0} text="Tarea A" completed={false} />
<Task key={1} text="Tarea B" completed={true}  />
<Task key={2} text="Tarea C" completed={false} />

// El usuario elimina "Tarea A"

// Con índices como key:
<Task key={0} text="Tarea B" completed={true}  />  // ← key reasignada
<Task key={1} text="Tarea C" completed={false} />  // ← key reasignada
// React: "key=0 ahora es Tarea B" → puede mezclar estado ❌

// Con IDs como key:
<Task key="b" text="Tarea B" completed={true}  />  // ← key estable
<Task key="c" text="Tarea C" completed={false} />  // ← key estable
// React: "key='a' se eliminó, las demás siguen igual" ✅

✅ Resumen de la Clase

🗂️ JSON externo     — separar datos de la UI
🧩 JobCard          — componente que muestra un trabajo
📋 JobListings      — componente que renderiza la lista
🔄 .map()           — array de datos → array de componentes JSX
🔑 key obligatoria  — React la necesita para identificar elementos
🎯 Keys únicas/estables — usar ID, no Math.random() ni index
⚠️ Índices peligrosos  — solo seguros en listas completamente estáticas
🎭 Lista vacía      — siempre manejar el estado de array vacío
