Zustand: Store de Favoritos — Un Ejemplo Real

🎯 El Escenario: Tablón de Ofertas de Trabajo

Imaginá una aplicación de búsqueda de empleo:

  • Mirás ofertas de trabajo
  • Marcás jobs como favoritos (clickeás una ⭐)
  • Los desmaracás después
  • Ves cuántos favoritos tenés
  • Sabés si un job está en tus favoritos

Sin una store global: Pesadilla de prop drilling. Cada componente necesita favoritos pasados.

Con Zustand: Un import, listo. Simple y limpio.

📦 Parte 1: Entendiendo la Store

Una store de Zustand es como una caja global 🎁 de datos que:

  • Vive fuera de tus componentes
  • Se puede leer desde cualquier lado
  • Se puede modificar desde cualquier lado
  • Notifica a los componentes cuando cambia

const useFavoritesStore = create((set, get) => ({
  // Estado
  favorites: [],

  // Métodos (acciones)
  addFavorite: (jobId) => { /* ... */ },
  removeFavorite: (jobId) => { /* ... */ },
  toggleFavorite: (jobId) => { /* ... */ },
  isFavorite: (jobId) => { /* ... */ },
  favoritesCount: () => { /* ... */ }
}))

La función `create()` te devuelve tu hook. Usalo en cualquier componente:


const favorites = useFavoritesStore((state) => state.favorites)

🔧 Parte 2: Funciones Clave Explicadas

1️⃣ set() → Cambia el Estado

set() es cómo modificás la store.


set((state) => ({
  favorites: [...state.favorites, jobId]
}))

Pensalo así: "Actualizar la caja 📦"

Reglas importantes:

  • Siempre devolvé el nuevo estado
  • No mutés directamente (devolvé un array/objeto nuevo)
  • Zustand notifica automáticamente a los suscriptores

2️⃣ get() → Lee el Estado

get() te deja leer el estado actual dentro de una acción.


const current = get().favorites
if (current.includes(jobId)) {
  // Ya está en favoritos
}

Pensalo así: "Mirar dentro de la caja 👀"

3️⃣ addFavorite() → Agrega a Favoritos

Objetivo: Agregar un job ID a favoritos, pero sin duplicados.


addFavorite: (jobId) => {
  const current = get().favorites

  // ✅ Verificar si ya está en favoritos
  if (!current.includes(jobId)) {
    set({ favorites: [...current, jobId] })
  }
}

Flujo visual:


Usuario clickea ⭐
    ↓
addFavorite(123) se ejecuta
    ↓
get() lee favoritos actuales [456, 789]
    ↓
¿123 ya está ahí? No.
    ↓
set() actualiza: [456, 789, 123]
    ↓
React ve el cambio → Re-renderiza componentes que usan favoritos

4️⃣ removeFavorite() → Quita de Favoritos

Objetivo: Remover un job ID usando filter().


removeFavorite: (jobId) => {
  set((state) => ({
    favorites: state.favorites.filter(id => id !== jobId)
  }))
}

Desglose visual de filter():


Original:     [456, 789, 123, 999]
Quitar 789:        ↓
Filtrado:     [456,      123, 999]
              (mantuvo todos excepto 789)

filter() mantiene solo los items donde la condición es verdadera:

  • 456 !== 789 → ✅ mantener
  • 789 !== 789 → ❌ quitar
  • 123 !== 789 → ✅ mantener
  • 999 !== 789 → ✅ mantener

5️⃣ isFavorite() → Verifica Si es Favorito

Objetivo: Respuesta simple sí/no.


isFavorite: (jobId) => {
  return get().favorites.includes(jobId)
}

Devuelve true o false.

Usalo en componentes:


const isFav = useFavoritesStore((s) => s.isFavorite(123))

return (
  <button style={{ color: isFav ? '🟡' : '⚪' }}>
    {isFav ? 'Quitar' : 'Agregar'} Favorito
  </button>
)

6️⃣ toggleFavorite() → La Elegante

Objetivo: Función inteligente que decide: ¿agregar o quitar?


toggleFavorite: (jobId) => {
  const isFav = get().isFavorite(jobId)

  if (isFav) {
    // Ya está en favoritos → quitalo
    useFavoritesStore.getState().removeFavorite(jobId)
  } else {
    // No está en favoritos → agregalo
    useFavoritesStore.getState().addFavorite(jobId)
  }
}

Árbol de decisión visual:


Usuario clickea botón favorito
    ↓
toggleFavorite(123) se ejecuta
    ↓
¿123 está en favoritos?
    ├─ SÍ → removeFavorite(123)
    └─ NO → addFavorite(123)

7️⃣ favoritesCount() → Calcula Sin Guardar

Objetivo: Devolver el count de favoritos.


favoritesCount: () => {
  return get().favorites.length
}

¿Por qué NO guardar una variable count separada?

  • ❌ Datos duplicados = responsabilidad duplicada
  • ❌ Riesgo: count y array se desincronizar
  • ✅ Calcular bajo demanda → siempre preciso

Usalo:


const count = useFavoritesStore((s) => s.favoritesCount())
return <p>Tenés {count} favoritos</p>

🎒 Parte 3: Usando en Componentes

Paso 1: Importar el hook


import { useFavoritesStore } from './store'

Paso 2: Usalo en tu componente


function JobCard({ job }) {
  const toggleFavorite = useFavoritesStore(s => s.toggleFavorite)
  const isFav = useFavoritesStore(s => s.isFavorite(job.id))

  return (
    <div>
      <h3>{job.title}</h3>
      <button onClick={() => toggleFavorite(job.id)}>
        {isFav ? '⭐ Quitar' : '☆ Agregar a Favoritos'}
      </button>
    </div>
  )
}

No necesitas Provider. No envuelves la app. Solo importá y usá.

🔄 Parte 4: Cómo Fluye la Información


1. Usuario clickea botón
   ↓
2. onClick ejecuta toggleFavorite(123)
   ↓
3. toggleFavorite verifica: ¿123 ya está en favoritos?
   ↓
4. Llama a addFavorite(123) o removeFavorite(123)
   ↓
5. set() actualiza el estado de la store
   ↓
6. React AUTOMÁTICAMENTE detecta el cambio
   ↓
7. Componentes que usan ese dato de favoritos se re-renderizan
   ↓
8. UI se actualiza: el color del botón cambia, el contador sube

La magia: React sabe exactamente qué componentes usan la store de favoritos, así que solo ESE se re-renderiza. No toda la app.

⚖️ Parte 5: Zustand vs Context

Context (lo que aprendiste antes):


const FavContext = createContext()

function App() {
  const [favorites, setFavorites] = useState([])

  return (
    <FavContext.Provider value={{ favorites, setFavorites }}>
      <TuApp />
    </FavContext.Provider>
  )
}

// Después en componentes:
const { favorites } = useContext(FavContext)

Overhead: Necesitas un Provider que envuelva todo.

Zustand:


const useFavoritesStore = create(...)

// En cualquier componente, en cualquier lado:
const favorites = useFavoritesStore(s => s.favorites)

Simplicidad: Solo importá y usá. Nada de wrappers.

Comparación Visual:


Context:
  App envuelve todo
    └─ Provider
         └─ Tu app adentro

Zustand:
  Store existe de manera independiente
    └─ Componentes la importan directamente
       (sin wrapper)

🚀 Parte 6: La Regla de Oro

Zustand = una caja global inteligente 🎁 en el frontend.

  • set() = Actualizar la caja
  • get() = Leer de la caja
  • toggle() = Lógica inteligente que decide
  • Componentes importan y usan directamente
  • Sin Providers, sin wrappers
  • Solo componentes que usan los datos se re-renderizan

📋 Resumen

  • ✅ Crear una store con create()
  • ✅ Definir estado y acciones dentro
  • ✅ Usar set() para cambiar, get() para leer
  • ✅ Evitar duplicar datos (calcula counts, no los guardes)
  • ✅ Importar el hook en componentes y usar
  • ✅ Sin necesidad de Provider envolviendo
  • ✅ Perfecto para: auth, favoritos, theme, settings

📁 Archivo: store.js (La Store)


import { create } from 'zustand'

export const useFavoritesStore = create((set, get) => ({
  // ESTADO
  favorites: [],

  // ACCIONES
  addFavorite: (jobId) => {
    const current = get().favorites
    if (!current.includes(jobId)) {
      set({ favorites: [...current, jobId] })
    }
  },

  removeFavorite: (jobId) => {
    set((state) => ({
      favorites: state.favorites.filter(id => id !== jobId)
    }))
  },

  isFavorite: (jobId) => {
    return get().favorites.includes(jobId)
  },

  toggleFavorite: (jobId) => {
    const isFav = get().isFavorite(jobId)
    if (isFav) {
      get().removeFavorite(jobId)
    } else {
      get().addFavorite(jobId)
    }
  },

  favoritesCount: () => {
    return get().favorites.length
  }
}))

📁 Archivo: JobCard.js (Componente de Job Individual)


import { useFavoritesStore } from './store'

export function JobCard({ job }) {
  const { toggleFavorite, isFavorite } = useFavoritesStore()

  const isFav = isFavorite(job.id)

  return (
    <div className="job-card">
      <h3>{job.title}</h3>
      <p>{job.company}</p>
      <p>💰 {job.salary}</p>

      <button
        className={isFav ? 'favorite-btn active' : 'favorite-btn'}
        onClick={() => toggleFavorite(job.id)}
      >
        {isFav ? '⭐ Quitar Favorito' : '☆ Agregar Favorito'}
      </button>
    </div>
  )
}

📁 Archivo: JobList.js (Lista de Jobs)


import { useFavoritesStore } from './store'
import { JobCard } from './JobCard'

export function JobList({ jobs }) {
  const count = useFavoritesStore((state) => state.favoritesCount())

  return (
    <div>
      <p>📌 Favoritos: {count}</p>
      <div className="jobs-grid">
        {jobs.map(job => (
          <JobCard key={job.id} job={job} />
        ))}
      </div>
    </div>
  )
}

🎬 Escenario Real: Flujo Completo

Escenario: Usuario Favoritea un Job

Estado Inicial:


Store favoritos: []
Job #5: Senior Desarrollador React
Botón muestra: ☆ Agregar a Favoritos
Contador muestra: 0

Usuario clickea ☆ en Job #5

Paso 1: toggleFavorite(5) se ejecuta


toggleFavorite: (jobId) => {
  const isFav = get().isFavorite(5)  // ¿5 está en favoritos?
  // Verifica: [].includes(5) → false

  if (isFav) { /* ... */ }
  else {
    get().addFavorite(5)  // Llama a addFavorite
  }
}

Paso 2: addFavorite(5) verifica duplicados


addFavorite: (jobId) => {
  const current = get().favorites  // current = []
  if (!current.includes(5)) {      // [] no incluye 5 ✓
    set({ favorites: [...[], 5] }) // Set a [5]
  }
}

Paso 3: La store se actualiza


ANTES: favoritos: []
DESPUÉS: favoritos: [5]

Paso 4: React re-renderiza componentes suscritos

  • JobCard para Job #5 → isFavorite(5) ahora devuelve true → botón se re-renderiza
  • JobList → favoritesCount() ahora devuelve 1 → contador se actualiza

Estado Final:


Store favoritos: [5]
Job #5: Senior Desarrollador React
Botón muestra: ⭐ Quitar Favorito (color cambia)
Contador muestra: 1 favoritos

🔁 Escenario 2: Usuario Favoritea Otro Job, Después Quita el Primero

Estado actual:


Store favoritos: [5]
Usuario clickea ☆ en Job #8 (Consultor React)

toggleFavorite(8) → addFavorite(8)


ANTES: [5]
set({ favorites: [...[5], 8] })
DESPUÉS: [5, 8]

Ahora usuario clickea ⭐ en Job #5 para quitarlo

toggleFavorite(5) → removeFavorite(5)


ANTES: [5, 8]
filter(id => id !== 5)
DESPUÉS: [8]

Proceso:
  5 !== 5? NO  → quitar
  8 !== 5? YES → mantener
Resultado: [8]

Estado Final:


Store favoritos: [8]
Contador muestra: 1 favoritos
Botón Job #5: ☆ Agregar a Favoritos (vuelve a normal)
Botón Job #8: ⭐ Quitar Favorito (se mantiene favoritado)

🧠 ¿Por Qué Esto es Mejor Que Prop Drilling?

❌ Sin Zustand (con prop drilling):


function App() {
  const [favorites, setFavorites] = useState([])

  return (
    <JobList favorites={favorites} setFavorites={setFavorites} />
  )
}

function JobList({ favorites, setFavorites }) {
  return (
    <>
      {jobs.map(job => (
        <JobCard
          job={job}
          favorites={favorites}           {/* Drilling */}
          setFavorites={setFavorites}     {/* Drilling */}
        />
      ))}
    </>
  )
}

function JobCard({ job, favorites, setFavorites }) {
  const toggle = () => {
    // Lógica larga acá
  }
  return <button onClick={toggle}>...</button>
}

Cada componente en el medio tiene que saber de favoritos, aunque no los use.

✅ Con Zustand:


function App() {
  return <JobList />  {/* ¡Limpio! */}
}

function JobList() {
  const count = useFavoritesStore(s => s.favoritesCount())
  return (
    <>
      <p>{count} favoritos</p>
      {jobs.map(job => <JobCard key={job.id} job={job} />)}
    </>
  )
}

function JobCard({ job }) {
  const toggle = useFavoritesStore(s => s.toggleFavorite)
  const isFav = useFavoritesStore(s => s.isFavorite(job.id))

  return (
    <button onClick={() => toggle(job.id)}>
      {isFav ? '⭐' : '☆'}
    </button>
  )
}

Limpio. Claro. Sin drilling. Cada componente obtiene lo que necesita.

💡 Puntos Clave

  • Una sola fuente de verdad: favoritos vive en un solo lugar (la store)
  • Acceso directo: Componentes agarran lo que necesitan directamente
  • Actualizaciones predecibles: set() siempre actualiza, get() siempre lee
  • Sin duplicados: Nunca guardes count separado; calculalo
  • Toggle inteligente: Una función que decide agregar o quitar
  • Re-renders eficientes: Solo componentes que usan esos datos se actualizan
  • Sin overhead de Provider: No necesitas wrapper en la app
