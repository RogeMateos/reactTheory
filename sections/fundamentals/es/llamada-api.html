Llamada a la API: Consumiendo datos reales con fetch

📦 El problema: datos estáticos vs datos dinámicos

Hasta ahora, los datos venían de un archivo local (.json) que nunca cambia. En aplicaciones reales, los datos viven en un servidor: se actualizan, se filtran en el backend, se paginan. Usar datos estáticos no escala y no representa cómo funcionan las apps profesionales. 

// ❌ Datos estáticos — nunca cambian
import jobsData from './data/jobs.json'

function App() {
  const [jobs] = useState(jobsData)
  // ... datos congelados en el tiempo
}
// ✅ Datos dinámicos desde una API
function App() {
  const [jobs, setJobs] = useState([])
  const [loading, setLoading] = useState(true)

  useEffect(() => {
    fetch('/api/jobs')
      .then((res) => res.json())
      .then((data) => {
        setJobs(data)
        setLoading(false)
      })
  }, [])

  if (loading) return <div>Cargando empleos...</div>

  return jobs.map((job) => <JobCard key={job.id} job={job} />)
}

🚫 ¿Por qué NO se puede hacer fetch directamente en el render?

React ejecuta el cuerpo del componente en cada render. Si ponés fetch() ahí directamente, causás un loop infinito: 

// ❌ NUNCA hagas esto — loop infinito garantizado
function JobSearch() {
  const [jobs, setJobs] = useState([])

  fetch('/api/jobs')
    .then((res) => res.json())
    .then((data) => setJobs(data)) // ← re-render → fetch() → re-render → ...

  return <JobListing jobs={jobs} />
}
// El problema en detalle:
// 1. Componente se renderiza
// 2. fetch() se ejecuta
// 3. Respuesta llega → setJobs()
// 4. setJobs() causa un re-render
// 5. Volvemos al paso 1 → ¡Loop infinito! 💥

✅ La solución: useEffect con dependencias vacías

useEffect con un array de dependencias vacío [] solo se ejecuta una vez, cuando el componente se monta. Es el lugar correcto para efectos secundarios como llamadas a APIs. 

function JobSearch() {
  const [jobs, setJobs] = useState([])

  useEffect(() => {
    fetch('https://jscamp-api.vercel.app/api/jobs')
      .then((res) => res.json())
      .then((data) => setJobs(data))
  }, []) // ← [] = solo al montar el componente

  return <JobListing jobs={jobs} />
}

🧠 Preparando los estados necesarios

Para manejar una llamada a la API correctamente, necesitás al menos tres estados: 

function JobSearch() {
  // 1. Array con los empleos (empieza vacío)
  const [jobs, setJobs] = useState([])

  // 2. Booleano para mostrar feedback de carga
  const [loading, setLoading] = useState(true)

  // 3. Total de resultados para paginación
  const [total, setTotal] = useState(0)
}

¿Por qué loading empieza en true? Porque antes de que el componente se monte, ya sabemos que vamos a cargar datos. Arrancar en true evita mostrar una pantalla vacía sin explicación al usuario.

🔎 La API que usaremos

En este módulo consumimos una API real de empleos: GET https://jscamp-api.vercel.app/api/jobs

Parámetro Tipo Descripción

🏗️ Implementación completa del fetch

Así se ve la llamada a la API implementada correctamente con async/await, manejo de errores y estado de carga: 

function JobSearch() {
  const [jobs, setJobs] = useState([])
  const [loading, setLoading] = useState(true)
  const [total, setTotal] = useState(0)

  useEffect(() => {
    // Función async DENTRO del efecto (no el callback directamente)
    async function fetchJobs() {
      try {
        setLoading(true)

        const response = await fetch('https://jscamp-api.vercel.app/api/jobs')
        const json = await response.json()

        setJobs(json.data)   // solo el array, no el objeto completo
        setTotal(json.total) // total de resultados para paginación
      } catch (error) {
        console.error('Error al cargar empleos:', error)
      } finally {
        setLoading(false) // siempre se ejecuta, haya error o no
      }
    }

    fetchJobs()
  }, []) // dependencias vacías = solo al montar

  if (loading) return <div>Cargando empleos...</div>

  return <JobListing jobs={jobs} total={total} />
}

¿Por qué no usar el callback de useEffect como async directamente?
Porque useEffect espera que su callback retorne undefined o una función de cleanup. Una función async retorna una Promise, lo que rompe el contrato de useEffect. Por eso se declara la función async adentro y se la llama.

📦 Estructura de la respuesta de la API

// La API devuelve ESTO:
{
  "total": 34,      // total de empleos disponibles
  "limit": 10,      // cuántos devuelve por página
  "offset": 0,      // desde qué posición empieza
  "results": 10,    // cuántos hay en esta respuesta
  "data": [         // ← el array que nos importa
    {
      "id": "7a4d1d8b-...",
      "titulo": "Desarrollador de Software Senior",
      "empresa": "Tech Solutions Inc.",
      "ubicacion": "Remoto",
      "descripcion": "...",
      "data": {
        "technology": ["react", "node", "javascript"],
        "modalidad": "remoto",
        "nivel": "senior"
      }
    }
  ]
}
// Por eso hacemos esto:
setJobs(json.data)   // ✅ array de empleos
setTotal(json.total) // ✅ número total

// Y NO esto:
setJobs(json) // ❌ guardás el objeto completo → jobs.map() falla

🔄 El ciclo completo de carga

// Flujo paso a paso:

1. Componente se monta
   ├─ loading = true
   └─ jobs = []

2. useEffect se ejecuta
   └─ Llama a fetchJobs()

3. fetchJobs() hace la petición
   └─ Usuario ve: "Cargando empleos..."
   └─ Esperando respuesta del servidor...

4. API responde
   ├─ setJobs(json.data)
   ├─ setTotal(json.total)
   └─ setLoading(false)  ← en el finally

5. React re-renderiza
   ├─ loading = false
   └─ Se renderiza JobListing con los empleos reales

⏱️ Simulando una conexión lenta (solo en desarrollo)

Agregá un delay artificial para ver bien el estado de carga. Acordate de sacarlo antes de ir a producción. 

useEffect(() => {
  async function fetchJobs() {
    try {
      setLoading(true)

      // Delay de 5 segundos — solo para pruebas visuales
      await new Promise((resolve) => setTimeout(resolve, 5000))

      const response = await fetch('https://jscamp-api.vercel.app/api/jobs')
      const json = await response.json()

      setJobs(json.data)
      setTotal(json.total)
    } catch (error) {
      console.error(error)
    } finally {
      setLoading(false)
    }
  }

  fetchJobs()
}, [])

if (loading) {
  return (

return <JobListing jobs={jobs} total={total} />

Sin este indicador el usuario ve una pantalla en blanco y no sabe qué está pasando. Con el loading, el feedback es inmediato.

🐛 Error clásico: jobs.map is not a function

Es el error más común al consumir una API por primera vez. Aparece cuando guardás el objeto completo de la respuesta en lugar del array. 

// ❌ Causa del error
const response = await fetch('/api/jobs')
const json = await response.json()

setJobs(json) // ← guardás el OBJETO, no el array

// Luego en el render:
jobs.map(job => ...) // TypeError: jobs.map is not a function
// ✅ Solución
setJobs(json.data)   // solo el array
setTotal(json.total) // el total por separado

Regla de oro: Antes de hacer setJobs(x), preguntate: "¿x es un array?" Si no lo es, navegá hasta el array dentro del objeto primero.

🔍 Verificando con DevTools

Para confirmar que los datos vienen de la API correctamente:

  • Abrí DevTools → pestaña Network
  • Recargá la página
  • Buscá la petición a /api/jobs
  • Verificá que el Status Code sea 200 OK
  • En Response deberías ver el JSON con total, data, etc.

También podés throttlear la red a Slow 3G en DevTools para ver el estado de carga sin delay artificial.

🧠 Reflexión: ¿qué lógica va en qué custom hook?

Un hook debe representar algo concreto y cohesivo. No mezclés lógicas no relacionadas: 

// ✅ Bien: cada hook tiene una sola responsabilidad
function useFilters() {
  const [filters, setFilters] = useState({ text: '', type: '', location: '' })
  const [page, setPage] = useState(1)

  const updateFilter = (name, value) =>
    setFilters((prev) => ({ ...prev, [name]: value }))

  const nextPage = () => setPage((prev) => prev + 1)
  const prevPage = () => setPage((prev) => Math.max(1, prev - 1))

  return { filters, updateFilter, page, nextPage, prevPage }
}

function useDocumentTitle(title) {
  useEffect(() => {
    document.title = title
  }, [title])
}
// ❌ Mal: responsabilidades mezcladas
function useFilters() {
  const [filters, setFilters] = useState({})

  // Esto NO pertenece aquí — el título no tiene que ver con los filtros
  useEffect(() => {
    document.title = `Empleos - Filtros aplicados`
  }, [filters])

  return { filters, setFilters }
}

✅ Buenas prácticas y ❌ qué evitar

✅ Hacer siempre:

  • Usar try-catch-finally en todas las llamadas asíncronas
  • Gestionar el estado loading para dar feedback al usuario
  • Separar el array de datos del objeto completo de la respuesta
  • Definir la función async dentro del efecto, no el callback directamente
  • Separar responsabilidades en custom hooks distintos

❌ Nunca hacer:

  • Hacer fetch() directamente en el cuerpo del componente
  • Guardar el objeto completo de la API en el estado cuando necesitás un array
  • Mezclar lógicas no relacionadas en un mismo hook
  • Ignorar errores de las peticiones
  • Dejar delays artificiales en código de producción

🚀 Próximos pasos
