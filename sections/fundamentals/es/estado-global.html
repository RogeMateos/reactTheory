Estado Global: Prop Drilling vs Context API

Uno de los desafíos más grandes de React no es hacer componentes. Es manejar el estado cuando muchos componentes necesitan los mismos datos.

Acá está el problema:

Imaginate que tenés una casa (tu app). Para hacer llegar una carta de la sala al dormitorio, tenés dos opciones:

🏠 La Analogía de la Casa

Pensá en tu app de React como una casa con habitaciones: 

App (Sala de estar)
 ├── Layout
 │    ├── Sidebar
 │    │    └── Menu
 │    │         └── UserAvatar  ← quiere datos del usuario
 │    └── Header
 │         └── UserProfile  ← quiere datos del usuario
 └── MainContent
      └── UserSettings  ← quiere datos del usuario

Los datos del usuario están en la sala de estar (componente App), pero tres habitaciones diferentes los necesitan.

📦 Opción 1: Prop Drilling (Pasar la carta de persona en persona)

Pasas los datos como props a través de cada componente, aunque no los usen. 

// App.js - la sala de estar
function App() {
  const [user, setUser] = useState({ name: 'Juan', role: 'admin' })

  return <Layout user={user} />
}

// Layout.js - primer pasillo
function Layout({ user }) {
  return (
    <div>
      <Sidebar user={user} />
      <Header user={user} />
    </div>
  )
}

// Sidebar.js - segundo pasillo
function Sidebar({ user }) {
  return (
    <div>
      <Menu user={user} />
    </div>
  )
}

// Menu.js - tercer pasillo
function Menu({ user }) {
  return <UserAvatar user={user} />
}

// UserAvatar.js - por fin llegó al dormitorio
function UserAvatar({ user }) {
  return <img src={user.avatar} alt={user.name} />
}

¿Es malo el Prop Drilling? NO!

Esto es completamente normal y está bien cuando tenés 1-3 niveles de anidamiento.

Muchas apps React usan prop drilling por años. No es una mala práctica — es React.

El Problema Real 🚨

El prop drilling se vuelve un quilombo cuando:

  • Anidamiento profundo (7+ niveles) — Componentes que pasan datos que no usan
  • Muchas ramas — El dato pasa por muchísimos componentes
  • Refactoring duro — Cambiar la estructura = cambiar todos los componentes intermedios
  • Fatiga mental — Necesitás seguir el camino de cada prop

📫 Analogía del Paquete: El prop drilling es como una carta que pasa de persona en persona: Persona A la tiene → Persona B la pasa → Persona C la pasa → Persona D por fin la abre. ¡Si hay 7 personas entre A y D, es un quilombo!

📢 Opción 2: Context API (El tablón de anuncios)

En vez de pasar props, creás un "tablón de anuncios" que cualquiera puede leer. 

// UserContext.js
import { createContext, useState } from 'react'

const UserContext = createContext()

export function UserProvider({ children }) {
  const [user, setUser] = useState({ name: 'Juan', role: 'admin' })

  return (
    <UserContext.Provider value={{ user, setUser }}>
      {children}
    </UserContext.Provider>
  )
}

export function useUser() {
  return useContext(UserContext)
}

// App.js - envolvé toda la app
function App() {
  return (
    <UserProvider>
      <Layout />
    </UserProvider>
  )
}

// Layout.js - ¡sin props!
function Layout() {
  return (
    <div>
      <Sidebar />
      <Header />
    </div>
  )
}

// Sidebar.js - ¡todavía sin props!
function Sidebar() {
  return <Menu />
}

// Menu.js - ¡limpio!
function Menu() {
  return <UserAvatar />
}

// UserAvatar.js - lee del tablón de anuncios
function UserAvatar() {
  const { user } = useUser()
  return <img src={user.avatar} alt={user.name} />
}

Fijate: Layout, Sidebar y Menu no necesitan saber de los datos del usuario.

⚠️ El Problema Oculto del Context API

Context es poderoso, pero tiene una trampa:

Cuando el estado del Context cambia, TODOS los consumidores se re-renderean — aunque solo usen un campo.

// UserContext.js
const [user, setUser] = useState({
  name: 'Juan',
  email: 'juan@example.com',
  avatar: 'juan.jpg',
  preferences: { theme: 'dark', notifications: true },
  // ...20 campos más
})

// Después, actualizás solo el tema
setUser({ ...user, preferences: { ...user.preferences, theme: 'light' } })

// PROBLEMA: UserAvatar se re-renderea aunque solo use avatar!
// No le importa el tema, pero se re-renderea igual.

¿Cuándo es bueno el Context? ✅

Context es perfecto para estado que cambia pocas veces y se lee mucho:

  • 🌙 Tema (claro/oscuro) — cambia una vez por sesión
  • 🇪🇸 Idioma (EN/ES) — cambia una vez por sesión
  • 👤 Usuario actual (logueado) — cambia al login/logout
  • 🎫 Token de autenticación — cambia al login/logout

¿Cuándo es malo el Context? ❌

Context causa problemas de performance cuando estado cambia seguido y los consumidores necesitan campos diferentes:

  • 🖱️ Posición del mouse — cambia en cada movimiento (60+ FPS)
  • ✍️ Inputs de formulario — cambian en cada tecla
  • 🎬 Animaciones — cambian múltiples veces por segundo
  • 📜 Posición de scroll — cambia constantemente
  • ⏱️ Timer/contador — se actualiza cada milisegundo

🏙️ Analogía de la Ciudad:
• Prop drilling = Carta de persona en persona
• Context API = Tablón de anuncios en la plaza del pueblo (todos pueden leer)
• Zustand/Redux = Almacén central (optimizado para búsquedas rápidas, sin re-renders innecesarios)

🏦 Cuándo entra Zustand en Escena

Hasta ahora tenemos dos opciones. Pero ¿qué pasa si necesitás:

  • Estado global que cambia seguido
  • Muchos componentes leyendo campos diferentes
  • Sin re-renders cuando cambia estado no relacionado
  • Estado complejo con muchísimas acciones

Acá es donde entra Zustand (o Redux).

// store.js (Zustand)
import create from 'zustand'

const useAppStore = create((set) => ({
  user: { name: 'Juan' },
  theme: 'dark',
  notifications: [],
  setUser: (user) => set({ user }),
  setTheme: (theme) => set({ theme }),
  addNotification: (notif) => set((state) => ({
    notifications: [...state.notifications, notif]
  }))
}))

// Componente A lee SOLO user
function UserAvatar() {
  const user = useAppStore((state) => state.user)
  return <img src={user.avatar} />
}

// Componente B lee SOLO theme
function Header() {
  const theme = useAppStore((state) => state.theme)
  return <div style={{ background: theme === 'dark' ? '#000' : '#fff' }}>
}

// Cuando cambia user: solo Componente A se re-renderea
// Cuando cambia theme: solo Componente B se re-renderea
// ¡Sin re-renders innecesarios! ✓

🎓 La Regla del Senior Developer

Esto es lo que te separa de un junior:

¿El anidamiento es de 1-3 niveles?
  ↓ SÍ → Usa Props (¡mantelo simple!)
  ↓ NO

¿El estado cambia pocas veces? (tema, idioma, auth)
  ↓ SÍ → Usa Context API
  ↓ NO

¿El estado cambia seguido? (animaciones, inputs, etc)
  ↓ SÍ → Usa Zustand / Redux

Árbol de decisión completo ✓

💡 Qué hace un Junior: "Uso Context para todo" → Problemas de performance
💡 Qué hace un Senior: "Déjame revisar la frecuencia de cambio y la profundidad" → Herramienta correcta siempre

🎤 La Respuesta para Entrevista

Si alguien te pregunta sobre prop drilling en una entrevista, decí esto:

"El prop drilling es normal y está bien para 1-3 niveles de anidamiento. Es explícito y fácil de testear. Cuando tenés árboles más profundos o muchos componentes que pasan datos que no usan, Context API es mejor — nos permite saltear componentes intermedios. Pero Context tiene trampas: si el estado cambia seguido, los consumidores se re-renderean sin necesidad. Para eso usamos Zustand o Redux, que nos dejan seleccionar campos específicos del estado."

Si preguntan "¿Cuándo usarías cada uno?", mostrá tu árbol de decisión de arriba. Eso es la marca de un developer que piensa, no que solo code.

Ejemplo 3: El Problema de Performance (mal uso de Context)


// AppContext.js
import { createContext, useContext, useState } from 'react'

const AppContext = createContext()

export function AppProvider({ children }) {
  // Montón de estado todo junto
  const [state, setState] = useState({
    user: { name: 'Juan', email: 'juan@example.com' },
    theme: 'light',
    notifications: [],
    sidebarOpen: true,
    // ... muchísimos campos más
  })

  // Acá está el problema:
  // Cuando CUALQUIER campo cambia, TODOS los consumidores se re-renderean
  const toggleTheme = () => {
    setState({ ...state, theme: state.theme === 'light' ? 'dark' : 'light' })
  }

  return (
    <AppContext.Provider value={{ state, toggleTheme }}>
      {children}
    </AppContext.Provider>
  )
}

export function useApp() {
  return useContext(AppContext)
}

// Componente 1: Solo le importa el tema
function DarkModeToggle() {
  const { state, toggleTheme } = useApp()

  return (
    <button onClick={toggleTheme}>
      Tema actual: {state.theme}
    </button>
  )
}

// Componente 2: Solo le importa el usuario
function UserProfile() {
  const { state } = useApp()

  return <div>¡Hola {state.user.name}!</div>
}

// PROBLEMA: Cuando toggleTheme() es llamado:
// 1. state cambia
// 2. AppContext value se convierte en un objeto nuevo
// 3. AMBOS DarkModeToggle Y UserProfile se re-renderean
// 4. UserProfile no le importa el tema, ¡pero se re-renderea igual!

Ejemplo 4: La Solución (Zustand para estado que cambia seguido)


// store.js (Zustand)
import create from 'zustand'

const useAppStore = create((set) => ({
  // Separar las responsabilidades
  user: { name: 'Juan', email: 'juan@example.com' },
  theme: 'light',
  notifications: [],
  sidebarOpen: true,

  // Acciones
  setTheme: (theme) => set({ theme }),
  setUser: (user) => set({ user }),
  addNotification: (notif) => set((state) => ({
    notifications: [...state.notifications, notif]
  }))
}))

// Componente 1: Solo se suscribe al tema
function DarkModeToggle() {
  const theme = useAppStore((state) => state.theme)
  const setTheme = useAppStore((state) => state.setTheme)

  return (
    <button onClick={() => setTheme(theme === 'light' ? 'dark' : 'light')}>
      Tema actual: {theme}
    </button>
  )
}

// Componente 2: Solo se suscribe al usuario
function UserProfile() {
  const user = useAppStore((state) => state.user)

  return <div>¡Hola {user.name}!</div>
}

// SOLUCIÓN: Zustand usa el patrón de selector
// Cuando cambia el tema:
// 1. Solo DarkModeToggle se re-renderea (seleccionó theme)
// 2. UserProfile NO se re-renderea (no seleccionó theme)
// 3. ¡La performance está optimizada automáticamente! ✓

// Uso en la app
function App() {
  return (
    <div>
      <DarkModeToggle />
      <UserProfile />
    </div>
  )
}

export default App

Ejemplo 5: Estructura real en una app


// contexts/ThemeContext.js
// Propósito: El tema no cambia mucho (una vez por sesión)
// Decisión: Context API está bien
import { createContext, useContext, useState } from 'react'

const ThemeContext = createContext()

export function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light')

  const toggleTheme = () => {
    setTheme(theme === 'light' ? 'dark' : 'light')
  }

  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      {children}
    </ThemeContext.Provider>
  )
}

export function useTheme() {
  return useContext(ThemeContext)
}

// store/userStore.js
// Propósito: Los datos del usuario cambian seguido (múltiples campos, animaciones)
// Decisión: Zustand para optimizar performance
import create from 'zustand'

export const useUserStore = create((set) => ({
  user: null,
  isLoading: false,
  error: null,

  fetchUser: async (id) => {
    set({ isLoading: true })
    try {
      const response = await fetch(`/api/users/${id}`)
      const user = await response.json()
      set({ user, error: null })
    } catch (error) {
      set({ error: error.message })
    } finally {
      set({ isLoading: false })
    }
  },

  updateUser: (updates) => set((state) => ({
    user: { ...state.user, ...updates }
  }))
}))

// App.js
import { ThemeProvider } from './contexts/ThemeContext'

function App() {
  return (
    <ThemeProvider>
      <Layout />
    </ThemeProvider>
  )
}

// pages/UserPage.js
import { useUserStore } from '../store/userStore'
import { useTheme } from '../contexts/ThemeContext'

function UserPage() {
  const { user, isLoading } = useUserStore()
  const { theme } = useTheme()

  const containerStyle = {
    background: theme === 'dark' ? '#1a1a1a' : '#fff'
  }

  if (isLoading) return <p>Cargando...</p>
  if (!user) return <p>Usuario no encontrado</p>

  return (
    <div style={containerStyle}>
      <h1>{user.name}</h1>
      <p>{user.email}</p>
    </div>
  )
}

export default UserPage

Cómo Elegir: Árbol de Decisión

Paso 1: ¿Qué profundidad tiene el anidamiento?

function App() {
  <Level1>          ← 1 nivel
    <Level2>        ← 2 niveles
      <Level3>      ← 3 niveles
        <Level4>    ← 4 niveles
          <Level5>  ← 5 niveles (...)

Si es 1-3 niveles: Props están bien. Frená ahí. ✓

Paso 2: ¿Con qué frecuencia cambia el estado?

Pocas veces (una vez por sesión o menos):

  • Toggle de tema (claro/oscuro)
  • Cambio de idioma (EN/ES)
  • Login/logout del usuario
  • Feature flags

→ Usa Context API. Es simple y funciona de diez para esto. ✓

Seguido (múltiples veces por segundo):

  • Inputs de formulario (onChange events)
  • Posición del mouse
  • Progreso de animación
  • Notificaciones en tiempo real

→ Usa Zustand / Redux. Necesitás selectors para evitar re-renders innecesarios. ✓

Paso 3: La Matriz de Decisión Completa

┌─────────────────────────────────────────────────────────────────┐
│                     MATRIZ DE DECISIÓN                          │
├──────────────────────────┬──────────────┬──────────────────────┤
│ Profundidad / Frecuencia │ Props        │ Context / Zustand    │
├──────────────────────────┼──────────────┼──────────────────────┤
│ 1-3 niveles / Cualquier  │ ✓ USA PROPS  │ ✗ Excesivo           │
├──────────────────────────┼──────────────┼──────────────────────┤
│ 4+ niveles / Cambios rare│ ✗ Tedioso    │ ✓ USA CONTEXT        │
├──────────────────────────┼──────────────┼──────────────────────┤
│ 4+ niveles / Cambios     │ ✗ Tedioso    │ ⚠ Context arriesgado │
│ seguidos                 │              │ ✓ USA ZUSTAND       │
├──────────────────────────┼──────────────┼──────────────────────┤
│ Múltiples stores         │ ✗ No aplica  │ ✓ Múltiples contexts │
│                          │              │ ✓ Múltiples Zustand  │
└──────────────────────────┴──────────────┴──────────────────────┘

Ejemplo Real: Construyendo un Dashboard

Escenario: Dashboard con temas, autenticación del usuario y datos en tiempo real.

1. Tema (claro/oscuro)
   └─ Cambios: una vez por sesión
   └─ Decisión: Context API ✓

2. Idioma (EN/ES)
   └─ Cambios: una vez por sesión
   └─ Decisión: Context API ✓

3. Usuario actual (logueado)
   └─ Cambios: solo en login/logout
   └─ Decisión: Context API ✓

4. Notificaciones en vivo (llegando en tiempo real)
   └─ Cambios: muchísimas veces por segundo
   └─ Decisión: Zustand (evitar spam de re-renders) ✓

5. Input de formulario (usuario escribiendo)
   └─ Cambios: en cada tecla
   └─ Decisión: Estado local (useState) ✓
   └─ Razón: El dato está en ese componente solo

6. Datos del gráfico (trae una vez al cargar)
   └─ Cambios: una sola vez cuando la página carga
   └─ Decisión: Estado local (useState) está bien, o Zustand si se comparte ✓

Errores Comunes (Los Juniors Hacen Esto)

Error 1: Context para todo

// ❌ MALO
const AppContext = createContext()

// Ponés TODO el estado en un Context:
// - tema (cambia pocas veces)
// - inputs de formulario (cambian todo el tiempo)
// - notificaciones (llegan rapidísimo)

// Resultado: Cada tecla causa que TODO se re-renderee

Mejor enfoque:

// ✓ BIEN
const ThemeContext = createContext()  // Cambios raros
const NotificationStore = Zustand()   // Cambios frecuentes
const formState = useState()           // Local al componente

Error 2: Props para todo

// ❌ MALO - Prop drilling a través de 10 niveles
<App user={user}>
  <Layout user={user}>
    <Sidebar user={user}>
      <Menu user={user}>
        <UserAvatar user={user}>
          ...anidado 5 niveles más
        </UserAvatar>
      </Menu>
    </Sidebar>
  </Layout>
</App>

// Es quilombo refactorizar y quilombo entender

Mejor enfoque:

// ✓ BIEN - Usa Context para estado global
<UserProvider>
  <App />
</UserProvider>

// Entonces cualquier componente puede useUser() sin cadenas de props

Error 3: Zustand para estado simple

// ❌ EXCESO
const useFormStore = Zustand((set) => ({
  email: '',
  password: '',
  setEmail: (e) => set({ email: e }),
  setPassword: (p) => set({ password: p })
}))

// Este formulario se usa en UN SOLO componente
// useState es más simple y más rápido

Mejor enfoque:

// ✓ BIEN - Mantelo local
function LoginForm() {
  const [email, setEmail] = useState('')
  const [password, setPassword] = useState('')

  return (
    <form>
      <input value={email} onChange={(e) => setEmail(e.target.value)} />
      <input value={password} onChange={(e) => setPassword(e.target.value)} />
    </form>
  )
}

Pregunta de Entrevista: "¿Qué usarías acá?"

Escenario 1: Un dashboard que muestra el perfil del usuario en la esquina superior derecha.

App
 ├── Header (muestra nombre del usuario arriba a la derecha)
 ├── Sidebar
 └── MainContent
      ├── Page1
      ├── Page2
      └── Page3

App tiene los datos del usuario, Header los necesita.
Profundidad: 1 nivel (App → Header)

Respuesta: "Usaría props. Es poco profundo, claro y performante."

Escenario 2: Un menú multi-nivel donde cada item necesita el idioma actual (EN/ES).

App
 └── Layout
      └── Sidebar
           └── Menu (nivel 1)
                └── Submenu (nivel 2)
                     └── Submenu (nivel 3)
                          └── MenuItem (nivel 4)
                               └── ItemLabel (necesita idioma)

Profundidad: 5 niveles. Idioma cambia una vez por sesión.

Respuesta: "Usaría Context. Es profundo, el idioma es infrecuente,
y no queremos pasar idioma a través de cada componente."

Escenario 3: Una app de chat con mensajes llegando (1 por segundo), inputs de formulario y tema.

Para mensajes: Zustand (cambios frecuentes, selectores por campo)
Para tema: Context (cambios raros)
Para input: useState (local solo)

Respuesta: "Separaría las responsabilidades. Los mensajes y el tema
tienen patrones de cambio diferentes, así que necesitan soluciones distintas."
