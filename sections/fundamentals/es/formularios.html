Gestión de Formularios y Filtros

😰 El problema: ¿dónde viven los datos del formulario?

Cuando App renderiza JobList y SearchForm como componentes hermanos, SearchForm no puede comunicarle directamente sus valores a JobList — simplemente no se conocen entre sí.

La solución es levantar el estado al padre común (App). El padre guarda los filtros, y los pasa hacia abajo como props a cada hijo que los necesita. 

                    App
                   /   \
          SearchForm   JobList
          (produce)    (consume)

Los datos suben con callbacks → App los guarda → App los pasa hacia abajo como props

📋 FormData: leer un formulario de una sola vez

En lugar de crear un useState por cada campo (ubicación, nivel, etc.), la API nativa FormData te permite leer todos los campos de un formulario de una sola vez cuando se hace el submit.

La clave está en el atributo name de cada <select> o <input>: es exactamente esa clave la que usa FormData para identificar cada campo. 

<select name="location">   ← el name es la clave
<select name="level">      ← el name es la clave

const data = new FormData(e.target)
data.get("location")  → "remote"
data.get("level")     → "senior"

¿Por qué evitar un useState por campo? Con FormData no necesitás sincronizar el estado en cada keystroke. Solo leés los valores cuando el usuario hace submit. Menos re-renders, código más limpio.

🔗 Callback: el puente entre hijo y padre

El patrón es siempre el mismo: el padre crea la función (handleSearch) y la pasa como prop al hijo. El hijo la llama con los datos cuando el formulario se envía. El padre actualiza su estado, y React re-renderiza con las nuevas props. 

// App.jsx
const handleSearch = (filters) => setFilters(filters)
<SearchForm onSearch={handleSearch} />

// SearchForm.jsx
props.onSearch({ location: "remote", level: "senior" })
// ↑ sube los datos al padre

El hijo nunca guarda estado de filtros: solo notifica. El padre es el único dueño del estado.

🔍 Filtros en cascada: el orden importa

El insight clave: no filtramos el array original múltiples veces de forma independiente. Filtramos en secuencia: primero por dropdowns, después por texto libre sobre ese resultado, y después paginamos ese resultado. 

jobs (lista original)
  ↓ filtrar por selects (location, level)
jobsFiltradosPorFiltros
  ↓ filtrar por texto libre
jobsConFiltroTexto
  ↓ paginar
resultadosPaginados   ← esto va al JobList

Si filtrás de forma paralela sobre el array original, los filtros se ignoran entre sí. En cascada, cada filtro trabaja sobre el resultado del anterior — y así se comporta como el usuario espera.

⌨️ Filtro de texto en tiempo real (onChange)

A diferencia del submit del formulario (que dispara una sola vez), la búsqueda por texto usa onChange y dispara en cada keystroke. Actualiza el estado textToFilter en App, que inmediatamente re-filtra y re-renderiza JobList.

¿Por qué resetear currentPage a 1? Si estás en la página 3 y filtrás a 5 resultados totales, la página 3 ya no existe. Sin el reset, el componente de paginación quedaría en un estado inconsistente.

🔧 SearchForm con FormData y useId

El componente SearchForm usa useId para generar IDs accesibles y FormData para leer todos los campos en el submit. 

import { useId } from 'react'

export function SearchForm({ onSearch }) {
  const locationId = useId()
  const experienceId = useId()

  const handleSubmit = (e) => {
    e.preventDefault()
    const data = new FormData(e.target)
    const filters = {
      location: data.get('location'),
      experienceLevel: data.get('experience-level'),
    }
    onSearch(filters)
  }

  return (
    <form className="search-form" onSubmit={handleSubmit}>
      <div className="form-group">
        <label htmlFor={locationId}>Ubicación</label>
        <select id={locationId} name="location">
          <option value="">Todas</option>
          <option value="remote">Remoto</option>
          <option value="cdmx">CDMX</option>
          <option value="barcelona">Barcelona</option>
        </select>
      </div>

      <div className="form-group">
        <label htmlFor={experienceId}>Nivel de experiencia</label>
        <select id={experienceId} name="experience-level">
          <option value="">Todos</option>
          <option value="junior">Junior</option>
          <option value="mid">Mid-level</option>
          <option value="senior">Senior</option>
        </select>
      </div>

      <button type="submit">Buscar</button>
    </form>
  )
}

🏠 App.jsx: el estado centralizado

App es el único dueño del estado. Centraliza los tres estados y aplica los filtros en cascada antes de pasarle los resultados al JobList. 

import { useState } from 'react'
import { JobList } from './components/JobList'
import { SearchForm } from './components/SearchForm'
import { Pagination } from './components/Pagination'
import { JOBS } from './data/jobs'

const JOBS_PER_PAGE = 5

export function App() {
  const [filters, setFilters] = useState({
    location: '',
    experienceLevel: '',
  })
  const [textToFilter, setTextToFilter] = useState('')
  const [currentPage, setCurrentPage] = useState(1)

  const handleSearch = (newFilters) => {
    setFilters(newFilters)
    setCurrentPage(1)
  }

  const handleTextFilter = (e) => {
    setTextToFilter(e.target.value)
    setCurrentPage(1)
  }

  // 1️⃣ Filtrar por selects
  const jobsFiltradosPorFiltros = JOBS.filter((job) => {
    if (filters.location && job.location !== filters.location) return false
    if (filters.experienceLevel && job.experienceLevel !== filters.experienceLevel) return false
    return true
  })

  // 2️⃣ Filtrar por texto libre
  const jobsConFiltroTexto = jobsFiltradosPorFiltros.filter((job) => {
    if (!textToFilter) return true
    const search = textToFilter.toLowerCase()
    return (
      job.title.toLowerCase().includes(search) ||
      job.company.toLowerCase().includes(search)
    )
  })

  // 3️⃣ Paginar
  const totalPages = Math.ceil(jobsConFiltroTexto.length / JOBS_PER_PAGE)
  const resultadosPaginados = jobsConFiltroTexto.slice(
    (currentPage - 1) * JOBS_PER_PAGE,
    currentPage * JOBS_PER_PAGE
  )

  return (
    <main>
      <SearchForm onSearch={handleSearch} />

      <input
        type="search"
        placeholder="Buscar por título o empresa..."
        value={textToFilter}
        onChange={handleTextFilter}
      />

      <JobList jobs={resultadosPaginados} />

      <Pagination
        currentPage={currentPage}
        totalPages={totalPages}
        onPageChange={setCurrentPage}
      />
    </main>
  )
}

🗺️ Diagrama completo del flujo de datos

App.jsx
├── state: filters       { location: '', experienceLevel: '' }
├── state: textToFilter  ''
├── state: currentPage   1
│
├── SearchForm
│   └── onSearch={handleSearch}  ← callback que actualiza filters
│
├── <input onChange={handleTextFilter}>  ← actualiza textToFilter
│
├── filtros en cascada:
│   jobs → [filtros select] → [filtro texto] → [paginación]
│
├── JobList
│   └── jobs={resultadosPaginados}
│
└── Pagination
    ├── currentPage={currentPage}
    ├── totalPages={totalPages}
    └── onPageChange={setCurrentPage}

🔎 Cómo funciona el filtro de texto

const jobsConFiltroTexto = jobsFiltradosPorFiltros.filter((job) => {
  if (!textToFilter) return true   // sin texto → pasa todo
  const search = textToFilter.toLowerCase()
  return (
    job.title.toLowerCase().includes(search) ||
    job.company.toLowerCase().includes(search)
  )
})

.toLowerCase() en ambos lados para que "REACT" encuentre "React". .includes() es una coincidencia parcial — "rea" ya matchea "React Developer".

🐛 Errores frecuentes

Error 1: Olvidar el atributo name en los selects

// ❌ FormData no puede leer esto
<select id={locationId}>

// ✅ Con name, FormData sí lo lee
<select id={locationId} name="location">

Sin el atributo name, data.get('location') devuelve null. El id es para accesibilidad (conecta el label), el name es para FormData.

Error 2: Filtrar siempre sobre jobs en vez de en cascada

// ❌ Cada filtro trabaja sobre el original — se ignoran entre sí
const filtroLocation = jobs.filter(...)
const filtroTexto = jobs.filter(...)  // ← ignora filtroLocation

// ✅ Cada filtro trabaja sobre el resultado anterior
const filtroLocation = jobs.filter(...)
const filtroTexto = filtroLocation.filter(...)  // ← cascada correcta

Error 3: No resetear currentPage al filtrar

// ❌ Si estás en página 3 y filtrás a 5 resultados → página 3 no existe
const handleSearch = (filters) => {
  setFilters(filters)
  // setCurrentPage(1)  ← olvidado
}

// ✅ Siempre volver a página 1 al cambiar filtros
const handleSearch = (filters) => {
  setFilters(filters)
  setCurrentPage(1)  // ✅
}

✅ Resumen de la clase

📋 FormData         — lee todos los campos del form de una vez con data.get('name')
🔗 Lifting state    — el padre guarda, el hijo notifica con callbacks
🎯 onSearch prop    — callback que sube los filtros al padre
🔍 Filtros en cascada — select → texto → paginación (orden importa)
⌨️ onChange         — texto en tiempo real, dispara en cada tecla
🔄 resetear página  — siempre volver a página 1 al cambiar filtros
🆔 useId + name     — useId para accesibilidad, name para FormData
