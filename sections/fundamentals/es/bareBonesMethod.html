04.1 — The Bare Bone Method: Arquitectura de Estado en React

🎯 Objetivo de esta lección

Aprender a:

  • Diseñar primero la estructura
  • Definir correctamente el flujo de datos
  • Colocar el estado en el lugar correcto
  • Evitar errores clásicos como:
    • Props drilling innecesario
    • Estado duplicado
    • Global state mal usado

🧠 1. Mentalidad: Coder vs Engineer

Antes de escribir código, piensa.

❌ Coder:

  • Empieza con useState
  • Prueba cosas hasta que funcionen
  • Parchea problemas

✅ Engineer:

  • Define estructura
  • Analiza flujo
  • Decide arquitectura
  • Luego implementa

🏗 2. Paso 1 — Component Structure

Nunca empieces con estado.

Primero rompe la UI en piezas independientes.

Ejemplo: App tipo IMDb

Componentes posibles:


<App>
  <Search />
  <ResultsPage>
    <MovieList>
      <MovieCard />
    </MovieList>
  </ResultsPage>
</App>

🔎 Pregunta clave:

¿Qué partes visuales tienen responsabilidad propia?

Regla senior:

Un componente = una responsabilidad clara.

🔄 3. Paso 2 — Data Flow

Una vez tienes la estructura, pregunta:

¿Quién cambia el estado?

Ejemplo:

  • <Search /> cambia el texto
  • <MovieList /> usa ese texto
  • Ambos necesitan el mismo valor.

❌ Error común (Junior)


function Search() {
  const [query, setQuery] = useState("")
}

Problema: MovieList no puede acceder.

✅ Solución correcta — Lift State Up

Mover el estado al padre común.


function App() {
  const [query, setQuery] = useState("")

  return (
    <>
      <Search query={query} setQuery={setQuery} />
      <MovieList query={query} />
    </>
  )
}

Principio:

Si dos hermanos necesitan el mismo estado → súbelo al padre común.

🧵 4. Props Drilling

¿Qué es?

Pasar props por múltiples niveles:


App
 └── ResultsPage
      └── MovieList
           └── MovieCard

Si query se pasa por todos aunque no lo usen → props drilling.

🤔 ¿Siempre es malo?

No.

  • ✔ 1–2 niveles → aceptable
  • ❌ 4–5 niveles → mala arquitectura

🌍 5. Global State

¿Cuándo usarlo?

Cuando el estado:

  • Es necesario en muchas partes
  • No cambia constantemente
  • Representa algo transversal

Ejemplos:

  • Autenticación
  • Idioma
  • Tema
  • Datos de usuario

Ejemplo con Context


const AuthContext = createContext(null)

function App() {
  const [user, setUser] = useState(null)

  return (
    <AuthContext.Provider value={{ user, setUser }}>
      <Search />
      <ResultsPage />
    </AuthContext.Provider>
  )
}

Ahora cualquier componente puede acceder sin props drilling.

🧠 6. Reducer — Cuando el estado crece

Si el estado tiene múltiples transiciones:


const initialState = {
  status: "idle",
  data: [],
  error: null
}

Usa useReducer.


function reducer(state, action) {
  switch (action.type) {
    case "FETCH_START":
      return { ...state, status: "loading" }

    case "FETCH_SUCCESS":
      return { ...state, status: "success", data: action.payload }

    case "FETCH_ERROR":
      return { ...state, status: "error" }

    default:
      return state
  }
}

Esto convierte tu UI en una mini state machine.

Nivel profesional.

📜 7. Principios Fundamentales del Estado

1️⃣ Usa el menor estado posible

❌ No guardes lo que puedes derivar.


const fullName = firstName + lastName

No necesitas useState para eso.

2️⃣ Mantén el estado lo más cerca posible

  • Modal abierto → local
  • Toggle → local
  • Auth → global

3️⃣ Evita redundancia

❌ Mal:


const [movies, setMovies]
const [filteredMovies, setFilteredMovies]

✔ Mejor:


const filteredMovies = movies.filter(...)

Estado derivado ≠ estado real.

🧩 8. Decidir Qué Tipo de Estado Usar

Situación Solución
Solo un componente lo usa useState local
Dos hermanos lo necesitan Lift state up
Muchas partes lo usan Context / Store
Estado complejo useReducer
Flujo avanzado State machine
