Módulos JavaScript — Import y Export

😱 El Problema: Todo en un Archivo

A medida que tu aplicación crece, tener todo en un solo archivo se vuelve insostenible: 

// App.jsx — 500 líneas de código 😱

function Header() { return <header>...</header> }
function Footer() { return <footer>...</footer> }
function JobCard({ job }) { return <article>...</article> }
function JobList({ jobs }) { return <div>...</div> }
function SearchForm() { return <form>...</form> }

function App() {
  return (
    <div>
      <Header />
      <SearchForm />
      <JobList />
      <Footer />
    </div>
  )
}

export default App
📜 Difícil de leer     — 500+ líneas en un archivo
🔍 Difícil de encontrar — ¿Dónde está el componente que necesito?
🐛 Difícil de depurar  — Todo mezclado
👥 Difícil de colaborar — Conflictos en Git
♻️ No reutilizable     — No podés usar estos componentes en otros archivos

✅ La Solución: Módulos

Separamos cada componente en su propio archivo y usamos import / export:

src/
├── components/
│   ├── Header.jsx
│   ├── Footer.jsx
│   ├── JobCard.jsx
│   ├── JobList.jsx
│   └── SearchForm.jsx
└── App.jsx

Paso 1 — Crear el archivo del componente

// src/components/Header.jsx

function Header() {
  return (
    <header className="header">
      <h1>DevJobs</h1>
      <nav>
        <a href="#jobs">Trabajos</a>
        <a href="#about">Acerca</a>
      </nav>
    </header>
  )
}

export default Header

Paso 2 — Importarlo en App.jsx

// src/App.jsx
import Header from './components/Header'

function App() {
  return (
    <div className="app">
      <Header />
    </div>
  )
}

export default App

📤 Export Default vs Export Named

Export Default — una exportación principal por archivo

// Header.jsx
function Header() {
  return <header>...</header>
}

export default Header
// Al importar, podés usar CUALQUIER nombre
import Header from './components/Header'
import MyHeader from './components/Header'  // ✅ también funciona
import Cabecera from './components/Header'  // ✅ también funciona

⚠️ Problema: Si cada importador elige un nombre diferente para el mismo componente, el código se vuelve inconsistente y difícil de seguir.

Export Named — múltiples exports con nombres específicos

// Header.jsx
export function Header() {
  return <header>...</header>
}

export function HeaderLogo() {
  return <img src="logo.png" alt="Logo" />
}

export function HeaderNav() {
  return <nav>...</nav>
}
// Al importar, DEBÉS usar el nombre exacto con llaves {}
import { Header, HeaderLogo, HeaderNav } from './components/Header'

🏆 ¿Por qué Preferimos Named Exports?

Consistencia en toda la app

// ❌ Con default export — cada archivo puede elegir un nombre diferente
import Header from './components/Header'      // App.jsx
import MyHeader from './components/Header'    // OtherPage.jsx
import TopBar from './components/Header'      // AnotherPage.jsx
// 😱 Tres nombres distintos para el mismo componente

// ✅ Con named export — siempre el mismo nombre
import { Header } from './components/Header'  // App.jsx
import { Header } from './components/Header'  // OtherPage.jsx
import { Header } from './components/Header'  // AnotherPage.jsx
// 🎉 Un solo nombre en toda la app

Refactoring seguro

Con named exports, si renombrás el componente, tu IDE puede actualizar todas las referencias automáticamente. Con default exports, cada archivo puede tener un nombre distinto y el IDE no sabe cuáles actualizar.

Autocomplete

// Named: el IDE sugiere el nombre exacto mientras escribís
import { Header } from './components/Header'
//       ^ Autocomplete del nombre correcto ✅

// Default: tenés que recordar o adivinar el nombre
import Header from './components/Header'
//     ^ Sin guía del IDE ⚠️

✏️ Dos Formas de Escribir Named Exports

// Forma 1: export inline (recomendada)
export function Header() {
  return <header>...</header>
}

export function HeaderLogo() {
  return <img src="logo.png" />
}
// Forma 2: export al final del archivo
function Header() {
  return <header>...</header>
}

function HeaderLogo() {
  return <img src="logo.png" />
}

export { Header, HeaderLogo }

Ambas son equivalentes. La forma 1 es más común en React moderno.

🔄 Renombrando Imports con as

A veces dos archivos diferentes exportan algo con el mismo nombre. Usás as para resolver la colisión:

// ❌ Error — ambos se llaman "Button"
import { Button } from './components/Button'
import { Button } from './ui/Button'
// ✅ Renombrar con as
import { Button } from './components/Button'
import { Button as UIButton } from './ui/Button'

function App() {
  return (
    <>
      <Button>Componente Button</Button>
      <UIButton>UI Button</UIButton>
    </>
  )
}
// Ejemplo real con dos "Card" distintos
import { Card as JobCard } from './components/JobCard'
import { Card as UserCard } from './components/UserCard'

function Dashboard() {
  return (
    <>
      <JobCard job={job} />
      <UserCard user={user} />
    </>
  )
}

📂 Rutas de Importación

// Mismo directorio
import { Header } from './Header'

// Subdirectorio
import { Header } from './components/Header'

// Directorio padre
import { Header } from '../Header'

// Dos niveles arriba
import { Header } from '../../Header'

📥 Import de Múltiples Elementos

// Varios named exports a la vez
import { Header, Footer, Nav } from './components'

// Default + named juntos
import App, { config, API_URL } from './App'

// Todo como namespace (usar con moderación)
import * as Utils from './utils'
Utils.formatDate()
Utils.formatCurrency()

🛠️ Ejemplo Completo: Separando la App

// src/components/JobCard.jsx
export function JobCard({ job, isRemote, isFeatured }) {
  const { title, company, location } = job

  return (
    <article className={`job-card ${isFeatured ? 'featured' : ''}`}>
      <h3>{title}</h3>
      <p>{company}</p>
      <p>📍 {location}</p>
      {isRemote && <span className="badge">🏠 Remoto</span>}
    </article>
  )
}
// src/App.jsx
import { JobCard } from './components/JobCard'

function App() {
  return (
    <main>
      <JobCard
        job={{ title: 'Frontend Dev', company: 'TechCorp', location: 'Madrid' }}
        isRemote={true}
        isFeatured={true}
      />
    </main>
  )
}

export default App

🐛 Errores Comunes — y Cómo Evitarlos

1. Olvidar el export

// ❌ El componente existe pero no se puede importar
function Header() {
  return <header>...</header>
}

// ✅ Correcto
export function Header() {
  return <header>...</header>
}

2. Confundir default con named

// Header.jsx — exportación NAMED
export function Header() {
  return <header>...</header>
}

// ❌ Error — Header no es default export
import Header from './components/Header'

// ✅ Correcto — named export requiere llaves
import { Header } from './components/Header'

3. Ruta sin el ./

// ❌ Error — el bundler busca en node_modules
import { Header } from 'components/Header'

// ✅ Correcto — ruta relativa empieza con ./
import { Header } from './components/Header'

4. Extensión en Vite

// Vite resuelve la extensión automáticamente
import { Header } from './components/Header'      // ✅
import { Header } from './components/Header.jsx'  // ✅ también funciona

💪 Ejercicio: Separar la App en Módulos

Tomá este código monolítico y separalo en archivos individuales:

// App.jsx — todo junto ❌

function Badge({ children, type }) {
  return <span className={`badge badge-${type}`}>{children}</span>
}

function JobCard({ job, isRemote }) {
  return (
    <article>
      <h3>{job.title}</h3>
      <p>{job.company}</p>
      {isRemote && <Badge type="info">🏠 Remoto</Badge>}
    </article>
  )
}

function App() {
  return (
    <main>
      <JobCard
        job={{ title: 'Frontend Dev', company: 'TechCorp' }}
        isRemote={true}
      />
    </main>
  )
}

export default App
Estructura objetivo:

src/
├── components/
│   ├── Badge.jsx      → export function Badge
│   └── JobCard.jsx    → export function JobCard
└── App.jsx            → import { Badge } from './components/Badge'
                          import { JobCard } from './components/JobCard'

✅ Resumen de la Clase

📦 Módulos       — separar código en múltiples archivos
📤 export default — una exportación principal por archivo
📤 export named  — múltiples exportaciones con nombre exacto
📥 import        — traer código de otros archivos
🔄 as            — renombrar imports para evitar colisiones
🏆 Named exports  — preferidos: consistentes y con autocomplete
🗂️ Estructura    — un componente por archivo, en /components
