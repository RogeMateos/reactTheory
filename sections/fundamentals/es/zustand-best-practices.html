Zustand: Buenas Prácticas y Suscripciones Eficientes

🎯 Resumen Rápido

Este tutorial explica cómo usar Zustand de forma eficiente, evitando renders innecesarios y manteniendo la aplicación sincronizada entre distintas páginas.

Tres ideas que conviene internalizar:

  • ✅ No te suscribas a toda la store si no lo necesitás.
  • ✅ Suscribite siempre al estado que afecta a la UI.
  • ✅ En un logout no basta con cambiar isLogged; también hay que limpiar el estado relacionado con el usuario.

❌ Problema 1: Suscribirse a Toda la Store

const store = useStore()

Esto hace que el componente escuche todos los cambios de la store.


Store
│
├── isLogged
├── favorites
├── theme
├── user
└── filters
      │
      ▼
 Componente

⬅️ Cualquier cambio provoca render

Aunque solo uses store.isLogged, React va a re-renderizar el componente si cambia favorites, theme o cualquier otra propiedad.

✅ Solución: Suscribite Solo a lo que Necesitás

const isLogged = useStore((state) => state.isLogged)

o:

const favorites = useStore((state) => state.favorites)

Ahora el componente solo escucha una parte del estado.


Store
│
├── isLogged  ◀──── Componente
├── favorites
├── theme
└── user

Si cambia theme → el componente ni se inmuta.

✔️ Menos renders. ✔️ Mejor rendimiento.

✅ Extraer Solo Acciones

const toggleFavorite = useStore(
  (state) => state.toggleFavorite
)

Las acciones normalmente mantienen la misma referencia. Por eso toggleFavorite siempre apunta a la misma función — no provoca renders innecesarios.

⚠️ Problema 2: Suscribirse Demasiado Poco

Este es probablemente el concepto más importante del tutorial.

❌ Ejemplo incorrecto:

const toggleFavorite = useStore(
  (state) => state.toggleFavorite
)

// Y después en algún lugar:
const favorite = isFavorite(id)

El componente nunca escucha los cambios de favoritos.


Store
│
├── favorites  ❌ no escuchado
└── toggleFavorite ✅ escuchado

Usuario cambia favorito
        │
        ▼
Componente NO se actualiza

✅ Versión mejorada:

const favorites = useStore((state) => state.favorites)
const toggleFavorite = useStore((state) => state.toggleFavorite)

// Ahora derivamos desde el estado suscripto:
const isFavorite = favorites.includes(id)

🧠 La Regla de Oro de las Suscripciones

No pienses:

"Quiero suscribirme al mínimo posible."

Pensá:

"Quiero suscribirme a todo lo que pueda cambiar lo que el usuario ve."

Ese enfoque mantiene la aplicación sincronizada, evita renders innecesarios y hace que el estado global siga siendo predecible y fácil de mantener.

🌐 Estado Compartido Entre Páginas

La mayor ventaja de Zustand es compartir estado entre páginas o componentes no relacionados, sin prop drilling.


          Zustand Store
        ┌─────────────────┐
        │                 │
        │ favorites       │
        │ isLogged        │
        │ user            │
        └─────────────────┘
           ▲           ▲
           │           │
     JobList      JobDetail
           ▲           ▲
           └─────┬─────┘
                 │
            FavoriteButton

Si cambiás un favorito:


JobList
   │
toggleFavorite()
   │
   ▼
Store actualiza favorites
   │
   ▼
Todos los componentes suscritos reciben el cambio

No hace falta pasar props entre páginas.

🚪 Problema 3: Logout Incompleto

Un error muy común al hacer logout:

logout()
// isLogged = false ... pero:

// favorites = [1, 4, 7]  ← sigue en memoria
// cart = [...]            ← sigue en memoria
// user = {...}            ← sigue en memoria

Cuando otro usuario inicia sesión puede encontrarse datos de la sesión anterior.

✅ Logout limpio:

logout: () =>
  set({
    isLogged: false,
    user: null,
    favorites: [],
  })

O incluso mejor, usando un objeto de estado inicial:

const initialState = {
  isLogged: false,
  user: null,
  favorites: [],
}

logout: () => set(initialState)

Antes (incompleto):
logout → solo cambia isLogged
❌ Estado "zombi" persiste

Después (limpio):
logout → resetea todo el estado relacionado
✅ Estado limpio
✅ UI consistente
✅ Sin datos antiguos

📋 Resumen

  • ✅ Usá selectores: useStore((state) => state.x)
  • ✅ Suscribite a todo lo que determina lo que el usuario ve
  • ✅ Extraé las acciones por separado — no causan renders
  • ✅ Derivá valores computados desde el estado suscripto
  • ✅ En logout, reseteá TODO el estado del usuario, no solo isLogged
  • ✅ Usá objetos initialState para resets seguros y predecibles

❌ Anti-patrón: Suscripción a Toda la Store

// ❌ Esto re-renderiza con CUALQUIER cambio en la store
function JobCard({ job }) {
  const store = useStore()

  return (
    <button onClick={() => store.toggleFavorite(job.id)}>
      {store.isFavorite(job.id) ? '⭐' : '☆'}
    </button>
  )
}

✅ Patrón: Suscripciones Selectivas

// ✅ Re-renderiza solo cuando favorites o toggleFavorite cambian
function JobCard({ job }) {
  const favorites = useStore((state) => state.favorites)
  const toggleFavorite = useStore((state) => state.toggleFavorite)

  const isFav = favorites.includes(job.id)

  return (
    <button onClick={() => toggleFavorite(job.id)}>
      {isFav ? '⭐' : '☆'}
    </button>
  )
}

✅ Patrón: Suscribirse a Múltiples Propiedades Relacionadas

// Cuando las propiedades siempre se usan juntas, podés agruparlas
const { isLogged, user } = useAuthStore((state) => ({
  isLogged: state.isLogged,
  user: state.user,
}))

Usá esto cuando las propiedades siempre aparecen juntas y tienen relación lógica. Evitalo si genera un objeto nuevo en cada render con campos no relacionados.

✅ Patrón: Auth Store Limpia con Logout

import { create } from 'zustand'

const initialState = {
  isLogged: false,
  user: null,
  favorites: [],
}

export const useAuthStore = create((set) => ({
  ...initialState,

  login: (userData) =>
    set({
      isLogged: true,
      user: userData,
    }),

  // ✅ Reset completo — sin estado zombi
  logout: () => set(initialState),
}))

✅ Patrón: Derivar Valores desde el Estado Suscripto

function FavoriteButton({ jobId }) {
  // ✅ Suscribite al array, después derivá el booleano
  const favorites = useStore((state) => state.favorites)
  const toggleFavorite = useStore((state) => state.toggleFavorite)

  const isFav = favorites.includes(jobId)

  return (
    <button
      className={isFav ? 'btn-active' : 'btn'}
      onClick={() => toggleFavorite(jobId)}
    >
      {isFav ? '⭐ Quitar' : '☆ Agregar a Favoritos'}
    </button>
  )
}

✅ Patrón: Header que Usa Auth

function Header() {
  // Solo re-renderiza cuando isLogged o user cambian
  const isLogged = useAuthStore((state) => state.isLogged)
  const user = useAuthStore((state) => state.user)
  const logout = useAuthStore((state) => state.logout)

  return (
    <header>
      {isLogged ? (
        <>
          <span>Bienvenido, {user.name}</span>
          <button onClick={logout}>Cerrar sesión</button>
        </>
      ) : (
        <a href="/login">Iniciar sesión</a>
      )}
    </header>
  )
}

🎬 Escenario: App Multi-Usuario con Favoritos

El Usuario A inicia sesión y guarda favoritos


Estado después de la sesión del Usuario A:
  isLogged: true
  user: { id: 1, name: 'Alice' }
  favorites: [4, 7, 12]

❌ Logout incompleto (error común)

// ❌ Solo se cambia isLogged
logout: () => set({ isLogged: false })
    

Estado después de este logout:
  isLogged: false       ← cambió
  user: { id: 1, name: 'Alice' }  ← SIGUE AHÍ
  favorites: [4, 7, 12]           ← SIGUE AHÍ

El Usuario B inicia sesión y ve los favoritos y datos de Alice. Eso es un bug — y potencialmente un problema de privacidad.

✅ Logout limpio

const initialState = {
  isLogged: false,
  user: null,
  favorites: [],
}

logout: () => set(initialState)
    

Estado después del logout limpio:
  isLogged: false   ✅
  user: null        ✅
  favorites: []     ✅

El Usuario B inicia sesión con un estado completamente limpio.

🔁 Escenario: FavoriteButton en Dos Páginas

La página JobList y la página JobDetail muestran ambas un FavoriteButton para el mismo trabajo.

❌ Sin suscribirse a favorites:

function FavoriteButton({ jobId }) {
  const toggleFavorite = useStore((state) => state.toggleFavorite)

  // isFavorite se llama pero el componente nunca re-renderiza cuando cambia
  const isFav = useStore.getState().isFavorite(jobId)

  return <button onClick={() => toggleFavorite(jobId)}>{isFav ? '⭐' : '☆'}</button>
}

Usuario marca favorito el trabajo #5 en JobList
        │
        ▼
Store actualiza favorites: [5]
        │
        ▼
FavoriteButton en JobDetail → sigue mostrando ☆ (no re-renderiza)

✅ Con suscripción correcta:

function FavoriteButton({ jobId }) {
  const favorites = useStore((state) => state.favorites)
  const toggleFavorite = useStore((state) => state.toggleFavorite)

  const isFav = favorites.includes(jobId)

  return <button onClick={() => toggleFavorite(jobId)}>{isFav ? '⭐' : '☆'}</button>
}

Usuario marca favorito el trabajo #5 en JobList
        │
        ▼
Store actualiza favorites: [5]
        │
        ▼
FavoriteButton en JobDetail → re-renderiza → muestra ⭐

Mismo componente, misma store. El estado se comparte automáticamente. Sin pasar props, sin bugs de sincronización.

💡 Ideas Clave para Recordar

  • Selector = suscripción: (state) => state.x significa "re-renderizar cuando x cambia"
  • Sin selector = suscripto a todo: nunca hagas esto en apps reales
  • Las acciones son estables: extraer funciones no causa renders extra
  • Derivá, no dupliques: calculá booleanos desde el array suscripto
  • Logout = reset completo: usá un objeto initialState y reseteá todo
  • El estado compartido es gratis: cualquier componente en cualquier página puede suscribirse a la misma store
