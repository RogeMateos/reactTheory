Filtros Automáticos y Routing

😤 El problema: filtros que no filtran solos

Pensá en la experiencia de usuario típica: el usuario cambia un select → nada pasa. Cambia otro select → nada pasa. Escribe en el input → nada pasa. Finalmente hace clic en "Buscar" → recién ahí se aplican los filtros. Mala UX.

La solución es eliminar el botón de submit y hacer que el formulario dispare automáticamente en cada cambio de campo, usando onChange en el elemento <form> mismo, no en cada campo individual.

Tip: Cuando ponés onChange en el <form> (no en cada campo), React captura los cambios de TODOS los campos hijo con un solo handler.

🎯 event.target vs event.currentTarget — la diferencia que importa

<form onChange={handleChange}>   ← tiene el listener
  <input name="search" />        ← puede lanzar el evento
  <select name="location">       ← puede lanzar el evento
</form>

event.target       = el campo que CAMBIÓ (input o select)
event.currentTarget = el elemento que TIENE el listener (siempre el form)

¿Por qué importa esta diferencia? Porque FormData necesita el formulario completo, no solo el campo que cambió. Si le pasás event.target (el select que cambió), FormData solo puede leer ESE elemento, no todos los campos.

⚠️ Error muy común: Este es uno de los errores más comunes con formularios en React. Siempre usá event.currentTarget cuando el handler está en el <form> y necesitás leer todos los campos.

🗂️ Routing manual: páginas sin React Router

Una app real necesita múltiples páginas: Home (/), Search (/search), 404. Sin React Router, podemos usar window.location.pathname, una propiedad nativa del navegador que devuelve la ruta actual de la URL. 

// navegás a: http://localhost:5173/
window.location.pathname  →  "/"

// navegás a: http://localhost:5173/search
window.location.pathname  →  "/search"

// navegás a: http://localhost:5173/cualquier-cosa
window.location.pathname  →  "/cualquier-cosa"

La idea: leer este valor en App.jsx y renderizar distintos componentes de página según corresponda. Es como un gran if/else que decide qué mostrar.

📁 Separar páginas de componentes

La diferencia arquitectónica es importante:

  • Componentes (src/components/) = piezas reutilizables: Header, SearchForm, JobCard
  • Páginas (src/pages/) = vistas completas que el router muestra: Home, Search, NotFound
src/
├── components/    ← piezas reutilizables
│   ├── Header.jsx
│   ├── SearchForm.jsx
│   └── JobListings.jsx
└── pages/         ← vistas completas
    ├── Home.jsx
    ├── Search.jsx
    └── NotFound.jsx

App.jsx se convierte en únicamente el shell de layout más la decisión de routing. Toda la lógica de búsqueda se mueve a Search.jsx.

⚠️ La limitación del router manual

Seamos honestos: los links con <a href="..."> causan una recarga completa de la página. Esto:

  • Pierde el estado de React
  • Es lento
  • No se siente como una SPA real

Esto es educativo — para entender CÓMO funciona el routing internamente. En producción, siempre usá React Router (próximo tema). 

Característica       Router manual    React Router
─────────────────────────────────────────────────
Recarga la página    ❌ Sí            ✅ No
Mantiene el estado   ❌ No            ✅ Sí
Rutas dinámicas      ❌ Difícil       ✅ Fácil
Producción           ❌ No            ✅ Recomendado

🔧 SearchForm sin botón: onChange en el form

import { useId } from 'react'

export function SearchForm({ onSearch }) {
  const idText = useId()
  const idLocation = useId()

  const handleChange = (event) => {
    // currentTarget = el <form>, no el campo que cambió
    const formData = new FormData(event.currentTarget)
    const filters = {
      search: formData.get(idText),
      location: formData.get(idLocation),
    }
    onSearch(filters)
  }

  return (
    <form className="search-form" onChange={handleChange}>
      <input
        type="text"
        name={idText}
        id={idText}
        placeholder="Buscar..."
      />

      <select name={idLocation} id={idLocation}>
        <option value="">Todas las ubicaciones</option>
        <option value="remoto">Remoto</option>
        <option value="presencial">Presencial</option>
      </select>
      {/* Sin botón de submit — el onChange del form lo hace automático */}
    </form>
  )
}

No hay <button type="submit">. El formulario dispara en cada cambio de campo, automáticamente.

🔍 target vs currentTarget en acción

// ❌ INCORRECTO — target es el <select> que cambió
const handleChange = (event) => {
  const formData = new FormData(event.target)
  console.log(formData.get('search'))    // null ❌
  console.log(formData.get('location'))  // null ❌
}

// ✅ CORRECTO — currentTarget es siempre el <form>
const handleChange = (event) => {
  const formData = new FormData(event.currentTarget)
  console.log(formData.get('search'))    // "react" ✅
  console.log(formData.get('location'))  // "remoto" ✅
}

🏠 Router manual en App.jsx

import { Header } from './components/Header'
import { Footer } from './components/Footer'
import { Home } from './pages/Home'
import { Search } from './pages/Search'
import { NotFound } from './pages/NotFound'

function App() {
  const { pathname } = window.location

  let Page
  if (pathname === '/') {
    Page = Home
  } else if (pathname === '/search') {
    Page = Search
  } else {
    Page = NotFound
  }

  return (
    <>
      <Header />
      <Page />
      <Footer />
    </>
  )
}

export default App

🔄 Versión alternativa con &&

function App() {
  const { pathname } = window.location

  return (
    <>
      <Header />
      {pathname === '/' && <Home />}
      {pathname === '/search' && <Search />}
      {pathname !== '/' && pathname !== '/search' && <NotFound />}
      <Footer />
    </>
  )
}

Tip: Ambas versiones hacen lo mismo. La primera (if/else) es más legible cuando hay muchas rutas. La segunda (&&) es más compacta pero se vuelve difícil de leer con muchas condiciones.

⚠️ Errores frecuentes: HTML → JSX

// 1. Inputs sin cerrar
❌  <input type="text">
✅  <input type="text" />

// 2. class → className
❌  <div class="container">
✅  <div className="container">

// 3. for → htmlFor
❌  <label for="search">
✅  <label htmlFor="search">

// 4. Eventos en camelCase
❌  <button onclick={fn}>
✅  <button onClick={fn}>

// 5. Comentarios
❌  <!-- comentario -->
✅  {/* comentario */}

🐛 El error del target

// ❌ Problema: FormData del campo, no del form
<form onChange={handleChange}>
  <input name="search" />
  <select name="location">...</select>
</form>

const handleChange = (event) => {
  const formData = new FormData(event.target) // ← BUG
  // Si cambias el select:
  // formData.get('search') → null ❌
}

// ✅ Solución: usar currentTarget
const handleChange = (event) => {
  const formData = new FormData(event.currentTarget) // ← CORRECTO
  // formData.get('search') → "react" ✅
  // formData.get('location') → "remoto" ✅
}

🐛 Botón submit que queda por accidente

// ❌ Si dejás el botón, el form hace submit y recarga la página
<form onChange={handleChange}>
  <input name="search" />
  <button type="submit">Buscar</button> {/* ← SACAR */}
</form>

// ✅ Sin botón, solo onChange en el form
<form onChange={handleChange}>
  <input name="search" />
  {/* Sin botón — automático */}
</form>

✅ Resumen

onChange en <form>      — captura cambios de todos los campos hijo
event.currentTarget     — siempre el elemento con el listener (el form)
event.target            — el elemento que lanzó el evento (el campo)
FormData(currentTarget) — lee TODOS los campos del form ✅
window.location.pathname — ruta actual del navegador
src/pages/              — carpeta para vistas completas
src/components/         — carpeta para piezas reutilizables
Router manual           — educativo, en producción usar React Router
