Zustand: Persistencia y Logout Limpio

📌 Resumen Rápido

Este tutorial explica un problema muy común cuando usamos Zustand con persistencia:

Cerrar sesión NO significa que el estado global desaparezca.

Si no limpiamos la store, el siguiente usuario puede seguir viendo favoritos, carrito o preferencias del usuario anterior.

✅ Bloque 1 — Limpiar Favoritos al Hacer Logout

Esto funciona:

clearFavorites: () => set({ favorites: [] })

// Al hacer logout:
logout()
clearFavorites()

⚠️ El Problema

Mañana la store puede tener:

{
  favorites,
  filters,
  selectedJob,
  lastSearch,
  viewedJobs
}

Y terminarás haciendo:

clearFavorites()
clearFilters()
clearViewedJobs()
clearSelectedJob()
// ...

Empieza a ser difícil de mantener.

✅ Versión mejorada: definir un estado inicial

const initialState = {
  favorites: []
}

export const useFavoritesStore = create((set) => ({
  ...initialState,

  addFavorite: () => {},

  reset: () => set(initialState)
}))

Antes:


Logout
   │
   ├── limpiar favorites
   ├── limpiar filters
   ├── limpiar cart
   └── limpiar settings

Después:


Logout
   │
   └── reset()
            │
            ▼
      Estado inicial completo

Mucho más sencillo.

✅ Bloque 2 — Guardar el Estado Inicial

⚠️ El Problema

Muchos desarrolladores hacen esto:

reset: () =>
  set({
    favorites: []
  })

Meses después añaden campos nuevos pero olvidan actualizar el reset:


favorites   ✅ limpio
filters     ❌ sigue existiendo
theme       ❌ sigue existiendo

✅ Mejor versión: fuente única de verdad

const initialState = {
  favorites: [],
  filters: {},
  selectedJob: null
}

reset: () => set(initialState)

Ahora initialState es el único lugar a actualizar cuando agregás nuevos campos.

✅ Bloque 3 — Logout

const handleLogout = () => {
  logout()
  resetFavorites()

  window.location.reload()
}

💡 ¿Por qué hacer reload?

Sin reload:


React
 ├── UserStore
 ├── FavoritesStore
 ├── SearchStore
 ├── Cache
 └── Componentes montados

Logout
     │
     ▼
Todo sigue vivo en memoria

Con reload:


Logout
     │
     ▼
window.location.reload()
     │
     ▼
Nueva aplicación

React
 ├── UserStore vacío
 ├── Favorites vacío
 ├── Cache limpia
 └── Componentes reiniciados

Todo empieza desde cero.

✅ Bloque 4 — ¿Es Buena Práctica Refrescar?

Sí. Muchas aplicaciones grandes lo hacen. ¿Por qué?


Usuario A
    │
Login
    │
Favoritos
    │
Logout
    │
Refresh
    │
Login Usuario B
    │
Estado completamente limpio

Es una solución muy robusta.

✅ Bloque 5 — Si Tenés Varias Stores

En lugar de esto:

logout()
resetFavorites()
resetCart()
resetFilters()
resetSettings()
resetSearch()
resetNotifications()

Centralizá:

const resetAllStores = () => {
  useFavoritesStore.getState().reset()
  useCartStore.getState().reset()
  useFiltersStore.getState().reset()
}

const handleLogout = () => {
  logout()
  resetAllStores()

  window.location.reload()
}

🎯 Buenas Prácticas Recomendadas

✅ Mantené un initialState separado:

const initialState = {
  favorites: []
}

✅ Cada store debería tener un método reset():

reset: () => set(initialState)

✅ El logout debería ser el único responsable de limpiar la aplicación:


handleLogout()
      │
      ├── eliminar token
      ├── reset stores
      ├── limpiar persistencia
      └── reload

🚀 Conclusión

La idea clave:

La autenticación y el estado global son responsabilidades distintas.

Cerrar sesión elimina la identidad del usuario, pero no elimina automáticamente el estado de la aplicación. Por eso, una implementación profesional de logout sigue este flujo:


Usuario pulsa Logout
          │
          ▼
Eliminar token/sesión
          │
          ▼
Resetear todas las stores
          │
          ▼
Eliminar estado persistido
          │
          ▼
Recargar la aplicación
          │
          ▼
React vuelve a montar todo desde un estado limpio

Este enfoque es simple, predecible, evita estados zombi y escala mucho mejor conforme la aplicación crece.

📋 Resumen

  • ✅ Definí initialState fuera de la store — fuente única de verdad
  • ✅ Usá reset: () => set(initialState) en cada store
  • ✅ Centralizá todos los resets en una función resetAllStores()
  • ✅ Llamá a window.location.reload() después del logout para un estado realmente limpio
  • ✅ El logout es el único responsable de la limpieza — un lugar, una responsabilidad

❌ Anti-patrón: Limpieza manual campo por campo

// Difícil de mantener — hay que recordar cada campo
const handleLogout = () => {
  logout()
  clearFavorites()
  clearFilters()
  clearCart()
  clearSettings()
  // ¿Me olvidé alguno?
}

✅ Patrón: initialState + reset()

import { create } from 'zustand'
import { persist } from 'zustand/middleware'

const initialState = {
  favorites: [],
  filters: {},
  selectedJob: null,
}

export const useFavoritesStore = create(
  persist(
    (set) => ({
      ...initialState,

      addFavorite: (id) =>
        set((state) => ({
          favorites: [...state.favorites, id],
        })),

      removeFavorite: (id) =>
        set((state) => ({
          favorites: state.favorites.filter((f) => f !== id),
        })),

      // ✅ Un reset para dominarlos a todos
      reset: () => set(initialState),
    }),
    { name: 'favorites-store' }
  )
)

✅ Patrón: resetAllStores centralizado

// utils/resetStores.js
import { useFavoritesStore } from '../stores/favoritesStore'
import { useCartStore } from '../stores/cartStore'
import { useFiltersStore } from '../stores/filtersStore'

export const resetAllStores = () => {
  useFavoritesStore.getState().reset()
  useCartStore.getState().reset()
  useFiltersStore.getState().reset()
}

✅ Patrón: Handler de logout completo

import { useAuthStore } from '../auth/authStore'
import { resetAllStores } from '../utils/resetStores'

export function useLogout() {
  const logout = useAuthStore(state => state.logout)

  const handleLogout = () => {
    // 1. Limpiar auth (token, sesión)
    logout()

    // 2. Resetear todas las stores
    resetAllStores()

    // 3. Limpiar localStorage persistido (si usás persist middleware)
    localStorage.removeItem('favorites-store')
    localStorage.removeItem('cart-store')

    // 4. Reload completo — estado garantizadamente limpio
    window.location.reload()
  }

  return handleLogout
}

✅ Patrón: Usar el hook de logout en un componente

import { useLogout } from '../hooks/useLogout'

export function Header() {
  const handleLogout = useLogout()

  return (
    <header>
      <button onClick={handleLogout}>Cerrar sesión</button>
    </header>
  )
}

✅ Patrón: Auth store con initialState

import { create } from 'zustand'

const initialState = {
  isLoggedIn: false,
  user: null,
  token: null,
}

export const useAuthStore = create((set) => ({
  ...initialState,

  login: (userData, token) =>
    set({ isLoggedIn: true, user: userData, token }),

  logout: () => set(initialState),
}))

🎬 Escenario: Dos Usuarios en la Misma Sesión de Navegador

El Usuario A inicia sesión y agrega favoritos


Store (en memoria + localStorage):
  isLoggedIn: true
  user: { id: 1, name: 'Alice' }
  favorites: [4, 7, 12]
  filters: { salary: '> 100k' }

❌ Logout incompleto (error común)

// Solo se resetea la store de auth
logout()
// isLoggedIn = false

// Pero la store de favoritos sigue teniendo:
// favorites: [4, 7, 12]
// filters: { salary: '> 100k' }
// Y localStorage sigue con los datos antiguos

El Usuario B inicia sesión


Store:
  isLoggedIn: true
  user: { id: 2, name: 'Bob' }
  favorites: [4, 7, 12]         ← ¡Los favoritos de Alice! 🐛
  filters: { salary: '> 100k' } ← ¡Los filtros de Alice! 🐛

Bob ve los datos de Alice. Eso es un bug — y potencialmente un problema de privacidad.

✅ Flujo de logout correcto

const handleLogout = () => {
  logout()             // resetea la store de auth
  resetAllStores()     // resetea todas las stores a initialState

  // Limpiar entradas de localStorage persistidas
  localStorage.removeItem('favorites-store')

  window.location.reload()  // inicio limpio garantizado
}

El Usuario B inicia sesión — estado limpio


Store:
  isLoggedIn: true
  user: { id: 2, name: 'Bob' }
  favorites: []    ✅
  filters: {}      ✅

🔁 Escenario: La Store Crece con el Tiempo

Versión 1 de la store

const initialState = {
  favorites: [],
}
// reset() funciona perfectamente

Versión 2 — se agregan campos nuevos

// ❌ Se olvidó actualizar el reset manualmente:
reset: () => set({ favorites: [] })
// ¡filters y selectedJob NO se resetean!

// ✅ Con el patrón initialState:
const initialState = {
  favorites: [],
  filters: {},        // ← agregado
  selectedJob: null,  // ← agregado
}
// reset() incluye automáticamente los campos nuevos
reset: () => set(initialState)

El patrón initialState escala con la store. Actualizás un objeto, reset se encarga de todo.

🗂️ Estructura Recomendada


src/
  stores/
    favoritesStore.js   ← initialState + reset()
    cartStore.js        ← initialState + reset()
    filtersStore.js     ← initialState + reset()

  utils/
    resetStores.js      ← resetAllStores()

  hooks/
    useLogout.js        ← handleLogout() — dueño de la limpieza completa

  auth/
    authStore.js        ← initialState + logout()

💡 Ideas Clave para Recordar

  • El estado no desaparece al hacer logout — hay que limpiarlo explícitamente
  • Patrón initialState: un objeto, un reset, siempre sincronizado
  • resetAllStores: centralizá, no dispersés las llamadas de limpieza
  • window.location.reload(): la opción nuclear — estado limpio garantizado
  • También localStorage: el middleware persist escribe en localStorage — limpialo al hacer logout
