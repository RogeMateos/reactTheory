Context API y Redux: Entendiendo la Gestión de Estado

🍕 La Analogía de la Pizzería

Imaginate una pizzería. Para entender Context API, necesitamos tres componentes: 

🍕 Pizzería (createContext)
├─ El negocio existe y está listo para servir
│
📦 Mostrador con Porciones (Provider)
├─ El lugar físico donde los clientes consiguen su pizza
├─ Almacena el inventario (state)
├─ Lo pone disponible para quien lo pida
│
👤 Cliente (useContext)
├─ Llega al mostrador
├─ Pide pizza
├─ Recibe lo que necesita directamente

🎯 ¿Qué es Context API?

Context API es la solución nativa de React para compartir estado en tu aplicación sin pasar props por cada nivel (prop drilling). Tiene tres componentes clave:

Concepto Qué es Paralelo Pizzería
createContext() Crea el objeto contexto 🍕 Registrar la pizzería como negocio
Provider Componente que envuelve tu app y comparte el estado 📦 El mostrador donde se distribuye la pizza
useContext() Hook que deja a los componentes leer el estado compartido 👤 Cliente pidiendo pizza

🧠 Paso a Paso: Construyendo un Sistema de Auth

Paso 1: Crear el Context

// AuthContext.jsx
import { createContext } from 'react'

export const AuthContext = createContext()

💡 Qué está pasando: Estamos creando una caja vacía (contexto) que va a tener nuestros datos de autenticación.

Paso 2: Crear el Componente Provider

// AuthProvider.jsx
import { useState } from 'react'
import { AuthContext } from './AuthContext'

export function AuthProvider({ children }) {
  const [user, setUser] = useState(null)
  const [isLoading, setIsLoading] = useState(false)

  const login = (email, password) => {
    setIsLoading(true)
    // Simular llamada a API
    setTimeout(() => {
      setUser({ email, role: 'user' })
      setIsLoading(false)
    }, 1000)
  }

  const logout = () => {
    setUser(null)
  }

  const value = {
    user,
    isLoading,
    login,
    logout,
    isAuthenticated: user !== null,
  }

  return (
    <AuthContext.Provider value={value}>
      {children}
    </AuthContext.Provider>
  )
}

💡 Qué está pasando: El Provider envuelve tu app y pone el estado de auth disponible para todos los hijos. Es como el mostrador de la pizzería guardando todas las pizzas (estado) y ofreciéndolas a los clientes.

Paso 3: Envolver Tu App con el Provider

// main.jsx
import React from 'react'
import ReactDOM from 'react-dom/client'
import { AuthProvider } from './AuthProvider'
import App from './App'

ReactDOM.createRoot(document.getElementById('root')).render(
  <React.StrictMode>
    <AuthProvider>
      <App />
    </AuthProvider>
  </React.StrictMode>
)

✅ Resultado: Ahora cada componente dentro de <App /> puede acceder al contexto de auth.

Paso 4: Usar useContext en Tus Componentes

// Header.jsx
import { useContext } from 'react'
import { AuthContext } from './AuthContext'

export function Header() {
  const { user, logout, isAuthenticated } = useContext(AuthContext)

  return (
    <header>
      {isAuthenticated ? (
        <>
          <span>Bienvenido, {user.email}</span>
          <button onClick={logout}>Logout</button>
        </>
      ) : (
        <span>Por favor inicia sesión</span>
      )}
    </header>
  )
}

// Dashboard.jsx
export function Dashboard() {
  const { user, isLoading } = useContext(AuthContext)

  if (isLoading) return <p>Cargando...</p>
  if (!user) return <p>No autenticado</p>

  return (
    <div>
      <h1>Dashboard para {user.email}</h1>
      {/* Contenido protegido */}
    </div>
  )
}

✅ Beneficios: Header y Dashboard obtienen el estado de auth directamente del contexto. Sin prop drilling, aunque estén lejos en el árbol de componentes.

📦 La Fórmula Mágica

Recordá este patrón: 

1️⃣ createContext()      →  Crear el "negocio"
2️⃣ Componente Provider  →  Manejar estado + envolver con <Context.Provider>
3️⃣ Envolver tu app      →  <MyProvider><App /></MyProvider>
4️⃣ useContext(Context)  →  Leer estado en cualquier componente hijo

Resultado: El estado es accesible SIN prop drilling

🔍 7 Formas de Detectar si una App de React Usa Redux

Estás mirando una app de React y necesitás saber si usa Redux. Acá hay 7 formas infalibles de detectarlo:

1️⃣ Revisar package.json

La señal más obvia: busca paquetes relacionados a Redux en las dependencias. 

{
  "dependencies": {
    "react": "^18.0.0",
    "redux": "^4.2.0",              // ✅ Redux Clásico
    "react-redux": "^8.1.0",        // ✅ Bindings para React
    "@reduxjs/toolkit": "^1.9.0",   // ✅ Redux Toolkit (moderno)
    "redux-thunk": "^2.4.0"         // ✅ Middleware para async
  }
}

✅ Veredicto: Si ves redux, react-redux, o @reduxjs/toolkit, la app definitivamente usa Redux.

2️⃣ Buscar una Carpeta store/ o redux/

Las apps con Redux organizan el estado de una forma específica. Buscá: 

src/
├── store/
│   ├── index.js              // Configuración del store
│   ├── slices/               // Slices de Redux Toolkit
│   │   ├── userSlice.js
│   │   ├── authSlice.js
│   │   └── notificationsSlice.js
│   ├── reducers/             // Reducers de Redux clásico
│   ├── actions/              // Actions de Redux clásico
│   └── middleware/           // Middleware personalizado

O

src/
├── redux/
│   ├── store.js
│   ├── actions.js
│   ├── reducers.js

✅ Veredicto: Si ves una carpeta store o redux con slices/reducers, definitivamente usa Redux.

3️⃣ Revisar main.jsx o index.jsx en busca de <Provider>

Las apps con Redux envuelven el componente raíz con <Provider> de react-redux. 

// main.jsx o index.jsx
import React from 'react'
import ReactDOM from 'react-dom/client'
import { Provider } from 'react-redux'  // ✅ Este es el Provider de Redux
import { store } from './store'
import App from './App'

ReactDOM.createRoot(document.getElementById('root')).render(
  <Provider store={store}>      {/* ✅ Redux está envolviendo la app */}
    <App />
  </Provider>
)

⚠️ No confundas: Context API también usa <Provider>, pero importa de react, no de react-redux. El Provider de Redux requiere una prop store.

4️⃣ Buscar useSelector()

Este es el hook de Redux para leer estado. Si lo ves, Redux definitivamente se está usando. 

import { useSelector } from 'react-redux'

function Header() {
  const user = useSelector(state => state.auth.user)  // ✅ Redux
  const isDarkMode = useSelector(state => state.ui.isDarkMode)

  return <h1>Hola {user.name}</h1>
}

✅ Búsqueda Rápida: Abre DevTools (Ctrl+Shift+F o Cmd+Shift+F) y buscá useSelector en todo el código. Si lo encontrás, Redux se está usando.

5️⃣ Buscar useDispatch()

Este es el hook de Redux para disparar acciones. Se usa casi siempre en apps con Redux. 

import { useDispatch } from 'react-redux'
import { logout, updateUser } from './store/slices/authSlice'

function Profile() {
  const dispatch = useDispatch()  // ✅ Redux

  const handleLogout = () => {
    dispatch(logout())  // ✅ Disparando una acción
  }

  return <button onClick={handleLogout}>Logout</button>
}

✅ Búsqueda Rápida: Buscá useDispatch en tu código. Si lo encontrás, Redux se está usando.

6️⃣ Buscar Llamadas a createSlice()

Redux Toolkit usa createSlice para definir reducers, actions y state en un solo lugar. 

// store/slices/userSlice.js
import { createSlice } from '@reduxjs/toolkit'

export const userSlice = createSlice({
  name: 'user',
  initialState: {
    currentUser: null,
    users: [],
    loading: false,
  },
  reducers: {
    setCurrentUser: (state, action) => {
      state.currentUser = action.payload
    },
    addUser: (state, action) => {
      state.users.push(action.payload)
    },
  },
})

export const { setCurrentUser, addUser } = userSlice.actions
export default userSlice.reducer

✅ Búsqueda Rápida: Buscá createSlice en el código. Si lo encontrás, la app usa Redux Toolkit.

7️⃣ Buscar configureStore()

Redux Toolkit usa configureStore para configurar el store de Redux con mejores prácticas incorporadas. 

// store/index.js
import { configureStore } from '@reduxjs/toolkit'  // ✅ Redux Toolkit
import userReducer from './slices/userSlice'
import authReducer from './slices/authSlice'

export const store = configureStore({
  reducer: {
    user: userReducer,
    auth: authReducer,
  },
})

✅ Veredicto: Si ves configureStore, la app usa Redux Toolkit (la forma moderna de escribir Redux).

🚀 Referencia Rápida: Truco para Entrevista

En una entrevista, si te preguntan "¿Cómo revisás si una app de React usa Redux?", usá este árbol de decisión: 

1. Revisar package.json por "redux" o "@reduxjs/toolkit"
   └─ ¿Lo encontraste? → Usa Redux

2. Buscar "useSelector" o "useDispatch" en el código
   └─ ¿Lo encontraste? → Usa Redux

3. Buscar carpeta store/ o redux/ con estructura específica
   └─ ¿La encontraste? → Usa Redux

4. Revisar main.jsx por <Provider store={store}> de react-redux
   └─ ¿Lo encontraste? → Usa Redux

Si encontrás CUALQUIERA de estos, la app usa Redux. Si encontrás NINGUNO,
probablemente usa Context API u otra solución de state management.

✅ Checklist para Confirmación 100%

Acá está cómo estar completamente seguro: 

✓ package.json tiene "redux" o "@reduxjs/toolkit"
✓ Podés encontrar useSelector() en al menos un componente
✓ Podés encontrar useDispatch() en al menos un componente
✓ Podés encontrar el archivo de configuración del store (generalmente store/index.js)
✓ <Provider store={store}> envuelve el componente raíz

Si 3+ de estos son verdaderos → La app definitivamente usa Redux

📁 Estructura de Proyecto: Redux vs Context API

Acá está cómo se ven dos apps que usan diferentes soluciones de state management:

Estructura de App con Redux

my-redux-app/
├── src/
│   ├── store/
│   │   ├── index.js              // Setup del store con configureStore()
│   │   ├── slices/
│   │   │   ├── authSlice.js      // Contiene: state, reducers, actions
│   │   │   ├── userSlice.js
│   │   │   └── notificationsSlice.js
│   │   └── middleware/
│   │       └── logger.js
│   ├── components/
│   │   ├── Header.jsx            // Usa useSelector + useDispatch
│   │   ├── Dashboard.jsx
│   │   └── UserProfile.jsx
│   ├── pages/
│   │   ├── Home.jsx
│   │   └── Settings.jsx
│   ├── main.jsx                  // <Provider store={store}>
│   └── App.jsx
├── package.json                  // Tiene: redux, react-redux, @reduxjs/toolkit

Estructura de App con Context API

my-context-app/
├── src/
│   ├── context/
│   │   ├── AuthContext.jsx       // createContext + AuthProvider
│   │   ├── UserContext.jsx
│   │   └── NotificationsContext.jsx
│   ├── hooks/
│   │   ├── useAuth.js            // Hook personalizado envolviendo useContext
│   │   ├── useUser.js
│   │   └── useNotifications.js
│   ├── components/
│   │   ├── Header.jsx            // Usa useAuth() hook personalizado
│   │   ├── Dashboard.jsx
│   │   └── UserProfile.jsx
│   ├── providers/
│   │   └── AppProviders.jsx      // Envuelve todos los context providers
│   ├── main.jsx                  // <AppProviders>
│   └── App.jsx
├── package.json                  // Sin dependencias de Redux

🆚 Comparación de Flujo de Datos

Flujo de Datos en Redux

Componente
    │
    ├─ useDispatch() → dispatch(action)
    │
    ▼
Redux Store (el reducer procesa la acción)
    │
    ├─ Nuevo estado creado
    │
    ▼
Componente se suscribe con useSelector()
    │
    ├─ Re-render con el nuevo estado
    │
    ▼
La UI se actualiza

Flujo de Datos en Context API

Componente
    │
    ├─ useContext(MiContext)
    │
    ▼
Provider (tiene el estado actual)
    │
    ├─ Retorna el valor del contexto
    │
    ▼
Componente recibe estado + funciones
    │
    ├─ Llama a setState o función de acción
    │
    ▼
Provider re-renderiza todos los consumidores
    │
    ├─ Los componentes usando ese contexto se re-renderizan
    │
    ▼
La UI se actualiza

🆚 Tabla Comparativa de Características

Característica Redux Context API
Complejidad Setup Alta (store, slices, actions) Baja (context + provider)
Actualizar Estado dispatch(action) → reducer setState() o callbacks
DevTools Excelente (Redux DevTools time travel) Limitado (solo browser DevTools)
Soporte Middleware Sí (thunk, saga, etc.) No incorporado
Performance Optimizado (suscripciones granulares) Bueno (puede causar re-renders innecesarios)
Curva de Aprendizaje Empinada Suave
Mejor Para Apps grandes y complejas Apps pequeñas a medianas
Debugging Time travel, logs de acciones Logging manual

💡 ¿Cuándo Usar Cada Uno?

Usa Redux si:

✅ Tu app tiene estado complejo y anidado
✅ Múltiples componentes necesitan los mismos datos
✅ Necesitás state management predecible y centralizado
✅ Querés DevTools para debugging
✅ Tu equipo conoce los patrones de Redux
✅ Necesitás middleware para operaciones async

Usa Context API si:

✅ Tu app es pequeña a mediana
✅ El estado no cambia muy frecuentemente
✅ Necesitás compartir estado local simple
✅ Querés setup mínimo y pocas dependencias
✅ Es perfecto para theme, lenguaje, auth (configuración global)
