Integrando Filtros con la API

🔎 El problema inicial

La app ya hace una llamada HTTP a la API y los filtros se actualizan en el estado correctamente. Pero hay un problema crítico: la petición no envía esos filtros.

Los filtros se pasan al formulario, se actualizan cuando el usuario interactúa, pero esa información nunca llega a la URL de la petición. La API recibe siempre la misma consulta sin filtros. 

// ❌ La petición ignora completamente los filtros del usuario
useEffect(() => {
  fetch('https://jscamp-api.vercel.app/api/jobs')
    .then((res) => res.json())
    .then((data) => setJobs(data.data))
}, []) // ← nunca se re-ejecuta, nunca envía filtros

🎯 Objetivo

Cuando el usuario escribe "JavaScript" o cambia cualquier filtro (tecnología, tipo, nivel de experiencia), la app debe volver a consultar la API enviando esos filtros como query params.

🔧 Cómo enviar filtros a la API

La API acepta filtros en la URL mediante query params: 

GET https://jscamp-api.vercel.app/api/jobs?text=react&limit=10&offset=0
Parámetro Descripción
text Búsqueda libre de texto
technology Filtro por tecnología (javascript, python…)
type Filtro por modalidad (remote, onsite, hybrid)
level Nivel de experiencia (junior, mid, senior)

🛠️ URLSearchParams — el estándar de la plataforma web

Para construir los parámetros dinámicamente, usamos URLSearchParams, una API nativa del browser. No hace falta instalar nada. 

const params = new URLSearchParams()

// Solo se agregan si tienen valor
if (textToFilter)          params.append('text',       textToFilter)
if (filters.technology)    params.append('technology', filters.technology)
if (filters.location)      params.append('type',       filters.location)
if (filters.experienceLevel) params.append('level',    filters.experienceLevel)

const url = `https://jscamp-api.vercel.app/api/jobs?${params.toString()}`

Ventaja clave: si no hay ningún filtro activo, params.toString() devuelve una cadena vacía y la URL queda con un ? al final, lo cual funciona perfectamente.

🐛 Primer intento — por qué no funciona

Construís la URL con los filtros pero al seleccionar tecnología → JavaScript, no pasa nada. ¿Por qué? Porque el useEffect tiene el array de dependencias vacío: 

useEffect(() => {
  // Incluso si construís la URL con filtros acá...
  fetch(url)
    .then(...)
}, []) // ❌ Solo se ejecuta UNA VEZ al montar — nunca más

El linter también te va a advertir: "React Hook useEffect has missing dependencies". No lo ignorés — esa advertencia te está diciendo exactamente el problema.

✅ La solución: dependencias correctas

El efecto debe re-ejecutarse cada vez que cambia un filtro. Para eso, hay que listar cada filtro como dependencia: 

useEffect(() => {
  // ...lógica del fetch
}, [textToFilter, filters.technology, filters.location, filters.experienceLevel])
//  ↑ Ahora el efecto se re-ejecuta cuando cambia cualquiera de estos

🏗️ Implementación completa con filtros

El useEffect construye la URL dinámicamente con los filtros activos y se re-ejecuta cada vez que cambia alguno: 

useEffect(() => {
  const params = new URLSearchParams()

  if (textToFilter) {
    params.append('text', textToFilter)
  }
  if (filters.technology) {
    params.append('technology', filters.technology)
  }
  if (filters.location) {
    params.append('type', filters.location)
  }
  if (filters.experienceLevel) {
    params.append('level', filters.experienceLevel)
  }

  const queryParams = params.toString()

  fetch(`https://jscamp-api.vercel.app/api/jobs?${queryParams}`)
    .then((res) => res.json())
    .then((data) => setJobs(data.data))
}, [textToFilter, filters.technology, filters.location, filters.experienceLevel])

🔄 ¿Qué pasa cuando el usuario cambia un filtro?

// Flujo con filtros activos:

1. Usuario selecciona tecnología: "JavaScript"
   └─ filters.technology cambia → "javascript"

2. React detecta el cambio de dependencia
   └─ useEffect se re-ejecuta

3. URLSearchParams construye la URL:
   └─ ?technology=javascript

4. fetch() hace la nueva petición
   └─ https://jscamp-api.vercel.app/api/jobs?technology=javascript

5. API devuelve empleos filtrados
   └─ setJobs(data.data) → re-render con nuevos resultados

// Con múltiples filtros activos:
// ?text=javascript&type=remote&level=junior

🚫 Manejo del caso sin resultados

Cuando la combinación de filtros no encuentra nada, necesitás dar feedback al usuario. Sin esto, la pantalla queda vacía sin explicación: 

// En el JSX del componente:
{jobs.length === 0 ? (
  <p
    style={{
      padding: '2rem',
      textWrap: 'balance',
    }}
  >
    No se han encontrado empleos que coincidan con los criterios de búsqueda.
  </p>

📋 Comprobación en acción

// Pruebas que deberían funcionar:

✅ Texto: "Python"      → API recibe ?text=python
✅ Texto: "JavaScript"  → API recibe ?text=javascript
✅ Tecnología: React    → API recibe ?technology=react
✅ Tipo: Remoto         → API recibe ?type=remote

// Combinaciones:
✅ JavaScript + Remoto  → ?technology=javascript&type=remote
✅ JS + Remoto + Junior → ?technology=javascript&type=remote&level=junior

📦 Componente completo con todo integrado

function JobSearch({ filters, textToFilter }) {
  const [jobs, setJobs] = useState([])
  const [loading, setLoading] = useState(true)
  const [total, setTotal] = useState(0)

  useEffect(() => {
    async function fetchJobs() {
      try {
        setLoading(true)

        const params = new URLSearchParams()
        if (textToFilter)              params.append('text',       textToFilter)
        if (filters.technology)        params.append('technology', filters.technology)
        if (filters.location)          params.append('type',       filters.location)
        if (filters.experienceLevel)   params.append('level',      filters.experienceLevel)

        const res = await fetch(
          `https://jscamp-api.vercel.app/api/jobs?${params.toString()}`
        )
        const json = await res.json()

        setJobs(json.data)
        setTotal(json.total)
      } catch (error) {
        console.error('Error al cargar empleos:', error)
      } finally {
        setLoading(false)
      }
    }

    fetchJobs()
  }, [textToFilter, filters.technology, filters.location, filters.experienceLevel])

  if (loading) return <div>Cargando empleos...</div>

  return (
    <>
      {jobs.length === 0 ? (
        <p style={{ padding: '2rem', textWrap: 'balance' }}>
          No se han encontrado empleos que coincidan con los criterios de búsqueda.
        </p>
      ) : (
        <JobsList jobs={jobs} />
      )}
    </>
  )
}

🐛 Error clásico: filtro seleccionado pero nada cambia

El síntoma: el usuario selecciona un filtro y la lista no se actualiza. La causa casi siempre es el array de dependencias vacío o incompleto. 

// ❌ Causa
useEffect(() => {
  fetch(`...?technology=${filters.technology}`)  // lee el filtro
    .then(...)
}, []) // pero nunca se re-ejecuta cuando cambia

// ✅ Solución
useEffect(() => {
  fetch(`...?technology=${filters.technology}`)
    .then(...)
}, [filters.technology]) // ahora sí se re-ejecuta al cambiar

🐛 Error: advertencia del linter sobre dependencias faltantes

Si ves "React Hook useEffect has missing dependencies: 'filters.technology'", no lo ignorés con un // eslint-disable-next-line. El linter tiene razón — esa variable debería estar en las dependencias.

Regla: Si una variable se usa dentro del useEffect, tiene que estar en el array de dependencias. Sin excepciones.

🐛 Error: loop infinito al agregar objeto como dependencia

Si ponés el objeto filters completo como dependencia en lugar de sus propiedades, podés generar un loop infinito: 

// ❌ Peligroso — si filters se recrea en cada render, loop infinito
useEffect(() => {
  // ...
}, [filters]) // ← objeto completo

// ✅ Seguro — valores primitivos
useEffect(() => {
  // ...
}, [filters.technology, filters.location, filters.experienceLevel])

Los primitivos (string, number, boolean) se comparan por valor. Los objetos se comparan por referencia — si se recrea el objeto en cada render, React ve una "dependencia nueva" siempre y el efecto se ejecuta infinitamente.

✅ Buenas prácticas y ❌ qué evitar

✅ Hacer siempre:

  • Listar todas las variables usadas dentro del efecto como dependencias
  • Usar URLSearchParams para construir query strings — no concatenación manual
  • Agregar dependencias de forma granular (primitivos, no objetos completos)
  • Mostrar feedback cuando no hay resultados
  • Respetar las advertencias del linter sobre dependencias faltantes

❌ Nunca hacer:

  • Silenciar advertencias del linter con // eslint-disable
  • Pasar el objeto de filtros completo como dependencia si se recrea en cada render
  • Construir URLs con concatenación de strings manual ('?text=' + text + '&type=' + type)
  • Dejar el array de dependencias vacío si el efecto usa valores del estado o props
