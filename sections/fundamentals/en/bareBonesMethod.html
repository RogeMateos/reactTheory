04.1 — The Bare Bones Method: React State Architecture

🎯 Objective of This Lesson

Learn to:

  • Design the structure first
  • Define the data flow correctly
  • Place state in the right location
  • Avoid classic mistakes such as:
    • Unnecessary prop drilling
    • Duplicated state
    • Poorly used global state

🧠 1. Mindset: Coder vs Engineer

Before writing code, think.

❌ Coder:

  • Starts with useState
  • Tries things until they work
  • Patches problems

✅ Engineer:

  • Defines structure
  • Analyzes flow
  • Decides architecture
  • Then implements

🏗 2. Step 1 — Component Structure

Never start with state.

First break the UI into independent pieces.

Example: IMDb-like App

Possible components:


<App>
  <Search />
  <ResultsPage>
    <MovieList>
      <MovieCard />
    </MovieList>
  </ResultsPage>
</App>

🔎 Key Question:

What visual parts have their own responsibility?

Senior Rule:

One component = one clear responsibility.

🔄 3. Step 2 — Data Flow

Once you have the structure, ask:

Who changes the state?

Example:

  • <Search /> changes the text
  • <MovieList /> uses that text
  • Both need the same value.

❌ Common Error (Junior)


function Search() {
  const [query, setQuery] = useState("")
}

Problem: MovieList can't access it.

✅ Correct Solution — Lift State Up

Move the state to the common parent.


function App() {
  const [query, setQuery] = useState("")

  return (
    <>
      <Search query={query} setQuery={setQuery} />
      <MovieList query={query} />
    </>
  )
}

Principle:

If two siblings need the same state → lift it to the common parent.

🧵 4. Props Drilling

What is it?

Passing props through multiple levels:


App
 └── ResultsPage
      └── MovieList
           └── MovieCard

If query is passed through all of them even though they don't use it → props drilling.

🤔 Is it always bad?

No.

  • ✔ 1–2 levels → acceptable
  • ❌ 4–5 levels → bad architecture

🌍 5. Global State

When to use it?

When the state:

  • Is needed in many parts
  • Doesn't change constantly
  • Represents something transversal

Examples:

  • Authentication
  • Language
  • Theme
  • User data

Example with Context


const AuthContext = createContext(null)

function App() {
  const [user, setUser] = useState(null)

  return (
    <AuthContext.Provider value={{ user, setUser }}>
      <Search />
      <ResultsPage />
    </AuthContext.Provider>
  )
}

Now any component can access it without prop drilling.

🧠 6. Reducer — When State Grows

If the state has multiple transitions:


const initialState = {
  status: "idle",
  data: [],
  error: null
}

Use useReducer.


function reducer(state, action) {
  switch (action.type) {
    case "FETCH_START":
      return { ...state, status: "loading" }

    case "FETCH_SUCCESS":
      return { ...state, status: "success", data: action.payload }

    case "FETCH_ERROR":
      return { ...state, status: "error" }

    default:
      return state
  }
}

This turns your UI into a mini state machine.

Professional level.

📜 7. Fundamental State Principles

1️⃣ Use the least state possible

❌ Don't save what you can derive.


const fullName = firstName + lastName

You don't need useState for that.

2️⃣ Keep state as close as possible

  • Modal open → local
  • Toggle → local
  • Auth → global

3️⃣ Avoid redundancy

❌ Bad:


const [movies, setMovies]
const [filteredMovies, setFilteredMovies]

✔ Better:


const filteredMovies = movies.filter(...)

Derived state ≠ real state.

🧩 8. Deciding What Type of State to Use

Situation Solution
Only one component uses it useState local
Two siblings need it Lift state up
Many parts use it Context / Store
Complex state useReducer
Advanced flow State machine

🏁 Complete Professional Workflow

Always follow this order:

  1. Component Structure
  2. Data Flow
  3. State Placement
  4. Logic
  5. Styling

If you reverse the order → technical debt.

✅ Final Checklist

Before finishing a feature, ask yourself:

  • ☐ Did I divide the components well?
  • ☐ Do I know who changes the state?
  • ☐ Is the state in the correct place?
  • ☐ Am I duplicating state?
  • ☐ Do I really need global state?
  • ☐ Could this state be derived?

If you answer well → you're thinking like an engineer.
