Zustand: Best Practices & Efficient Subscriptions

🎯 Quick Summary

This lesson explains how to use Zustand efficiently, avoiding unnecessary renders and keeping your application synchronized across different pages.

Three ideas to internalize:

  • ✅ Don't subscribe to the entire store if you don't need to.
  • ✅ Always subscribe to the state that affects the UI.
  • ✅ On logout, changing isLogged alone is not enough — you must also clean all related user state.

❌ Problem 1: Subscribing to the Entire Store

const store = useStore()

This makes the component listen to every single change in the store.


Store
│
├── isLogged
├── favorites
├── theme
├── user
└── filters
      │
      ▼
 Component

⬅️ Any change causes a re-render

Even if you only use store.isLogged, React will re-render the component whenever favorites, theme, or any other property changes.

✅ Solution: Subscribe Only to What You Need

const isLogged = useStore((state) => state.isLogged)

or:

const favorites = useStore((state) => state.favorites)

Now the component only listens to one slice of state.


Store
│
├── isLogged  ◀──── Component
├── favorites
├── theme
└── user

If theme changes → component doesn't react at all.

✔️ Fewer renders. ✔️ Better performance.

✅ Extracting Only Actions

const toggleFavorite = useStore(
  (state) => state.toggleFavorite
)

Actions usually keep the same reference. So toggleFavorite always points to the same function — it does NOT cause unnecessary re-renders.

⚠️ Problem 2: Subscribing Too Little

This is probably the most important concept in this lesson.

❌ Incorrect example:

const toggleFavorite = useStore(
  (state) => state.toggleFavorite
)

// And then somewhere else:
const favorite = isFavorite(id)

The component never listens to favorites changes.


Store
│
├── favorites  ❌ not subscribed
└── toggleFavorite ✅ subscribed

User changes a favorite
        │
        ▼
Component does NOT update

✅ Correct version:

const favorites = useStore((state) => state.favorites)
const toggleFavorite = useStore((state) => state.toggleFavorite)

// Now derive from the subscribed state:
const isFavorite = favorites.includes(id)

🧠 The Golden Rule of Subscriptions

Don't think:

"I want to subscribe to as little as possible."

Think:

"I want to subscribe to everything that can change what the user sees."

That approach keeps the app synchronized, avoids unnecessary renders, and makes global state predictable and easy to maintain.

🌐 Shared State Across Pages

Zustand's biggest strength is sharing state between pages or unrelated components without prop drilling.


          Zustand Store
        ┌─────────────────┐
        │                 │
        │ favorites       │
        │ isLogged        │
        │ user            │
        └─────────────────┘
           ▲           ▲
           │           │
     JobList      JobDetail
           ▲           ▲
           └─────┬─────┘
                 │
            FavoriteButton

When a favorite changes:


JobList
   │
toggleFavorite()
   │
   ▼
Store updates favorites
   │
   ▼
All subscribed components receive the change

No need to pass props between pages.

🚪 Problem 3: Incomplete Logout

A common mistake when logging out:

logout()
// isLogged = false ... but:

// favorites = [1, 4, 7]  ← still in memory
// cart = [...]            ← still in memory
// user = {...}            ← still in memory

When the next user logs in, they may find leftover data from the previous session.

✅ Clean logout:

logout: () =>
  set({
    isLogged: false,
    user: null,
    favorites: [],
  })

Or even better, use an initial state object:

const initialState = {
  isLogged: false,
  user: null,
  favorites: [],
}

logout: () => set(initialState)

Before (incomplete):
logout → only changes isLogged
❌ "Zombie" state persists

After (clean):
logout → resets all related state
✅ Clean state
✅ Consistent UI
✅ No stale data

📋 Summary

  • ✅ Use selectors: useStore((state) => state.x)
  • ✅ Subscribe to all state that drives what the user sees
  • ✅ Extract actions separately — they don't cause renders
  • ✅ Derive computed values from subscribed state
  • ✅ On logout, reset ALL user-related state, not just isLogged
  • ✅ Use initialState objects for safe, predictable resets

❌ Anti-pattern: Full Store Subscription

// ❌ This re-renders on ANY store change
function JobCard({ job }) {
  const store = useStore()

  return (
    <button onClick={() => store.toggleFavorite(job.id)}>
      {store.isFavorite(job.id) ? '⭐' : '☆'}
    </button>
  )
}

✅ Pattern: Selective Subscriptions

// ✅ Re-renders only when favorites or toggleFavorite change
function JobCard({ job }) {
  const favorites = useStore((state) => state.favorites)
  const toggleFavorite = useStore((state) => state.toggleFavorite)

  const isFav = favorites.includes(job.id)

  return (
    <button onClick={() => toggleFavorite(job.id)}>
      {isFav ? '⭐' : '☆'}
    </button>
  )
}

✅ Pattern: Subscribing to Multiple Related Properties

// When properties are always used together, you can group them
const { isLogged, user } = useAuthStore((state) => ({
  isLogged: state.isLogged,
  user: state.user,
}))

Use this when properties always appear together and are logically related. Avoid it if it creates a new object on every render with unrelated fields.

✅ Pattern: Clean Auth Store with Logout

import { create } from 'zustand'

const initialState = {
  isLogged: false,
  user: null,
  favorites: [],
}

export const useAuthStore = create((set) => ({
  ...initialState,

  login: (userData) =>
    set({
      isLogged: true,
      user: userData,
    }),

  // ✅ Full reset — no zombie state
  logout: () => set(initialState),
}))

✅ Pattern: Deriving Values from Subscribed State

function FavoriteButton({ jobId }) {
  // ✅ Subscribe to the array, then derive the boolean
  const favorites = useStore((state) => state.favorites)
  const toggleFavorite = useStore((state) => state.toggleFavorite)

  const isFav = favorites.includes(jobId)

  return (
    <button
      className={isFav ? 'btn-active' : 'btn'}
      onClick={() => toggleFavorite(jobId)}
    >
      {isFav ? '⭐ Remove' : '☆ Add to Favorites'}
    </button>
  )
}

✅ Pattern: Header Component Using Auth

function Header() {
  // Only re-renders when isLogged or user changes
  const isLogged = useAuthStore((state) => state.isLogged)
  const user = useAuthStore((state) => state.user)
  const logout = useAuthStore((state) => state.logout)

  return (
    <header>
      {isLogged ? (
        <>
          <span>Welcome, {user.name}</span>
          <button onClick={logout}>Logout</button>
        </>
      ) : (
        <a href="/login">Login</a>
      )}
    </header>
  )
}

🎬 Scenario: Multi-User App with Favorites

User A logs in and favorites some jobs


State after User A session:
  isLogged: true
  user: { id: 1, name: 'Alice' }
  favorites: [4, 7, 12]

❌ Incomplete logout (common mistake)

// ❌ Only flipping isLogged
logout: () => set({ isLogged: false })
    

State after this logout:
  isLogged: false       ← changed
  user: { id: 1, name: 'Alice' }  ← STILL THERE
  favorites: [4, 7, 12]           ← STILL THERE

User B logs in. They see Alice's favorites and user data. That is a bug — and a potential privacy issue.

✅ Clean logout

const initialState = {
  isLogged: false,
  user: null,
  favorites: [],
}

logout: () => set(initialState)
    

State after clean logout:
  isLogged: false   ✅
  user: null        ✅
  favorites: []     ✅

User B logs in to a completely fresh state.

🔁 Scenario: FavoriteButton Across Two Pages

JobList page and JobDetail page both show a FavoriteButton for the same job.

❌ Without subscribing to favorites:

function FavoriteButton({ jobId }) {
  const toggleFavorite = useStore((state) => state.toggleFavorite)

  // isFavorite is called but the component never re-renders when it changes
  const isFav = useStore.getState().isFavorite(jobId)

  return <button onClick={() => toggleFavorite(jobId)}>{isFav ? '⭐' : '☆'}</button>
}

User favorites job #5 on JobList
        │
        ▼
Store updates favorites: [5]
        │
        ▼
JobDetail's FavoriteButton → still shows ☆ (no re-render)

✅ With proper subscription:

function FavoriteButton({ jobId }) {
  const favorites = useStore((state) => state.favorites)
  const toggleFavorite = useStore((state) => state.toggleFavorite)

  const isFav = favorites.includes(jobId)

  return <button onClick={() => toggleFavorite(jobId)}>{isFav ? '⭐' : '☆'}</button>
}

User favorites job #5 on JobList
        │
        ▼
Store updates favorites: [5]
        │
        ▼
JobDetail's FavoriteButton → re-renders → shows ⭐

Same component, same store. State is shared automatically. No prop passing, no synchronization bugs.

💡 Key Takeaways

  • Selector = subscription: (state) => state.x means "re-render when x changes"
  • No selector = subscribed to everything: never do this in real apps
  • Actions are stable: extracting functions doesn't cause extra renders
  • Derive, don't duplicate: compute booleans from the subscribed array
  • Logout = full reset: use an initialState object and reset it all
  • Shared state is free: any component on any page can subscribe to the same store
