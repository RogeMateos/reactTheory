API Call: Consuming Real Data with fetch

📦 The problem: static data vs dynamic data

Until now, data came from a local .json file that never changes. In real applications, data lives on a server: it gets updated, filtered in the backend, paginated. Using static data doesn't scale and doesn't represent how professional apps actually work. 

// ❌ Static data — never changes
import jobsData from './data/jobs.json'

function App() {
  const [jobs] = useState(jobsData)
  // ... data frozen in time
}
// ✅ Dynamic data from an API
function App() {
  const [jobs, setJobs] = useState([])
  const [loading, setLoading] = useState(true)

  useEffect(() => {
    fetch('/api/jobs')
      .then((res) => res.json())
      .then((data) => {
        setJobs(data)
        setLoading(false)
      })
  }, [])

  if (loading) return <div>Loading jobs...</div>

  return jobs.map((job) => <JobCard key={job.id} job={job} />)
}

🚫 Why can't you fetch directly in the render?

React runs the component body on every render. If you put fetch() directly there, you cause an infinite loop: 

// ❌ NEVER do this — infinite loop guaranteed
function JobSearch() {
  const [jobs, setJobs] = useState([])

  fetch('/api/jobs')
    .then((res) => res.json())
    .then((data) => setJobs(data)) // ← re-render → fetch() → re-render → ...

  return <JobListing jobs={jobs} />
}
// The problem in detail:
// 1. Component renders
// 2. fetch() executes
// 3. Response arrives → setJobs()
// 4. setJobs() triggers a re-render
// 5. Back to step 1 → Infinite loop! 💥

✅ The solution: useEffect with empty dependencies

useEffect with an empty dependency array [] runs only once, when the component mounts. It's the right place for side effects like API calls. 

function JobSearch() {
  const [jobs, setJobs] = useState([])

  useEffect(() => {
    fetch('https://jscamp-api.vercel.app/api/jobs')
      .then((res) => res.json())
      .then((data) => setJobs(data))
  }, []) // ← [] = only on component mount

  return <JobListing jobs={jobs} />
}

🧠 Preparing the required states

To handle an API call properly, you need at least three states: 

function JobSearch() {
  // 1. Array with jobs (starts empty)
  const [jobs, setJobs] = useState([])

  // 2. Boolean to show loading feedback
  const [loading, setLoading] = useState(true)

  // 3. Total results for pagination
  const [total, setTotal] = useState(0)
}

Why does loading start as true? Because before the component mounts, we already know we're going to load data. Starting at true avoids showing an unexplained blank screen to the user.

🏗️ Complete fetch implementation

Here's the API call implemented correctly with async/await, error handling, and loading state: 

function JobSearch() {
  const [jobs, setJobs] = useState([])
  const [loading, setLoading] = useState(true)
  const [total, setTotal] = useState(0)

  useEffect(() => {
    // async function INSIDE the effect (not the callback directly)
    async function fetchJobs() {
      try {
        setLoading(true)

        const response = await fetch('https://jscamp-api.vercel.app/api/jobs')
        const json = await response.json()

        setJobs(json.data)   // only the array, not the full object
        setTotal(json.total) // total results for pagination
      } catch (error) {
        console.error('Error loading jobs:', error)
      } finally {
        setLoading(false) // always runs, whether there's an error or not
      }
    }

    fetchJobs()
  }, []) // empty deps = only on mount

  if (loading) return <div>Loading jobs...</div>

  return <JobListing jobs={jobs} total={total} />
}

Why not make the useEffect callback async directly?
Because useEffect expects its callback to return either undefined or a cleanup function. An async function returns a Promise, which breaks the useEffect contract. That's why we declare the async function inside and call it.

// That's why we do this:
setJobs(json.data)   // ✅ array of jobs

🔄 The complete loading cycle

// Step by step flow:

1. Component mounts
   ├─ loading = true
   └─ jobs = []

2. useEffect runs
   └─ Calls fetchJobs()

3. fetchJobs() makes the request
   └─ User sees: "Loading jobs..."
   └─ Waiting for server response...

4. API responds
   ├─ setJobs(json.data)
   ├─ setTotal(json.total)
   └─ setLoading(false)  ← in the finally block

5. React re-renders
   ├─ loading = false
   └─ JobListing renders with real jobs

⏱️ Simulating a slow connection (dev only)

Add an artificial delay to properly see the loading state. Remember to remove it before going to production. 

useEffect(() => {
  async function fetchJobs() {

      setLoading(true)

      // 5 second delay — for visual testing only
      await new Promise((resolve) => setTimeout(resolve, 5000))

if (loading) {

🐛 Classic error: jobs.map is not a function

This is the most common error when consuming an API for the first time. It appears when you store the full response object instead of the array. 

// ❌ Cause of the error
const response = await fetch('/api/jobs')
const json = await response.json()

setJobs(json) // ← storing the OBJECT, not the array

// Later in render:
jobs.map(job => ...) // TypeError: jobs.map is not a function
// ✅ Solution
setJobs(json.data)   // only the array
setTotal(json.total) // the total separately

Golden rule: Before doing setJobs(x), ask yourself: "Is x an array?" If it's not, navigate to the array inside the object first.

🔍 Verifying with DevTools

To confirm data is coming from the API correctly:

  • Open DevTools → Network tab
  • Reload the page
  • Find the request to /api/jobs
  • Verify the Status Code is 200 OK
  • In Response you should see the JSON with total, data, etc.

You can also throttle the network to Slow 3G in DevTools to see the loading state without an artificial delay.

🧠 Reflection: what logic goes in which custom hook?

A hook should represent something concrete and cohesive. Don't mix unrelated logic: 

// ✅ Good: each hook has a single responsibility
function useFilters() {
  const [filters, setFilters] = useState({ text: '', type: '', location: '' })
  const [page, setPage] = useState(1)

  const updateFilter = (name, value) =>
    setFilters((prev) => ({ ...prev, [name]: value }))

  const nextPage = () => setPage((prev) => prev + 1)
  const prevPage = () => setPage((prev) => Math.max(1, prev - 1))

  return { filters, updateFilter, page, nextPage, prevPage }
}

function useDocumentTitle(title) {
  useEffect(() => {
    document.title = title
  }, [title])
}
// ❌ Bad: mixed responsibilities
function useFilters() {
  const [filters, setFilters] = useState({})

  // This does NOT belong here — the title has nothing to do with filters
  useEffect(() => {
    document.title = `Jobs - Filters applied`
  }, [filters])

  return { filters, setFilters }
}

✅ Best practices and ❌ what to avoid

✅ Always do:

  • Use try-catch-finally in all async calls
  • Manage loading state to give feedback to the user
  • Separate the data array from the full response object
  • Define the async function inside the effect, not the callback directly
  • Separate responsibilities into distinct custom hooks

❌ Never do:

  • Call fetch() directly in the component body
  • Store the full API object in state when you need an array
  • Mix unrelated logic in the same hook
  • Ignore errors from requests
  • Leave artificial delays in production code
