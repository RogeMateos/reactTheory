Zustand: Global State Made Simple

🏙️ The City Analogy: From Prop Drilling to Global State

Imagine a city with houses (components) that need information from the town hall (state). As the city grows, information has to pass through many houses to reach the one that needs it. 

Town Hall (Global State)
   │
   ├─ Information flows down through pipes
   │
   House 1
   │
   House 2
   │
   House 3 ✅ Finally gets the info, but it took 3 houses!

This is PROP DRILLING — painful and inefficient.

🚨 The Prop Drilling Problem

Without global state, you pass props through intermediate components that don't even use them: 

// ❌ Prop Drilling: App → Page → Card → Button
function App() {
  const [user, setUser] = useState({ name: 'Alex', isLoggedIn: true })
  return <Page user={user} />
}

function Page({ user }) {
  return <Card user={user} />  // doesn't use user!
}

function Card({ user }) {
  return <Button user={user} />  // doesn't use user!
}

function Button({ user }) {
  return <button>Welcome, {user.name}</button>  // finally uses it
}

⚠️ The pain: If you need to change where user comes from, you have to update App, Page, Card, AND Button. Add 10 levels of nesting? Now it's a nightmare.

🏪 Context API: A Central Supermarket

React's built-in solution is Context. You create a provider at the top that makes data available to all children without passing props: 

// Create the context
const UserContext = createContext()

// Provider wraps the app
function App() {
  const [user, setUser] = useState({ name: 'Alex' })
  return (
    <UserContext.Provider value={{ user, setUser }}>
      <Page />
    </UserContext.Provider>
  )
}

// Any child can consume it
function Button() {
  const { user } = useContext(UserContext)
  return <button>Welcome, {user.name}</button>  // ✅ Direct access!
}

Context solves prop drilling, BUT...

⚡ The Context Problem: Unnecessary Re-Renders

Context has a hidden cost. When ANY value in the provider changes, ALL components that use that context re-render, even if they only use one small piece of data. 

Context Logic:
  1. User logs in
  2. UserContext updates
  3. MyComponent watches UserContext
  4. MyComponent re-renders ✅
  5. ThemeComponent watches UserContext
  6. ThemeComponent re-renders ❌ (didn't need this!)
  7. NotificationComponent watches UserContext
  8. NotificationComponent re-renders ❌ (didn't need this either!)

❌ The issue: If 100 components use the same Context, and one tiny value changes, all 100 re-render. This kills performance on large apps.

🔑 Zustand: The Smart Key

Here's where Zustand comes in. It's like replacing the receptionist (Context) with smart keys. Each component only subscribes to the data it actually uses. 

Context = Receptionist
  "Who needs info?" she asks everyone
  Everyone listens to her answer
  Even if you don't need the info, you listen

Zustand = Smart Key
  You pick the exact info you need
  Only you get notified when YOUR data changes
  Nobody else cares

Zustand is a state management library that gives you global state with minimal re-renders and zero boilerplate.

📦 Key Differences at a Glance

PROPS
└─ Unidirectional data flow (parent → child)
└─ Fine for: Simple apps, single-level nesting
└─ Problem: Prop drilling with deep nesting

CONTEXT API
└─ Global state with Provider/Consumer pattern
└─ Fine for: Themes, languages, simple config
└─ Problem: Unnecessary re-renders of all subscribers

ZUSTAND ✅
└─ Global state with subscriptions
└─ Fine for: Any app, any size, complex state
└─ Benefit: Granular updates (only affected components re-render)
└─ Benefit: No Provider needed (optional)
└─ Benefit: Tiny bundle size (2.3 kB)

⚙️ How Zustand Works Under the Hood

1. You create a store with create()
2. The store is a hook (useAuthStore, useCartStore, etc.)
3. Your component calls that hook: const user = useAuthStore(...)
4. Component subscribes to specific state
5. When state changes, Zustand notifies ONLY subscribers of that part
6. Component re-renders ONLY if its subscribed state changed

This is way more efficient than Context.

✅ Why Developers Choose Zustand

✨ Minimal boilerplate — fewer lines than Redux or Context
⚡ Granular subscriptions — only re-render when you need to
🔒 Type-safe — works beautifully with TypeScript
🪶 Tiny bundle — 2.3 kB (smaller than Redux by far)
💾 Persistent — easy to save to localStorage
🧪 Easy to test — stores are just functions
🎯 No Provider hell — optional, but you can wrap if needed

🚀 Installation

npm install zustand

That's it. No configuration, no providers required (unless you want them).

📖 Creating Your First Store (Step by Step)

Here's the simplest possible Zustand store. We'll build a user authentication store: 

import { create } from 'zustand'

// Step 1: Use create() to define your store
const useAuthStore = create((set) => ({
  // Step 2: Define your initial state
  user: null,
  isLoggedIn: false,

  // Step 3: Define actions (functions that change state)
  login: (userData) => set({ user: userData, isLoggedIn: true }),
  logout: () => set({ user: null, isLoggedIn: false }),
}))

// That's it! useAuthStore is now a React hook

Breaking it down:

create()           → Zustand function to create a store
(set) => ({})      → Arrow function that receives the "set" function
set                → How you update state (updates the whole store)
user, isLoggedIn   → Your state
login, logout      → Your actions

🎣 Using the Store in a Component

// ✅ Use it like a hook! No Provider needed
function LoginButton() {
  const { user, login, logout, isLoggedIn } = useAuthStore()

  return (
    <div>
      {isLoggedIn ? (
        <>
          <p>Welcome, {user.name}</p>
          <button onClick={logout}>Logout</button>
        </>
      ) : (
        <button onClick={() => login({ name: 'Alex', email: 'alex@example.com' })}>
          Login
        </button>
      )}
    </div>
  )
}

🎯 Key insight: Notice that LoginButton is a regular component. No Context.Provider, no complex setup. It just uses the hook.

🎯 Selective Subscriptions (The Smart Part)

This is where Zustand shines. You can subscribe to ONLY the parts of state you need. If you only need isLoggedIn, only re-render when THAT changes: 

// ❌ Subscribe to whole store (re-renders on ANY change)
function MyComponent() {
  const state = useAuthStore()  // Whole store
  return <div>{state.isLoggedIn}</div>
}

// ✅ Subscribe to specific slice (re-renders ONLY on this change)
function MyComponent() {
  const isLoggedIn = useAuthStore((state) => state.isLoggedIn)
  return <div>{isLoggedIn}</div>
}

// ✅ Subscribe to multiple specific slices
function MyComponent() {
  const { isLoggedIn, user } = useAuthStore(
    (state) => ({ isLoggedIn: state.isLoggedIn, user: state.user })
  )
  return <div>{user.name} — {isLoggedIn}</div>
}

💡 This is performance magic: Component A only re-renders when isLoggedIn changes. Component B only re-renders when user changes. They don't interfere with each other.

🛒 Real-World Example: Shopping Cart Store

import { create } from 'zustand'

const useCartStore = create((set) => ({
  // State
  items: [],
  totalPrice: 0,

  // Actions
  addItem: (product) => set((state) => ({
    items: [...state.items, product],
    totalPrice: state.totalPrice + product.price,
  })),

  removeItem: (productId) => set((state) => {
    const removedProduct = state.items.find(item => item.id === productId)
    return {
      items: state.items.filter(item => item.id !== productId),
      totalPrice: state.totalPrice - (removedProduct?.price || 0),
    }
  }),

  clearCart: () => set({ items: [], totalPrice: 0 }),

  getItemCount: () => set(
    (state) => state.items.length
  ),
}))

// Usage in components
function CartWidget() {
  const itemCount = useCartStore((state) => state.items.length)
  return <div>Cart: {itemCount} items</div>
}

function CartTotal() {
  const totalPrice = useCartStore((state) => state.totalPrice)
  return <div>Total: €{totalPrice}</div>
}

🔄 Zustand vs Context Side-by-Side

Same feature, both solutions:

// ============ CONTEXT VERSION ============
const UserContext = createContext()

function App() {
  const [user, setUser] = useState(null)
  return (
    <UserContext.Provider value={{ user, setUser }}>
      <Page />
    </UserContext.Provider>
  )
}

function MyComponent() {
  const { user, setUser } = useContext(UserContext)
  return <div>{user?.name}</div>
}

// ============ ZUSTAND VERSION ============
const useUserStore = create((set) => ({
  user: null,
  setUser: (userData) => set({ user: userData }),
}))

function App() {
  return <Page />  // No Provider needed!
}

function MyComponent() {
  const { user } = useUserStore()
  return <div>{user?.name}</div>
}

The Zustand version is cleaner, simpler, and more efficient.

💾 Persist State to localStorage

Zustand has middleware for persistence. You can save your store to localStorage and restore it on app reload: 

import { create } from 'zustand'
import { persist } from 'zustand/middleware'

const useAuthStore = create(
  persist(
    (set) => ({
      user: null,
      login: (userData) => set({ user: userData }),
      logout: () => set({ user: null }),
    }),
    {
      name: 'auth-storage',  // localStorage key
    }
  )
)

// Now login/logout data persists across page refreshes!

🧪 Zustand Stores are Easy to Test

import { useAuthStore } from './stores/auth'

describe('AuthStore', () => {
  // Zustand stores are plain functions, so testing is straightforward
  it('should login and update user', () => {
    const store = useAuthStore.getState()
    store.login({ name: 'Alex' })

    expect(store.user.name).toBe('Alex')
    expect(store.isLoggedIn).toBe(true)
  })

  it('should logout', () => {
    const store = useAuthStore.getState()
    store.logout()

    expect(store.user).toBe(null)
    expect(store.isLoggedIn).toBe(false)
  })
})

🎯 Decision Tree: Which State Solution to Use

Does your state live in ONE component?
├─ YES → useState ✅
└─ NO, multiple components need it
   │
   ├─ Simple state (theme, language)?
   │  └─ Context API ✅
   │
   └─ Complex state (user, cart, notifications)?
      ├─ Many re-renders a concern?
      │  └─ Zustand ✅✅
      └─ Just a few consumers?
         └─ Context API ⚠️ (works, but Zustand is simpler)

✅ Use useState When:

• State lives in a single component
• Only that component and immediate children use it

Example: Toggle button state, form input state, modal open/close

✅ Use Context API When:

• Simple, rarely-changing configuration
• Theme (dark mode / light mode)
• Language selection (EN / ES)
• User's accessibility preferences
• Small number of subscribers (5-10 components max)

Why: Context is built into React, zero dependencies,
     and overkill state is not a problem for simple data.

✅✅ Use Zustand When:

• Complex application state
  - Authenticated user info
  - Shopping cart
  - Notifications
  - Favorites
  - Filters & search

• Performance matters
  - Many components subscribe to state
  - Frequent updates
  - Need granular re-renders

• You want clean code
  - No Provider wrapper
  - No useContext(...) everywhere
  - Simple actions

Why: Zustand is smaller, faster, and way cleaner
     for anything beyond trivial state.

🚀 The Senior Developer's Rule of Thumb

Start with useState.

If you need to pass state through 3+ levels → Context.
If Context causes re-renders → Zustand.

In practice:
• 80% of projects use useState + Zustand (no Context needed)
• Context for styling/config only
• Zustand for everything else

The anti-pattern: Using Context for business logic.
That's where most developers go wrong.
