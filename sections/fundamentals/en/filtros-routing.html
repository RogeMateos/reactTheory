Automatic Filters & Routing

😤 The problem: filters that don't filter on their own

Think about the typical user experience: user changes a select → nothing happens. Changes another select → nothing happens. Types in the input → nothing happens. Finally clicks "Search" → only then the filters apply. Bad UX.

The solution: remove the submit button and make the form fire automatically on every field change using onChange on the <form> element itself, not on each individual field.

Tip: When you put onChange on the <form> (not on each field), React captures changes from ALL child fields with a single handler.

🎯 event.target vs event.currentTarget — the difference that matters

<form onChange={handleChange}>   ← has the listener
  <input name="search" />        ← can fire the event
  <select name="location">       ← can fire the event
</form>

event.target       = the field that CHANGED (input or select)
event.currentTarget = the element that HAS the listener (always the form)

Why does this difference matter? Because FormData needs the entire form, not just the field that changed. If you pass event.target (the select that changed), FormData can only read THAT element, not all the fields.

⚠️ Very common mistake: This is one of the most common errors with forms in React. Always use event.currentTarget when the handler is on the <form> and you need to read all fields.

🗂️ Manual routing: pages without React Router

A real app needs multiple pages: Home (/), Search (/search), 404. Without React Router, we can use window.location.pathname — a native browser property that returns the current URL path. 

// navigate to: http://localhost:5173/
window.location.pathname  →  "/"

// navigate to: http://localhost:5173/search
window.location.pathname  →  "/search"

// navigate to: http://localhost:5173/anything-else
window.location.pathname  →  "/anything-else"

The idea: read this value in App.jsx and render different page components based on it. Like a big if/else that decides what to show.

📁 Separating pages from components

The architectural difference is important:

  • Components (src/components/) = reusable pieces: Header, SearchForm, JobCard
  • Pages (src/pages/) = full views that the router shows: Home, Search, NotFound
src/
├── components/    ← reusable pieces
│   ├── Header.jsx
│   ├── SearchForm.jsx
│   └── JobListings.jsx
└── pages/         ← full views
    ├── Home.jsx
    ├── Search.jsx
    └── NotFound.jsx

App.jsx becomes only the layout shell plus the routing decision. All search logic moves into Search.jsx.

⚠️ The limitation of manual routing

Let's be honest: links with <a href="..."> cause a full page reload. This:

  • Loses React state
  • Is slow
  • Doesn't feel like a real SPA

This is educational — to understand HOW routing works internally. In production, always use React Router (next topic). 

Feature              Manual Router    React Router
─────────────────────────────────────────────────
Reloads the page     ❌ Yes           ✅ No
Preserves state      ❌ No            ✅ Yes
Dynamic routes       ❌ Hard          ✅ Easy
Production           ❌ No            ✅ Recommended

🔧 SearchForm without a button: onChange on the form

import { useId } from 'react'

export function SearchForm({ onSearch }) {
  const idText = useId()
  const idLocation = useId()

  const handleChange = (event) => {
    // currentTarget = the <form>, not the field that changed
    const formData = new FormData(event.currentTarget)
    const filters = {
      search: formData.get(idText),
      location: formData.get(idLocation),
    }
    onSearch(filters)
  }

  return (
    <form className="search-form" onChange={handleChange}>
      <input
        type="text"
        name={idText}
        id={idText}
        placeholder="Search..."
      />

      <select name={idLocation} id={idLocation}>
        <option value="">All locations</option>
        <option value="remote">Remote</option>
        <option value="on-site">On-site</option>
      </select>
      {/* No submit button — the form's onChange handles it automatically */}
    </form>
  )
}

No <button type="submit"> needed. The form fires on every field change, automatically.

🔍 target vs currentTarget in action

// ❌ WRONG — target is the <select> that changed
const handleChange = (event) => {
  const formData = new FormData(event.target)
  console.log(formData.get('search'))    // null ❌
  console.log(formData.get('location'))  // null ❌
}

// ✅ CORRECT — currentTarget is always the <form>
const handleChange = (event) => {
  const formData = new FormData(event.currentTarget)
  console.log(formData.get('search'))    // "react" ✅
  console.log(formData.get('location'))  // "remote" ✅
}

🏠 Manual router in App.jsx

import { Header } from './components/Header'
import { Footer } from './components/Footer'
import { Home } from './pages/Home'
import { Search } from './pages/Search'
import { NotFound } from './pages/NotFound'

function App() {
  const { pathname } = window.location

  let Page
  if (pathname === '/') {
    Page = Home
  } else if (pathname === '/search') {
    Page = Search
  } else {
    Page = NotFound
  }

  return (
    <>
      <Header />
      <Page />
      <Footer />
    </>
  )
}

export default App

🔄 Alternative version with &&

function App() {
  const { pathname } = window.location

  return (
    <>
      <Header />
      {pathname === '/' && <Home />}
      {pathname === '/search' && <Search />}
      {pathname !== '/' && pathname !== '/search' && <NotFound />}
      <Footer />
    </>
  )
}

Tip: Both versions do the same thing. The first (if/else) is more readable when there are many routes. The second (&&) is more compact but gets hard to read with many conditions.

⚠️ Common mistakes: HTML → JSX

// 1. Unclosed inputs
❌  <input type="text">
✅  <input type="text" />

// 2. class → className
❌  <div class="container">
✅  <div className="container">

// 3. for → htmlFor
❌  <label for="search">
✅  <label htmlFor="search">

// 4. Events in camelCase
❌  <button onclick={fn}>
✅  <button onClick={fn}>

// 5. Comments
❌  <!-- comment -->
✅  {/* comment */}

🐛 Submit button left by accident

// ❌ If you leave the button, the form submits and reloads the page
<form onChange={handleChange}>
  <input name="search" />
  <button type="submit">Search</button> {/* ← REMOVE THIS */}
</form>

// ✅ No button, just onChange on the form
<form onChange={handleChange}>
  <input name="search" />
  {/* No button — automatic */}
</form>
