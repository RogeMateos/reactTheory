URL Synchronization with the Search Engine

🧠 What does this pattern solve?

Making the search state (text, filters, and current page) synchronize with the URL so the search is:

Benefit What it means
Shareable You can send the URL and the other person sees exactly the same thing
Persistent after F5 Reloading the page doesn't clear the filters
Multi-tab compatible Each tab can have its own search state
Browser history The back button works as expected

🎯 The goal

You have two sources of information that must always have the same data:

React State           URL
──────────────        ────────────────────────────────
text = React          /search?text=React
filter = JavaScript   &technology=javascript
page = 2              &page=2

⚠️ The initial problem

The search engine only updated results but never touched the URL:

useEffect(() => {
  fetchJobs({ filters, textToFilter, currentPage })
}, [filters, textToFilter, currentPage])

This works for loading data, but:

❌ Doesn't update the URL
❌ If you press F5, you lose everything
❌ You can't share the search

🔄 Two separate synchronizations

The pattern actually implements two separate flows:

1. React → URL
   User types "React"
   → State changes
   → URL changes

2. URL → React
   URL contains ?text=React&page=2
   → React reads the URL
   → Initializes the state

🔧 Block 1: writing to the URL (React → URL)

Every time filters, text or page change, a useEffect updates the URL:

useEffect(() => {
  const params = new URLSearchParams()

  if (textToFilter) params.set('text', textToFilter)
  if (filters.technology) params.set('technology', filters.technology)
  if (currentPage > 1) params.set('page', String(currentPage))

  const newURL = `${window.location.pathname}?${params.toString()}`
  window.history.pushState({}, '', newURL)
}, [filters, textToFilter, currentPage])

Visually:

User changes filter
        ↓
   filters changes
        ↓
  useEffect fires
        ↓
  URLSearchParams builds
        ↓
  /search?technology=javascript

✅ Why use URLSearchParams instead of concatenation

// ✅ Correct — URLSearchParams encodes automatically
const params = new URLSearchParams()
params.set('text', 'React & TypeScript')
// Result: text=React+%26+TypeScript

// ❌ Dangerous — manual concatenation breaks with special characters
const url = '/search?text=' + text + '&technology=' + technology

🔧 Block 4: input synchronized with state

This is the easiest detail to forget. If React has textToFilter = 'React' but the input starts empty:

// ❌ Desynced input — shows results for "React" but the field is empty
const [text, setText] = useState('')

// ✅ Synced input — the field shows the same value as state
const [text, setText] = useState(initialTextToFilter)

This way the complete flow is:

URL → State → Input

Everything shows the same value.

🔧 Block 2: reading the URL on startup (URL → React)

If you always initialize state to 1, you're ignoring what the URL says: 

// ❌ Ignores the URL — always starts on page 1
const [currentPage, setCurrentPage] = useState(1)

// ✅ Reads the URL — starts on the correct page
const [currentPage, setCurrentPage] = useState(() => {
  const params = new URLSearchParams(window.location.search)
  const page = Number(params.get('page'))
  return Number.isNaN(page) || page < 1 ? 1 : page
})

⚡ Important React concept: Lazy Initialization

Many people don't understand the difference between these two forms:

// Form 1 — runs on EVERY render (even if the value doesn't change)
const [page, setPage] = useState(getPageFromURL())

// Form 2 — runs ONLY ONCE (lazy initialization)
const [page, setPage] = useState(() => getPageFromURL())

When you pass a function to useState, React runs it only on the first render. That's why it's called lazy initialization.

🛡️ Block 3: URL validation

Great practice. Someone could manually type in the URL:

?page=hello   → Number('hello') = NaN → breaks logic
?page=-500    → Invalid page → unexpected behavior
function getPageFromURL() {
  const params = new URLSearchParams(window.location.search)
  const page = Number(params.get('page'))

  if (Number.isNaN(page) || page < 1) return 1

  return page
}

💡 Improvement: reusable helper

The tutorial repeats new URLSearchParams(window.location.search) several times. It could be extracted:

// Before — repeated in multiple places
const params = new URLSearchParams(window.location.search)

// After — a clean helper
function getSearchParams() {
  return new URLSearchParams(window.location.search)
}

// Usage
const params = getSearchParams()

🎯 The interview answer

If in an interview you're asked: "How would you sync filters and pagination with the URL?"

  1. Store filters, text and page in React State.
  2. Use a useEffect to update the URL when state changes.
  3. Use URLSearchParams to build query params without manual concatenation.
  4. Initialize useState by reading URL parameters.
  5. Validate values coming from the URL with Number.isNaN and ranges.
  6. Keep inputs and filters synced with the correct initial state.
