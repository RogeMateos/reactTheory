Zustand: Persistence & Clean Logout

📌 Quick Summary

This tutorial explains a very common problem when using Zustand with persistence:

Logging out does NOT mean global state disappears.

If you don't clean the store, the next user may still see the previous user's favorites, cart, or preferences.

✅ Block 1 — Clearing Favorites on Logout

This works:

clearFavorites: () => set({ favorites: [] })

// On logout:
logout()
clearFavorites()

⚠️ The Problem

Tomorrow the store might have:

{
  favorites,
  filters,
  selectedJob,
  lastSearch,
  viewedJobs
}

Now you're calling:

clearFavorites()
clearFilters()
clearViewedJobs()
clearSelectedJob()
// ...

It becomes hard to maintain.

✅ Better version: define an initial state

const initialState = {
  favorites: []
}

export const useFavoritesStore = create((set) => ({
  ...initialState,

  addFavorite: () => {},

  reset: () => set(initialState)
}))

Before:


Logout
   │
   ├── clear favorites
   ├── clear filters
   ├── clear cart
   └── clear settings

After:


Logout
   │
   └── reset()
            │
            ▼
      Complete initial state

Much simpler.

✅ Block 2 — Saving the Initial State

⚠️ The Problem

Many developers write this:

reset: () =>
  set({
    favorites: []
  })

Months later they add new fields but forget to update the reset:


favorites   ✅ cleared
filters     ❌ still there
theme       ❌ still there

✅ Better version: single source of truth

const initialState = {
  favorites: [],
  filters: {},
  selectedJob: null
}

reset: () => set(initialState)

Now initialState is the only place to update when adding new fields.

✅ Block 3 — Logout

const handleLogout = () => {
  logout()
  resetFavorites()

  window.location.reload()
}

💡 Why reload?

Without reload:


React
 ├── UserStore
 ├── FavoritesStore
 ├── SearchStore
 ├── Cache
 └── Mounted components

Logout
     │
     ▼
Everything still lives in memory

With reload:


Logout
     │
     ▼
window.location.reload()
     │
     ▼
Fresh application

React
 ├── UserStore empty
 ├── Favorites empty
 ├── Cache clean
 └── Components restarted

Everything starts from scratch.

✅ Block 4 — Is Reloading a Good Practice?

Yes. Many large applications do it. Here's why:


User A
    │
Login
    │
Favorites
    │
Logout
    │
Reload
    │
User B Login
    │
Completely clean state

It's a very robust solution.

✅ Block 5 — Multiple Stores

Instead of:

logout()
resetFavorites()
resetCart()
resetFilters()
resetSettings()
resetSearch()
resetNotifications()

Centralize:

const resetAllStores = () => {
  useFavoritesStore.getState().reset()
  useCartStore.getState().reset()
  useFiltersStore.getState().reset()
}

const handleLogout = () => {
  logout()
  resetAllStores()

  window.location.reload()
}

🎯 Recommended Best Practices

✅ Keep a separate initialState:

const initialState = {
  favorites: []
}

✅ Every store should have a reset() method:

reset: () => set(initialState)

✅ Logout should be the single responsible party for cleaning the application:


handleLogout()
      │
      ├── delete token
      ├── reset stores
      ├── clear persisted state
      └── reload

🚀 Conclusion

The key insight:

Authentication and global state are separate responsibilities.

Logging out removes the user's identity, but does NOT automatically remove application state. That's why a professional logout implementation follows this flow:


User clicks Logout
          │
          ▼
Delete token/session
          │
          ▼
Reset all stores
          │
          ▼
Clear persisted state
          │
          ▼
Reload the application
          │
          ▼
React mounts everything from a clean state

This approach is simple, predictable, avoids zombie state, and scales much better as the application grows.

📋 Summary

  • ✅ Define initialState outside the store — single source of truth
  • ✅ Use reset: () => set(initialState) in every store
  • ✅ Centralize all resets in a resetAllStores() function
  • ✅ Call window.location.reload() after logout for a truly clean state
  • ✅ Logout owns the cleanup — one place, one responsibility

❌ Anti-pattern: Manual field cleanup

// Hard to maintain — you have to remember every field
const handleLogout = () => {
  logout()
  clearFavorites()
  clearFilters()
  clearCart()
  clearSettings()
  // Did I forget anything?
}

✅ Pattern: initialState + reset()

import { create } from 'zustand'
import { persist } from 'zustand/middleware'

const initialState = {
  favorites: [],
  filters: {},
  selectedJob: null,
}

export const useFavoritesStore = create(
  persist(
    (set) => ({
      ...initialState,

      addFavorite: (id) =>
        set((state) => ({
          favorites: [...state.favorites, id],
        })),

      removeFavorite: (id) =>
        set((state) => ({
          favorites: state.favorites.filter((f) => f !== id),
        })),

      // ✅ One reset to rule them all
      reset: () => set(initialState),
    }),
    { name: 'favorites-store' }
  )
)

✅ Pattern: Centralized resetAllStores

// utils/resetStores.js
import { useFavoritesStore } from '../stores/favoritesStore'
import { useCartStore } from '../stores/cartStore'
import { useFiltersStore } from '../stores/filtersStore'

export const resetAllStores = () => {
  useFavoritesStore.getState().reset()
  useCartStore.getState().reset()
  useFiltersStore.getState().reset()
}

✅ Pattern: Complete logout handler

import { useAuthStore } from '../auth/authStore'
import { resetAllStores } from '../utils/resetStores'

export function useLogout() {
  const logout = useAuthStore(state => state.logout)

  const handleLogout = () => {
    // 1. Clear auth (token, session)
    logout()

    // 2. Reset all stores
    resetAllStores()

    // 3. Clear persisted storage (if using persist middleware)
    localStorage.removeItem('favorites-store')
    localStorage.removeItem('cart-store')

    // 4. Full page reload — guaranteed clean slate
    window.location.reload()
  }

  return handleLogout
}

✅ Pattern: Using the logout hook in a component

import { useLogout } from '../hooks/useLogout'

export function Header() {
  const handleLogout = useLogout()

  return (
    <header>
      <button onClick={handleLogout}>Logout</button>
    </header>
  )
}

✅ Pattern: Auth store with initialState

import { create } from 'zustand'

const initialState = {
  isLoggedIn: false,
  user: null,
  token: null,
}

export const useAuthStore = create((set) => ({
  ...initialState,

  login: (userData, token) =>
    set({ isLoggedIn: true, user: userData, token }),

  logout: () => set(initialState),
}))

🎬 Scenario: Two Users on the Same Browser Session

User A logs in and adds favorites


Store (in memory + localStorage):
  isLoggedIn: true
  user: { id: 1, name: 'Alice' }
  favorites: [4, 7, 12]
  filters: { salary: '> 100k' }

❌ Incomplete logout (common mistake)

// Only the auth store is reset
logout()
// isLoggedIn = false

// But favorites store still has:
// favorites: [4, 7, 12]
// filters: { salary: '> 100k' }
// AND localStorage still has the old data

User B logs in


Store:
  isLoggedIn: true
  user: { id: 2, name: 'Bob' }
  favorites: [4, 7, 12]   ← Alice's favorites! 🐛
  filters: { salary: '> 100k' }  ← Alice's filters! 🐛

Bob sees Alice's data. That's a bug and a privacy issue.

✅ Correct logout flow

const handleLogout = () => {
  logout()             // reset auth store
  resetAllStores()     // reset all stores to initialState

  // Clear persisted localStorage entries
  localStorage.removeItem('favorites-store')

  window.location.reload()  // guaranteed fresh start
}

User B logs in — clean slate


Store:
  isLoggedIn: true
  user: { id: 2, name: 'Bob' }
  favorites: []    ✅
  filters: {}      ✅

🔁 Scenario: Store Grows Over Time

Version 1 of the store

const initialState = {
  favorites: [],
}
// reset() works perfectly

Version 2 — new fields added

// ❌ Forgot to update reset manually:
reset: () => set({ favorites: [] })
// filters and selectedJob are NOT reset!

// ✅ With initialState pattern:
const initialState = {
  favorites: [],
  filters: {},        // ← added
  selectedJob: null,  // ← added
}
// reset() automatically includes new fields
reset: () => set(initialState)

The initialState pattern scales with the store. You update one object, reset handles everything.

🗂️ Recommended Structure


src/
  stores/
    favoritesStore.js   ← initialState + reset()
    cartStore.js        ← initialState + reset()
    filtersStore.js     ← initialState + reset()

  utils/
    resetStores.js      ← resetAllStores()

  hooks/
    useLogout.js        ← handleLogout() — owns the full cleanup

  auth/
    authStore.js        ← initialState + logout()

💡 Key Takeaways

  • State doesn't disappear on logout — you must clean it explicitly
  • initialState pattern: one object, one reset, always in sync
  • resetAllStores: centralize, don't scatter cleanup calls
  • window.location.reload(): the nuclear option — guaranteed clean slate
  • localStorage too: persist middleware writes to localStorage — clear it on logout
