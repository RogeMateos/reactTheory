Zustand: Favorites Store — A Real-World Example

🎯 The Scenario: Job Board with Favorites

Imagine a job listing app:

  • Browse job offers
  • Mark jobs as favorites (click a ⭐ icon)
  • Unmark them later
  • See how many jobs you've favorited
  • Know if a job is in your favorites

Without a global store: Prop drilling nightmare. Every component needs favorites passed down.

With Zustand: One import, done. Simple and clean.

📦 Part 1: Understanding the Store

A Zustand store is like a global box 🎁 of data that:

  • Lives outside your components
  • Can be read from anywhere
  • Can be modified from anywhere
  • Notifies components when it changes

const useFavoritesStore = create((set, get) => ({
  // State
  favorites: [],

  // Methods (actions)
  addFavorite: (jobId) => { /* ... */ },
  removeFavorite: (jobId) => { /* ... */ },
  toggleFavorite: (jobId) => { /* ... */ },
  isFavorite: (jobId) => { /* ... */ },
  favoritesCount: () => { /* ... */ }
}))

The `create()` function returns your hook. Use it in any component:


const favorites = useFavoritesStore((state) => state.favorites)

🔧 Part 2: Key Functions Explained

1️⃣ set() → Changes the State

set() is how you modify the store.


set((state) => ({
  favorites: [...state.favorites, jobId]
}))

Think of it as: "Update the box 📦"

Important rules:

  • Always return the new state
  • Don't mutate directly (return a new array/object)
  • Zustand automatically notifies subscribers

2️⃣ get() → Reads the State

get() lets you read the current state inside an action.


const current = get().favorites
if (current.includes(jobId)) {
  // Already favorited
}

Think of it as: "Look inside the box 👀"

3️⃣ addFavorite() → Add to Favorites

Goal: Add a job ID to favorites, but avoid duplicates.


addFavorite: (jobId) => {
  const current = get().favorites

  // ✅ Check if already favorited
  if (!current.includes(jobId)) {
    set({ favorites: [...current, jobId] })
  }
}

Visual flow:


User clicks ⭐
    ↓
addFavorite(123) called
    ↓
get() reads current favorites [456, 789]
    ↓
Is 123 already there? No.
    ↓
set() updates: [456, 789, 123]
    ↓
React sees change → Re-render components using favorites

4️⃣ removeFavorite() → Remove from Favorites

Goal: Remove a job ID using filter().


removeFavorite: (jobId) => {
  set((state) => ({
    favorites: state.favorites.filter(id => id !== jobId)
  }))
}

Visual breakdown of filter():


Original:     [456, 789, 123, 999]
Remove 789:        ↓
Filtered:     [456,      123, 999]
              (kept all except 789)

filter() keeps only items where the condition is true:

  • 456 !== 789 → ✅ keep
  • 789 !== 789 → ❌ remove
  • 123 !== 789 → ✅ keep
  • 999 !== 789 → ✅ keep

5️⃣ isFavorite() → Check If Favorited

Goal: Simple yes/no answer.


isFavorite: (jobId) => {
  return get().favorites.includes(jobId)
}

Returns true or false.

Use in components:


const isFav = useFavoritesStore((s) => s.isFavorite(123))

return (
  <button style={{ color: isFav ? '🟡' : '⚪' }}>
    {isFav ? 'Remove' : 'Add'} Favorite
  </button>
)

6️⃣ toggleFavorite() → The Elegant One

Goal: Smart function that decides: add or remove?


toggleFavorite: (jobId) => {
  const isFav = get().isFavorite(jobId)

  if (isFav) {
    // Already favorited → remove it
    useFavoritesStore.getState().removeFavorite(jobId)
  } else {
    // Not favorited → add it
    useFavoritesStore.getState().addFavorite(jobId)
  }
}

Visual decision tree:


User clicks favorite button
    ↓
toggleFavorite(123) called
    ↓
Is 123 in favorites?
    ├─ YES → removeFavorite(123)
    └─ NO → addFavorite(123)

7️⃣ favoritesCount() → Calculate Without Storing

Goal: Return the count of favorites.


favoritesCount: () => {
  return get().favorites.length
}

Why NOT store a separate count variable?

  • ❌ Duplicate data = duplicate responsibility
  • ❌ Risk: count and array get out of sync
  • ✅ Calculate on demand → always accurate

Use it:


const count = useFavoritesStore((s) => s.favoritesCount())
return <p>You have {count} favorites</p>

🎒 Part 3: Using in Components

Step 1: Import the hook


import { useFavoritesStore } from './store'

Step 2: Use it in your component


function JobCard({ job }) {
  const toggleFavorite = useFavoritesStore(s => s.toggleFavorite)
  const isFav = useFavoritesStore(s => s.isFavorite(job.id))

  return (
    <div>
      <h3>{job.title}</h3>
      <button onClick={() => toggleFavorite(job.id)}>
        {isFav ? '⭐ Remove' : '☆ Add to Favorites'}
      </button>
    </div>
  )
}

No Provider needed. No wrapping the app. Just import and use.

🔄 Part 4: How Information Flows


1. User clicks button
   ↓
2. onClick triggers toggleFavorite(123)
   ↓
3. toggleFavorite checks: is 123 already favorited?
   ↓
4. Calls addFavorite(123) or removeFavorite(123)
   ↓
5. set() updates the store state
   ↓
6. React AUTOMATICALLY detects the change
   ↓
7. Components using that favorite data re-render
   ↓
8. UI updates: the button color changes, count updates

Magic part: React knows exactly which components use the favorites store, so only THOSE re-render. Not the whole app.

⚖️ Part 5: Zustand vs Context

Context (you learned before):


const FavContext = createContext()

function App() {
  const [favorites, setFavorites] = useState([])

  return (
    <FavContext.Provider value={{ favorites, setFavorites }}>
      <YourApp />
    </FavContext.Provider>
  )
}

// Then in components:
const { favorites } = useContext(FavContext)

Overhead: Need a Provider wrapper.

Zustand:


const useFavoritesStore = create(...)

// In any component, anywhere:
const favorites = useFavoritesStore(s => s.favorites)

Simplicity: Just import and use. No wrapper needed.

Visual Comparison:


Context:
  App wraps everything
    └─ Provider
         └─ Your app inside

Zustand:
  Store exists independently
    └─ Components import it directly
       (no wrapper)

🚀 Part 6: The Golden Rule

Zustand = a smart global box 🎁 in the frontend.

  • set() = Update the box
  • get() = Read from the box
  • toggle() = Smart logic that decides
  • Components import and use directly
  • No Providers, no wrappers
  • Only components using the data re-render

📋 Summary

  • ✅ Create a store with create()
  • ✅ Define state and actions inside
  • ✅ Use set() to change, get() to read
  • ✅ Avoid duplicating data (calculate counts, don't store them)
  • ✅ Import the hook in components and use
  • ✅ No Provider wrapping needed
  • ✅ Perfect for: auth, favorites, theme, settings

📁 File: store.js (The Store)


import { create } from 'zustand'

export const useFavoritesStore = create((set, get) => ({
  // STATE
  favorites: [],

  // ACTIONS
  addFavorite: (jobId) => {
    const current = get().favorites
    if (!current.includes(jobId)) {
      set({ favorites: [...current, jobId] })
    }
  },

  removeFavorite: (jobId) => {
    set((state) => ({
      favorites: state.favorites.filter(id => id !== jobId)
    }))
  },

  isFavorite: (jobId) => {
    return get().favorites.includes(jobId)
  },

  toggleFavorite: (jobId) => {
    const isFav = get().isFavorite(jobId)
    if (isFav) {
      get().removeFavorite(jobId)
    } else {
      get().addFavorite(jobId)
    }
  },

  favoritesCount: () => {
    return get().favorites.length
  }
}))

📁 File: JobCard.js (Individual Job Component)


import { useFavoritesStore } from './store'

export function JobCard({ job }) {
  const { toggleFavorite, isFavorite } = useFavoritesStore()

  const isFav = isFavorite(job.id)

  return (
    <div className="job-card">
      <h3>{job.title}</h3>
      <p>{job.company}</p>
      <p>💰 {job.salary}</p>

      <button
        className={isFav ? 'favorite-btn active' : 'favorite-btn'}
        onClick={() => toggleFavorite(job.id)}
      >
        {isFav ? '⭐ Remove Favorite' : '☆ Add Favorite'}
      </button>
    </div>
  )
}

📁 File: JobList.js (List of Jobs)


import { useFavoritesStore } from './store'
import { JobCard } from './JobCard'

export function JobList({ jobs }) {
  const count = useFavoritesStore((state) => state.favoritesCount())

  return (
    <div>
      <p>📌 Favorites: {count}</p>
      <div className="jobs-grid">
        {jobs.map(job => (
          <JobCard key={job.id} job={job} />
        ))}
      </div>
    </div>
  )
}

const MOCK_JOBS = [

🎬 Real-World Scenario: Complete Flow

Scenario: User Favorites a Job

Initial State:


Store favorites: []
Job #5: Senior React Developer
Button shows: ☆ Add to Favorites
Count shows: 0

User clicks ☆ on Job #5

Step 1: toggleFavorite(5) is called


toggleFavorite: (jobId) => {
  const isFav = get().isFavorite(5)  // Is 5 in favorites?
  // Checks: [].includes(5) → false

  if (isFav) { /* ... */ }
  else {
    get().addFavorite(5)  // Call addFavorite
  }
}

Step 2: addFavorite(5) checks for duplicates


addFavorite: (jobId) => {
  const current = get().favorites  // current = []
  if (!current.includes(5)) {      // [] doesn't include 5 ✓
    set({ favorites: [...[], 5] }) // Set to [5]
  }
}

Step 3: Store is updated


BEFORE: favorites: []
AFTER:  favorites: [5]

Step 4: React re-renders subscribed components

  • JobCard for Job #5 → isFavorite(5) now returns true → button re-renders
  • JobList → favoritesCount() now returns 1 → counter updates

Final State:


Store favorites: [5]
Job #5: Senior React Developer
Button shows: ⭐ Remove Favorite (color changes)
Count shows: 1 favorites

🔁 Scenario 2: User Favorites Another Job, Then Removes the First

Current state:


Store favorites: [5]
User clicks ☆ on Job #8 (React Consultant)

toggleFavorite(8) → addFavorite(8)


BEFORE: [5]
set({ favorites: [...[5], 8] })
AFTER:  [5, 8]

Now user clicks ⭐ on Job #5 to remove it

toggleFavorite(5) → removeFavorite(5)


BEFORE: [5, 8]
filter(id => id !== 5)
AFTER:  [8]

Process:
  5 !== 5? NO  → remove
  8 !== 5? YES → keep
Result: [8]

Final State:


Store favorites: [8]
Count shows: 1 favorites
Job #5 button: ☆ Add to Favorites (back to normal)
Job #8 button: ⭐ Remove Favorite (stays favorited)

🧠 Why This is Better Than Props Drilling

❌ Without Zustand (with prop drilling):


function App() {
  const [favorites, setFavorites] = useState([])

  return (
    <JobList favorites={favorites} setFavorites={setFavorites} />
  )
}

function JobList({ favorites, setFavorites }) {
  return (
    <>
      {jobs.map(job => (
        <JobCard
          job={job}
          favorites={favorites}           {/* Drilling */}
          setFavorites={setFavorites}     {/* Drilling */}
        />
      ))}
    </>
  )
}

function JobCard({ job, favorites, setFavorites }) {
  const toggle = () => {
    // Long logic here
  }
  return <button onClick={toggle}>...</button>
}

Every component in the middle has to know about favorites, even if it doesn't use them.

✅ With Zustand:


function App() {
  return <JobList />  {/* Clean! */}
}

function JobList() {
  const count = useFavoritesStore(s => s.favoritesCount())
  return (
    <>
      <p>{count} favorites</p>
      {jobs.map(job => <JobCard key={job.id} job={job} />)}
    </>
  )
}

function JobCard({ job }) {
  const toggle = useFavoritesStore(s => s.toggleFavorite)
  const isFav = useFavoritesStore(s => s.isFavorite(job.id))

  return (
    <button onClick={() => toggle(job.id)}>
      {isFav ? '⭐' : '☆'}
    </button>
  )
}

Clean. Clear. No drilling. Each component gets what it needs.

💡 Key Takeaways

  • One source of truth: favorites lives in one place (the store)
  • Direct access: Components grab what they need directly
  • Predictable updates: set() always updates, get() always reads
  • No duplicates: Never store count separately; calculate it
  • Smart toggle: One function that decides add or remove
  • Efficient re-renders: Only components using that data update
  • No Provider overhead: No wrapper app needed
