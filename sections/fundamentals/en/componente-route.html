Creating a Declarative <Route> Component

🎯 The goal

Transform a router with imperative logic (based on if and switch) into a declarative system using a <Route> component. This drastically improves readability, scalability, and maintainability.

😤 The problem: imperative logic in App

Currently the router works with conditionals in the main component:

// App.jsx — with if
import { useRouter } from './hooks/useRouter'
import { Home } from './pages/Home'
import { Search } from './pages/Search'
import { About } from './pages/About'

function App() {
  const { currentPath } = useRouter()

  // ❌ Imperative logic
  if (currentPath === '/') return <Home />
  if (currentPath === '/search') return <Search />
  if (currentPath === '/about') return <About />

  return <NotFound />
}

Alternative with switch (still not ideal)

switch (currentPath) {
  case '/': return <Home />
  case '/search': return <Search />
  case '/about': return <About />
  default: return <NotFound />
}

Even though switch is cleaner than multiple ifs, both have issues:

  • ❌ Imperative: you describe how routing works, not what it should do
  • ❌ Doesn't scale: adding routes means modifying component logic
  • ❌ Hard to read: not obvious at a glance which routes exist
  • ❌ Coupled logic: App knows the routing details

✨ The solution: declarative <Route> component

Instead of conditionals, we create a component that encapsulates the matching logic. Each <Route> decides by itself whether to render.

Declarative vs Imperative: declarative says "what I want"; imperative says "how to achieve it step by step". React favors declarative.

Advantages of the declarative approach

  • ✅ Expresses intent: each route is clear and self-explanatory
  • ✅ Easy to read: you see all routes at a glance
  • ✅ Scalable: adding routes is just adding more <Route>
  • ✅ Separation of concerns: each route handles its own logic
  • ✅ No conditionals in the main component

🧩 Rendering pattern with null

Returning null is a very common pattern in React to mean "don't render anything":

  • ✅ Clean: doesn't pollute the DOM with empty elements
  • ✅ Performant: React optimizes null rendering
  • ✅ Idiomatic: the standard in React

🎨 Props with renaming (component: Component)

In JSX, components must start with uppercase. That's why when receiving a prop called component, we rename it: 

export function Route({ path, component: Component }) {
  // 'component' is renamed to 'Component' (uppercase)
  return <Component />
}

Without the rename, JSX would interpret <component /> as a native HTML tag.

Step 1 — Create the Route component

// components/Route.jsx
import { useRouter } from '../hooks/useRouter'

export function Route({ path, component: Component }) {
  // Current path from the router
  const { currentPath } = useRouter()

  // If it doesn't match, render nothing
  if (currentPath !== path) {
    return null
  }

  // If it matches, render the component
  return <Component />
}

Breakdown

  • useRouter() reuses the hook from the previous class
  • if (currentPath !== path) return null is conditional rendering
  • component: Component renames the prop so JSX accepts it

Step 2 — Use Route in App (declarative version)

// App.jsx — AFTER
import { Route } from './components/Route'
import { Home } from './pages/Home'
import { Search } from './pages/Search'
import { About } from './pages/About'
import { NotFound } from './pages/NotFound'

function App() {
  return (
    <>
      <Route path="/" component={Home} />
      <Route path="/search" component={Search} />
      <Route path="/about" component={About} />
      <Route path="/404" component={NotFound} />
    </>
  )
}

export default App

🔍 How it works internally

If we are on /

<Route path="/" component={Home}/>       → "/" === "/"       ✅ renders <Home />
<Route path="/search" component={Search}/> → "/" !== "/search" ❌ null
<Route path="/about" component={About}/>   → "/" !== "/about"  ❌ null

Result: only <Home /> is rendered

If we are on /search

<Route path="/" component={Home}/>       → "/search" !== "/"       ❌ null
<Route path="/search" component={Search}/> → "/search" === "/search" ✅ renders <Search />
<Route path="/about" component={About}/>   → "/search" !== "/about"  ❌ null

Result: only <Search /> is rendered

Step 3 — Adding a shared layout

Once it's declarative, it's trivial to add common elements that appear on every route: 

function App() {
  return (
    <>
      <Header />

      <main>
        <Route path="/" component={Home} />
        <Route path="/search" component={Search} />
        <Route path="/about" component={About} />
        <Route path="/contact" component={Contact} />
      </main>

      <Footer />
    </>
  )
}

What happens:

  • <Header /> always renders
  • Only one of the routes renders (the matching one)
  • <Footer /> always renders

⚖️ Imperative vs Declarative

Imperative (if/switch) Declarative (<Route>)
Describes HOW to decide Declares WHAT should happen
App knows all the details Each route is autonomous
Hard to add/remove routes Trivial to add/remove routes
Hard to test routes in isolation Easy to test each route
