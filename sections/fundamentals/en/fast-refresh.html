🔥 Fast Refresh and State Updates

🧠 What is Fast Refresh?

Fast Refresh is a React-specific technology that preserves the state of your components when you save code changes during development.

📖 Practical example

Imagine you're developing a counter:

function Counter() {
  const [count, setCount] = React.useState(0)

  return (
    <div>
      <p>Counter: {count}</p>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  )
}

❌ Without Fast Refresh

1. You've clicked several times → Counter: 15
2. You change the text "Increment" to "Add"
3. You save the file
4. The page reloads completely
5. ❌ The counter goes back to 0

✅ With Fast Refresh

1. You've clicked several times → Counter: 15
2. You change the text "Increment" to "Add"
3. You save the file
4. The component updates
5. ✅ The counter stays at 15

💡 The state is preserved! This is super useful when developing because you don't lose your application's state every time you save a change.

🧪 Experience Fast Refresh yourself

To see Fast Refresh in action, follow these steps in a Vite project:

Step 1: Initial code (App.jsx)

Counter: 15 ← Current state

✅ With Fast Refresh (edit code)

⚠️ When Fast Refresh does NOT preserve state

Fast Refresh loses state in these cases:

  • ❌ You change the component name (from App to Counter)
  • ❌ You change the hook signature (from useState(0) to useState({count: 0}))
  • ❌ There's a syntax error in the code
  • ❌ You change the export type (from default to named)

In these cases, Vite does a full reload and the state returns to its initial value.

🔄 Fast Refresh vs Hot Module Replacement (HMR)

Although they seem the same, there's an important difference:

🔧 Hot Module Replacement (HMR)

  • 🔧 Generic technology for any type of code
  • 📦 Works with JavaScript, CSS, images, etc.
  • 🔄 Replaces modules without reloading the page
  • ⚠️ Doesn't preserve state automatically
  • 💻 Used by Vite, Webpack, Parcel, etc.
HMR: "CSS file changed → Update only the CSS"
HMR: "JS file changed → Update that module"

⚛️ Fast Refresh

  • ⚛️ React-specific technology
  • 🎯 Only works with React components
  • 💾 Preserves state of components intelligently
  • 🔄 Re-renders only the components that changed
  • ✨ Built on top of HMR (uses it underneath)
Fast Refresh: "Component changed → Update AND keep the state"

📊 Visual comparison

Feature HMR Fast Refresh
React-specific ❌ ✅
Preserves state ⚠️ Manual ✅ Automatic
Works with CSS/Assets ✅ ❌ Components only
Speed ⚡ Fast ⚡⚡ Faster

🎯 In summary

HMR = Replaces code without reloading the page
Fast Refresh = HMR + Preserve React state

💡 Fast Refresh uses HMR as a foundation, but adds React-specific intelligence to maintain your components' state.

⚠️ The Problem: Updating state with direct value

Try to guess what happens when you click this button:

function Counter() {
  const [count, setCount] = React.useState(0)

  const incrementThreeTimes = () => {
    setCount(count + 1)
    setCount(count + 1)
    setCount(count + 1)
  }

  return (
    <div>
      <p>Counter: {count}</p>
      <button onClick={incrementThreeTimes}>+3</button>
    </div>
  )
}

The result? ❌ Only adds 1, not 3

🤔 Why does this happen?

When you call setCount, React doesn't update the state immediately. It does it asynchronously. 

const incrementThreeTimes = () => {
  // At this moment, count = 0
  setCount(count + 1) // Schedules: "set to 0 + 1 = 1"
  setCount(count + 1) // Schedules: "set to 0 + 1 = 1"
  setCount(count + 1) // Schedules: "set to 0 + 1 = 1"
  // count is still 0 here

  // React applies the updates later
  // Since all say "set to 1", the result is 1
}

🔍 Step-by-step visualization

Initial state: count = 0

Button click:
  ├─ setCount(0 + 1) → "Set count to 1"
  ├─ setCount(0 + 1) → "Set count to 1"
  └─ setCount(0 + 1) → "Set count to 1"

React processes:
  "All say count should be 1"
  → count = 1 ❌

Expected: count = 3 ✅

⚠️ The problem: It's like saying three times "set the counter to 1", instead of "add 1 to the counter".

✅ The Solution: Use a callback

The correct way is to use a callback function:

function Counter() {
  const [count, setCount] = React.useState(0)

  const incrementThreeTimes = () => {
    setCount((prevCount) => prevCount + 1)
    setCount((prevCount) => prevCount + 1)
    setCount((prevCount) => prevCount + 1)
  }

  return (
    <div>
      <p>Counter: {count}</p>
      <button onClick={incrementThreeTimes}>+3</button>
    </div>
  )
}

Now it works correctly: ✅ Adds 3

🎯 How does the callback work?

setCount((prevCount) => prevCount + 1)

You tell React: "Give me the most recent value and I'll return the new value"

🔍 Step-by-step visualization

Initial state: count = 0

Button click:
  ├─ setCount(prev => prev + 1)
  │    └─ prevCount = 0 → returns 1
  │
  ├─ setCount(prev => prev + 1)
  │    └─ prevCount = 1 → returns 2
  │
  └─ setCount(prev => prev + 1)
       └─ prevCount = 2 → returns 3

React processes:
  0 → 1 → 2 → 3
  → count = 3 ✅

✅ React processes updates in order, always using the most recent value.

📊 Side-by-side comparison

❌ Direct value (Incorrect) ✅ Callback (Correct) 
setCount(count + 1)
setCount(count + 1)
setCount(count + 1)

// All use the same count (0)
// Result: 1 ❌
 
setCount(prev => prev + 1)
setCount(prev => prev + 1)
setCount(prev => prev + 1)

// Each uses the updated value
// Result: 3 ✅

🧪 Other common examples

❌ Incorrect - Toggle boolean:

// If you click multiple times quickly, it can fail
setVisible(!visible)

✅ Correct - Toggle boolean:

// Always works correctly
setVisible(prev => !prev)

❌ Incorrect - Add to array:

// Can lose items if updated quickly
setItems([...items, newItem])

✅ Correct - Add to array:

// Guarantees you use the most recent array
setItems(prevItems => [...prevItems, newItem])

🎯 When to use each form?

📝 Direct value (simple form)

Use this form when the new value is INDEPENDENT of the current value:

// ✅ Good: the new value does NOT depend on the previous one
setName('Michael')
setShow(true)
setColor('#ff0000')
setUser({ id: 1, name: 'Anna' })

🔄 Callback (function form)

Use this form when the new value DEPENDS on the previous value:

// ✅ Good: the new value depends on the previous one
setCount(prev => prev + 1)
setItems(prevItems => [...prevItems, newItem])
setVisible(prev => !prev)
setUser(prev => ({ ...prev, age: prev.age + 1 }))

🚦 When to use callback (checklist)

Use setState(prev => ...) when:

  • ⚡ The new value depends on the previous value
  • 🔄 You're making multiple consecutive updates
  • 🎯 You want to guarantee using the most recent value
  • 🐛 You're experiencing stale state bugs

📚 Practice exercises

Exercise 1: Which is correct for this case?

// You need to increment a counter when the user clicks
const handleClick = () => {
  // Option A
  setCount(count + 1)

  // Option B
  setCount(prev => prev + 1)
}
See answer

Answer: Option B is safer. If the user clicks very quickly or there are multiple updates, Option A might lose some clicks. Option B guarantees each click is counted correctly.

Exercise 2: Which is correct for this case?

// You need to update the user's name when they type in an input
const handleChange = (e) => {
  // Option A
  setName(e.target.value)

  // Option B
  setName(prev => e.target.value)
}
See answer

Answer: Option A is fine. The new value (e.target.value) is completely independent of the previous value. You don't need the callback here.

Exercise 3: What will this code result in?

const [count, setCount] = useState(0)

const handleClick = () => {
  setCount(count + 1)
  setCount(count + 1)
  console.log(count) // What gets printed?
}
See answer

Prints: 0
1. The console.log runs immediately, before React updates the state
2. Both setCount calls schedule updates but don't execute them immediately
3. After the function finishes, React updates the state to 1 (not 2, due to the explained problem)

🎯 Golden rule

If you see the state name inside setState, use callback:

❌ setCount(count + 1)        → count is here!
✅ setCount(prev => prev + 1)  → Better with callback

✅ setName('Anna')             → Doesn't use previous value, it's fine
