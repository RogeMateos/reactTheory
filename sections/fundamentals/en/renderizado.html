Rendering and Virtual DOM in React

1️⃣ What Does "Rendering" Mean?

Rendering means that React takes your JSX and converts it into real HTML that appears in the browser.

It's like translating from one language to another:

JSX (React code)
  ↓
React processes
  ↓
HTML (what you see on screen)

Example:

// Your JSX code
function App() {
  return <h1>Hello World</h1>
}

// React converts it to HTML
<h1>Hello World</h1>

2️⃣ When Does React Render?

React renders (or re-renders) your component in these situations:

┌─────────────────────────────────────┐
│  RENDERING SITUATIONS               │
├─────────────────────────────────────┤
│  1. Initial mount (first time)      │
│  2. State changes                   │
│  3. Props change                    │
│  4. Parent component re-renders     │
└─────────────────────────────────────┘

🎯 Situation 1: Initial Mount

When React renders your component for the first time.

const root = ReactDOM.createRoot(document.getElementById('root'))
root.render(<App />)

// React says: "Ok, first time, I'll render App"

🔄 Situation 2: State Changes

This is the most important one.

function Counter() {
  const [count, setCount] = React.useState(0)

  // When you click:
  // 1. setCount(1) updates the state
  // 2. React detects the change
  // 3. React re-renders Counter

  return (
    <button onClick={() => setCount(count + 1)}>
      Clicks: {count}
    </button>
  )
}

Visual flow of re-rendering:

Initial state: count = 0
       ↓
User clicks
       ↓
setCount(1) executes
       ↓
🔄 React re-renders Counter
       ↓
Button now shows: "Clicks: 1"

3️⃣ Why Doesn't State Get Lost?

Many developers wonder:

"If React re-executes the component function, why doesn't the state reset?"

Answer: Because React stores state OUTSIDE the component.

🧠 Mental Model

┌─────────────────────────────────────────┐
│  COMPONENT (your code)                  │
│  ┌─────────────────────────────────┐    │
│  │ function Counter() {            │    │
│  │   const [count, setCount] = ... │    │
│  │   return <button>...</button>   │    │
│  │ }                               │    │
│  └─────────────────────────────────┘    │
│           ↑                              │
│           │ React connects               │
│           ↓                              │
│  ┌─────────────────────────────────┐    │
│  │  REACT'S MEMORY                 │    │
│  │  count = 5   ← stored here      │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘

📊 Step by Step

1. First execution
   ├─ useState(0) creates state
   ├─ React stores: count = 0
   └─ Returns: [0, setCount]

2. User clicks
   ├─ setCount(1) executes
   ├─ React updates in memory: count = 1
   └─ React re-executes the component

3. Second execution (re-render)
   ├─ useState() does NOT create new state
   ├─ React finds existing state: count = 1
   └─ Returns: [1, setCount]

💡 Key point: React maintains state between renders. It doesn't get lost because it's not inside your function, it's in React's internal memory.

5️⃣ Virtual DOM: The Magic Behind React

What is the Virtual DOM?

It's a copy of the real DOM that React keeps in memory. It's a lightweight representation (in JavaScript) of your interface.

🌳 Analogy: The Blueprint of a House

BLUEPRINT (Virtual DOM)    REAL HOUSE (Real DOM)
     📋                           🏠

Cheap to modify          Expensive to modify
Fast to change           Slow to change
Just paper               Wood, cement, etc.

React does the same:

  • First modifies the "blueprint" (Virtual DOM) → fast
  • Compares with previous blueprint
  • Only updates what's necessary in the "real house" (DOM) → efficient

📊 Visual Comparison

WITHOUT VIRTUAL DOM (Vanilla JS):
┌──────────────────────────────────┐
│  Small UI change                 │
│         ↓                        │
│  Re-render ENTIRE DOM            │
│         ↓                        │
│  💰 Very expensive               │
│  🐌 Very slow                    │
└──────────────────────────────────┘


WITH VIRTUAL DOM (React):
┌──────────────────────────────────┐
│  Small UI change                 │
│         ↓                        │
│  Update Virtual DOM              │
│         ↓                        │
│  Compare (diffing)               │
│         ↓                        │
│  Update ONLY what changed        │
│         ↓                        │
│  ⚡ Very efficient               │
│  🚀 Very fast                    │
└──────────────────────────────────┘

6️⃣ The Diffing Algorithm

Diffing = Finding the differences

React compares the previous Virtual DOM with the new one to know what changed.

🔍 Visual Example

BEFORE (Previous Virtual DOM):
┌─────────────────────────┐
│ <div>                   │
│   <h1>Count: 0</h1>     │
│   <button>+</button>    │
│ </div>                  │
└─────────────────────────┘


AFTER (New Virtual DOM):
┌─────────────────────────┐
│ <div>                   │
│   <h1>Count: 1</h1>  ⭐ │ ← Only this changed
│   <button>+</button>    │
│ </div>                  │
└─────────────────────────┘


REACT DETECTS:
  ✅ <div> → no changes
  ⭐ <h1> → text changed (0 → 1)
  ✅ <button> → no changes

RESULT:
React only updates the h1 text
Doesn't touch the rest of the DOM!

Code with Console.log

function Counter() {
  const [count, setCount] = React.useState(0)

  console.log('🔄 Component executing')

  return (
    <div>
      <h1>Count: {count}</h1>
      <button onClick={() => setCount(count + 1)}>+</button>
    </div>
  )
}

// Console output each time you click:
// 🔄 Component executing
//
// But in the real DOM, React ONLY updates the h1 text
// Doesn't rebuild the entire component

7️⃣ Reconciliation: The Complete Process

Reconciliation is the process React uses to decide what to update in the real DOM.

┌───────────────────────────────────────────────────────┐
│               RECONCILIATION PROCESS                  │
├───────────────────────────────────────────────────────┤
│                                                       │
│  STEP 1: Create New Virtual DOM                      │
│  ┌─────────────────────────────────────┐             │
│  │  React executes your function       │             │
│  │  Generates new JSX tree             │             │
│  └─────────────────────────────────────┘             │
│                    ↓                                  │
│                                                       │
│  STEP 2: Compare (Diffing Algorithm)                 │
│  ┌─────────────────────────────────────┐             │
│  │  Previous Virtual DOM  vs  New      │             │
│  │                                     │             │
│  │  Same element type?                 │             │
│  │  Same attributes?                   │             │
│  │  Same content?                      │             │
│  └─────────────────────────────────────┘             │
│                    ↓                                  │
│                                                       │
│  STEP 3: Generate List of Changes                    │
│  ┌─────────────────────────────────────┐             │
│  │  Update h1 text                     │             │
│  │  Change button class                │             │
│  │  Add new li                         │             │
│  └─────────────────────────────────────┘             │
│                    ↓                                  │
│                                                       │
│  STEP 4: Apply Changes to Real DOM                   │
│  ┌─────────────────────────────────────┐             │
│  │  Execute minimal changes            │             │
│  │  UI updates                         │             │
│  └─────────────────────────────────────┘             │
│                                                       │
└───────────────────────────────────────────────────────┘

Comparative Example

// State 1: applied = false
<button className="button-apply-job">
  Apply
</button>

// User clicks → applied = true
// React re-renders

// State 2: applied = true
<button className="button-apply-job is-applied" disabled>
  Applied!
</button>

// React detects:
// ✅ Element: <button> (same type)
// ⭐ className: changed
// ⭐ disabled: added
// ⭐ text: changed

// React applies ONLY those 3 changes to the real DOM

8️⃣ Batch Updates: React is Smart

React does NOT re-render immediately after each setState. It batches multiple changes together.

Example with Multiple States

function Form() {
  const [name, setName] = React.useState('')
  const [email, setEmail] = React.useState('')
  const [age, setAge] = React.useState(0)

  const handleSubmit = () => {
    setName('Miguel')   // 1
    setEmail('m@g.com') // 2
    setAge(25)          // 3

    // React does NOT re-render 3 times
    // React batches the 3 changes
    // Re-renders ONLY ONCE
  }

  console.log('🎨 Rendering...')

  return (
    <button onClick={handleSubmit}>Submit</button>
  )
}

🔄 Batch Updates Flow

User clicks
       ↓
setName('Miguel')   ─┐
setEmail('m@g.com') ─┼─ React batches
setAge(25)          ─┘
       ↓
React re-renders ONCE
       ↓
console.log executes ONCE

💡 Automatic optimization: React automatically batches multiple setState calls into a single re-render for better performance.

9️⃣ When React Does NOT Re-render

React is smart and does NOT re-render in these situations:

┌────────────────────────────────────────┐
│  CASES WITHOUT RE-RENDERING            │
├────────────────────────────────────────┤
│  1. setState with SAME value           │
│  2. Local variables change             │
│  3. Props don't change                 │
└────────────────────────────────────────┘

Case 1: setState with Same Value

function Counter() {
  const [count, setCount] = React.useState(0)

  const handleClick = () => {
    setCount(0) // ← Already 0
    // React detects it didn't change
    // Does NOT re-render
  }

  console.log('🎨 Rendering...')

  return <button onClick={handleClick}>{count}</button>
}

// Console output:
// 🎨 Rendering... (initial mount)
// (user clicks)
// (nothing in console - no re-render)

Case 2: Local Variables

function Counter() {
  const [count, setCount] = React.useState(0)
  let clicks = 0 // ← Local variable

  const handleClick = () => {
    clicks = clicks + 1 // ❌ This does NOT cause re-render
    console.log('clicks:', clicks) // You'll see the change
    // But UI does NOT update
  }

  return <button onClick={handleClick}>{clicks}</button>
}

// UI always shows 0
// Because clicks is NOT state
// React doesn't know it changed

⚠️ Important rule: Only STATE and PROPS changes cause re-rendering. Local variables don't.

🔟 Re-rendering by Props

When a parent component re-renders, its children also re-render (by default).

Example

function Parent() {
  const [count, setCount] = React.useState(0)

  console.log('🎨 Parent rendering')

  return (
    <div>
      <button onClick={() => setCount(count + 1)}>
        Count: {count}
      </button>
      <Child />
    </div>
  )
}

function Child() {
  console.log('🎨 Child rendering')
  return <p>I'm the child</p>
}

// Console output:
// 🎨 Parent rendering
// 🎨 Child rendering
// (user clicks)
// 🎨 Parent rendering  ← Parent re-renders
// 🎨 Child rendering   ← Child also re-renders

🔍 Visual Flow

Parent re-renders
       ↓
React executes Parent() function
       ↓
Finds <Child /> in the JSX
       ↓
React executes Child() function
       ↓
Child re-renders too

Even though Child's props don't change, React re-renders by default.

📋 What You Must Remember

  • Rendering = React converts JSX to real HTML
  • Re-rendering = React updates the UI when state/props change
  • Virtual DOM = Lightweight copy of DOM in memory
  • Diffing = Compare previous vs new Virtual DOM
  • Reconciliation = Decide what to update in real DOM
  • Batch Updates = React groups multiple setState into one re-render
  • React does NOT re-render if state didn't actually change
  • Local variables do NOT cause re-rendering

1️⃣3️⃣ Typical Interview Question

❓ "Why does React use Virtual DOM?"

Correct answer:

Because updating the real DOM is expensive and slow. React uses Virtual DOM to:

  • Calculate changes quickly in memory
  • Compare previous and new versions (diffing)
  • Apply only the minimal necessary changes to the real DOM
  • Optimize performance automatically

Analogy for the interview:

"It's like making a draft on paper before building. Testing changes on the draft is fast and cheap. Only when you know exactly what to change, do you modify the real construction."

function MultipleStates() {
  const [name, setName] = React.useState('')
  const [email, setEmail] = React.useState('')
  const [age, setAge] = React.useState(0)

  // Render counter
  const renderCount = React.useRef(0)
  renderCount.current = renderCount.current + 1

  const handleUpdate = () => {
    // Three state changes
    setName('Miguel')
    setEmail('miguel@example.com')
    setAge(25)

    // React batches these 3 changes
    // Only re-renders ONCE
  }

  return (

      <p>Renders: {renderCount.current}</p>
      <p>Name: {name}</p>
      <p>Email: {email}</p>
      <p>Age: {age}</p>
      <button onClick={handleUpdate}>
        Update All
      </button>
    </div>

}

// Result: Only increments by 1 per click

Exercise 4: Batch Updates

Challenge: Demonstrate that React batches multiple setState.

Solution:


function BatchDemo() {
  const [a, setA] = React.useState(0)
  const [b, setB] = React.useState(0)
  const [c, setC] = React.useState(0)

  const renderCount = React.useRef(0)
  renderCount.current = renderCount.current + 1

  const handleBatch = () => {
    setA(a + 1)
    setB(b + 1)
    setC(c + 1)

    // React batches these 3 changes
    // Render count increases by only 1
  }

  const handleSeparate = () => {
    setA(a + 1)

    setTimeout(() => {
      setB(b + 1)
    }, 100)

    setTimeout(() => {
      setC(c + 1)
    }, 200)

    // These do NOT get batched (before React 18)
    // Render count increases by 3
  }

  return (
    <div>
      <p>Renders: {renderCount.current}</p>
      <p>A: {a}, B: {b}, C: {c}</p>

      <button onClick={handleBatch}>
        Batch Update (1 render)
      </button>

      <button onClick={handleSeparate}>
        Separate Updates (3 renders)
      </button>
    </div>
  )
}

Exercise 5: Prevent Unnecessary Re-renders

Challenge: Use React.memo to prevent re-renders of a child when not necessary.

Solution:


function Parent() {
  const [parentCount, setParentCount] = React.useState(0)
  const [childCount, setChildCount] = React.useState(0)

  console.log('🎨 Parent rendering')

  return (
    <div>
      <h1>Parent: {parentCount}</h1>
      <button onClick={() => setParentCount(parentCount + 1)}>
        Increment Parent
      </button>

      <button onClick={() => setChildCount(childCount + 1)}>
        Increment Child
      </button>

      {/* Child only re-renders if childCount changes */}
      <MemoizedChild count={childCount} />
    </div>
  )
}

// Without React.memo, Child re-renders whenever Parent does
const MemoizedChild = React.memo(function Child({ count }) {
  console.log('🎨 Child rendering')

  return <p>Child Count: {count}</p>
})

// Now Child ONLY re-renders when count changes
// Not when parentCount changes

Exercise 6: Visualize the Reconciliation Process

Challenge: Create a component that shows which parts of the DOM change.

Solution:


function ReconciliationDemo() {
  const [count, setCount] = React.useState(0)
  const [name, setName] = React.useState('Miguel')

  // Only count and name will change in the DOM
  // The h1, h2, button elements remain the same

  return (
    <div>
      <h1>Static Title</h1>
      <h2>Count: {count}</h2>  {/* ⭐ Only this changes */}
      <h2>Name: {name}</h2>    {/* ⭐ Only this changes */}

      <button onClick={() => setCount(count + 1)}>
        Increment Count
      </button>

      <button onClick={() => setName(name === 'Miguel' ? 'Ana' : 'Miguel')}>
        Change Name
      </button>

      <p>React only updates the content of the h2 elements</p>
      <p>Doesn't rebuild the entire component</p>
    </div>
  )
}
