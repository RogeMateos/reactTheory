Keys and List Rendering

😱 The Problem: Manually Repeated Components

When data is dynamic, repeating JSX manually doesn't scale:

// ❌ Repetitive, doesn't scale, data mixed with UI
function App() {
  return (
    <div>
      <article><h3>Frontend Developer</h3><p>TechCorp</p></article>
      <article><h3>Backend Developer</h3><p>DataStack</p></article>
      <article><h3>Full Stack Developer</h3><p>StartupX</p></article>
    </div>
  )
}
📝 Repetitive        — copy-pasting the same HTML
🔢 Doesn't scale     — what if you have 100 jobs?
📊 Hardcoded data    — UI and data are mixed together
🔄 Hard to update    — changing something touches multiple places

The solution: separate data into a JSON file and render with .map().

🔄 What does .map() do?

.map() iterates over an array and returns a new transformed array. In React, we transform data into JSX elements: 

const jobs = [
  { id: 1, title: 'Frontend Developer' },
  { id: 2, title: 'Backend Developer' },
  { id: 3, title: 'Full Stack Developer' },
]

// Data → Components
jobs.map((job) => <JobCard key={job.id} job={job} />)

// Result: array of React components
// [
//   <JobCard key={1} job={{...}} />,
//   <JobCard key={2} job={{...}} />,
//   <JobCard key={3} job={{...}} />
// ]

🔑 Why is the key Required?

The key is a unique identifier that lets React know which elements changed, were added, or were removed when the list updates.

Without keys — React doesn't know what changed

// You add an element at the start
Before:  [Frontend, Backend, FullStack]
After:   [DevOps, Frontend, Backend, FullStack]

// Without keys, React compares position by position
// position 0: Frontend → DevOps   ← "changed"
// position 1: Backend  → Frontend ← "changed"
// position 2: FullStack → Backend  ← "changed"
// position 3: (none)   → FullStack ← "new"
// React re-renders EVERYTHING 🐌

With keys — React knows exactly what happened

// With key={id}, React compares by identity
Before:  [key=1 Frontend, key=2 Backend, key=3 FullStack]
After:   [key=4 DevOps, key=1 Frontend, key=2 Backend, key=3 FullStack]

// key=4 is new → only renders that one ⚡
// key=1, 2, 3 didn't change → React reuses them

📋 Rules for Keys

1. Must be unique among siblings

// ✅ Unique IDs
{jobs.map((job) => <JobCard key={job.id} job={job} />)}

// ❌ All the same — React can't tell them apart
{jobs.map((job) => <JobCard key="job" job={job} />)}

2. Must be stable (not change between renders)

// ❌ Math.random() generates a different key every render
{jobs.map((job) => <JobCard key={Math.random()} job={job} />)}

// ✅ The ID doesn't change between renders
{jobs.map((job) => <JobCard key={job.id} job={job} />)}

3. Indexes: only for static lists

// ✅ OK — static list that never changes
const DAYS = ['Monday', 'Tuesday', 'Wednesday']
{DAYS.map((day, index) => <li key={index}>{day}</li>)}

// ❌ Dangerous — dynamic list with reorder/delete operations
{tasks.map((task, index) => <Task key={index} task={task} />)}

// ✅ Correct — dynamic list with unique ID
{tasks.map((task) => <Task key={task.id} task={task} />)}

🧩 JobCard Component

// src/components/JobCard.jsx
export function JobCard({ job }) {
  return (
    <article
      className="job-listing-card"
      data-modalidad={job.data.modalidad}
      data-nivel={job.data.nivel}
    >
      <div>
        <h3>{job.titulo}</h3>
        <small>{job.empresa} | {job.ubicacion}</small>
        <p>{job.descripcion}</p>
      </div>
      <button>Apply</button>
    </article>
  )
}

📋 JobListings Component with .map()

// src/components/JobListings.jsx
import { JobCard } from './JobCard'
import jobsData from '../data/jobs.json'

export function JobListings() {
  return (
    <div className="jobs-grid">
      {jobsData.map((job) => (
        <JobCard key={job.id} job={job} />
      ))}
    </div>
  )
}

jobs.json
  │
  ▼
jobsData (array of objects)
  │
  .map((job) => <JobCard key={job.id} job={job} />)
  │
  ▼
Array of rendered React components

🎭 Empty List: Conditional Rendering

// With early return
export function JobListings({ jobs }) {
  if (jobs.length === 0) {
    return (
      <div className="no-jobs">
        <p>No jobs available at the moment.</p>
      </div>
    )
  }

  return (
    <div className="jobs-grid">
      {jobs.map((job) => (
        <JobCard key={job.id} job={job} />
      ))}
    </div>
  )
}
// With inline ternary
export function JobListings({ jobs }) {
  return jobs.length === 0 ? (
    <p>No jobs available.</p>
  ) : (
    <div className="jobs-grid">
      {jobs.map((job) => (
        <JobCard key={job.id} job={job} />
      ))}
    </div>
  )
}

⚠️ The Index Bug: Visual Example

Why using index as a key can cause bugs in dynamic lists:

// Initial state
<Task key={0} text="Task A" completed={false} />
<Task key={1} text="Task B" completed={true}  />
<Task key={2} text="Task C" completed={false} />

// User deletes "Task A"

// With index as key:
<Task key={0} text="Task B" completed={true}  />  // ← key reassigned
<Task key={1} text="Task C" completed={false} />  // ← key reassigned
// React: "key=0 is now Task B" → may mix up state ❌

// With ID as key:
<Task key="b" text="Task B" completed={true}  />  // ← stable key
<Task key="c" text="Task C" completed={false} />  // ← stable key
// React: "key='a' was removed, the rest are unchanged" ✅

🧩 JobCard            — component that displays one job
📋 JobListings        — component that renders the list
🔄 .map()             — array of data → array of JSX components
🔑 Required key       — React needs it to identify elements
🎯 Unique/stable keys — use IDs, not Math.random() or index
⚠️ Dangerous indexes  — only safe in completely static lists
🎭 Empty list         — always handle the empty array case
