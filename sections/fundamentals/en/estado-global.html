Global State: Prop Drilling vs Context API

One of React's biggest challenges isn't components. It's managing state when many components need the same data.

Here's the problem:

Imagine you have a house (your app). To get a letter from the living room to the bedroom, you have two choices:

🏠 The House Analogy

Think of your React app as a house with rooms: 

App (Living room)
 ├── Layout
 │    ├── Sidebar
 │    │    └── Menu
 │    │         └── UserAvatar  ← wants user data
 │    └── Header
 │         └── UserProfile  ← wants user data
 └── MainContent
      └── UserSettings  ← wants user data

The user data lives in the living room (App component), but three different rooms need it.

📦 Option 1: Prop Drilling (Passing the letter person-to-person)

You pass the data as props through every component, even if they don't use it. 

// App.js - the living room
function App() {
  const [user, setUser] = useState({ name: 'John', role: 'admin' })

  return <Layout user={user} />
}

// Layout.js - first hallway
function Layout({ user }) {
  return (
    <div>
      <Sidebar user={user} />
      <Header user={user} />
    </div>
  )
}

// Sidebar.js - second hallway
function Sidebar({ user }) {
  return (
    <div>
      <Menu user={user} />
    </div>
  )
}

// Menu.js - third hallway
function Menu({ user }) {
  return <UserAvatar user={user} />
}

// UserAvatar.js - finally got to the bedroom
function UserAvatar({ user }) {
  return <img src={user.avatar} alt={user.name} />
}

Is Prop Drilling Bad? NO!

This is completely normal and fine when you have 1-3 levels of nesting.

Many React apps use prop drilling for years. It's not a bad practice — it's just React.

The Real Problem 🚨

Prop drilling becomes a headache when:

  • Deep nesting (7+ levels) — Components pass data they don't use
  • Many branches — Data goes through many components
  • Hard to refactor — Change the structure = change all intermediate components
  • Mental overhead — You need to trace the path of every prop

📫 Package Analogy: Prop drilling is like a letter passed from person to person: Person A has it → Person B passes it → Person C passes it → Person D finally opens it. If 7 people stand between A and D, it gets tedious!

📢 Option 2: Context API (The bulletin board)

Instead of passing props, you create a "bulletin board" that anyone can read. 

// UserContext.js
import { createContext, useState } from 'react'

const UserContext = createContext()

export function UserProvider({ children }) {
  const [user, setUser] = useState({ name: 'John', role: 'admin' })

  return (
    <UserContext.Provider value={{ user, setUser }}>
      {children}
    </UserContext.Provider>
  )
}

export function useUser() {
  return useContext(UserContext)
}

// App.js - wrap the whole app
function App() {
  return (
    <UserProvider>
      <Layout />
    </UserProvider>
  )
}

// Layout.js - no props needed!
function Layout() {
  return (
    <div>
      <Sidebar />
      <Header />
    </div>
  )
}

// Sidebar.js - still no props!
function Sidebar() {
  return <Menu />
}

// Menu.js - still clean!
function Menu() {
  return <UserAvatar />
}

// UserAvatar.js - reads from the bulletin board
function UserAvatar() {
  const { user } = useUser()
  return <img src={user.avatar} alt={user.name} />
}

Notice: Layout, Sidebar, and Menu don't need to know about user data anymore.

⚠️ The Hidden Problem with Context API

Context is powerful, but it has a gotcha:

When Context state changes, ALL consumers re-render — even if they only use one field.

// UserContext.js
const [user, setUser] = useState({
  name: 'John',
  email: 'john@example.com',
  avatar: 'john.jpg',
  preferences: { theme: 'dark', notifications: true },
  // ...20 more fields
})

// Later, you update just the theme
setUser({ ...user, preferences: { ...user.preferences, theme: 'light' } })

// PROBLEM: UserAvatar re-renders even though it only uses avatar!
// It doesn't care about theme, but it re-renders anyway.

When is Context Good? ✅

Context is perfect for state that changes rarely and is read frequently:

  • 🌙 Theme (light/dark) — changes once per session
  • 🇪🇸 Language (EN/ES) — changes once per session
  • 👤 Current user (logged in) — changes on login/logout
  • 🎫 Authentication token — changes on login/logout

When is Context Bad? ❌

Context causes performance problems when state changes often and consumers care about different fields:

  • 🖱️ Mouse position — changes on every movement (60+ FPS)
  • ✍️ Form inputs — change on every keystroke
  • 🎬 Animations — change multiple times per second
  • 📜 Scroll position — changes constantly
  • ⏱️ Timer/counter — updates every millisecond

🏙️ City Analogy:
• Prop drilling = Letter passed person-to-person
• Context API = Bulletin board in town square (everyone can read)
• Zustand/Redux = Central warehouse (optimized for quick lookups, no re-renders on unrelated updates)

┌────────────────────┬─────────────────────────┬──────────────────────────┐
│ Feature            │ Prop Drilling           │ Context API              │
├────────────────────┼─────────────────────────┼──────────────────────────┤

│ Good for...        │ 1-3 levels, clear flow  │ Infrequent, global state │
│ Bad for...         │ Deep trees, many levels │ Frequently changing data │
│ Developer DX       │ Clear prop chain        │ "Magic" data from air    │
│ Testing            │ Easy to mock props      │ Need to wrap in Provider │
└────────────────────┴─────────────────────────┴──────────────────────────┘

🏦 When Zustand Enters the Picture

So far we have two options. But what if you need:

  • Global state that changes frequently
  • Many components reading different fields
  • No re-renders when unrelated state changes
  • Complex state with many actions

This is where Zustand (or Redux) comes in.

// store.js (Zustand)
import create from 'zustand'

const useAppStore = create((set) => ({
  user: { name: 'John' },
  theme: 'dark',
  notifications: [],
  setUser: (user) => set({ user }),
  setTheme: (theme) => set({ theme }),
  addNotification: (notif) => set((state) => ({
    notifications: [...state.notifications, notif]
  }))
}))

// Component A reads ONLY user
function UserAvatar() {
  const user = useAppStore((state) => state.user)
  return <img src={user.avatar} />
}

// Component B reads ONLY theme
function Header() {
  const theme = useAppStore((state) => state.theme)
  return <div style={{ background: theme === 'dark' ? '#000' : '#fff' }}>
}

// When user changes: only Component A re-renders
// When theme changes: only Component B re-renders
// No unnecessary re-renders! ✓

🎓 The Senior Developer's Rule

This is what separates juniors from seniors:

Is the nesting 1-3 levels deep?
  ↓ YES → Use Props (keep it simple!)
  ↓ NO

Does the state change rarely? (theme, language, user auth)
  ↓ YES → Use Context API
  ↓ NO

Does the state change frequently? (animations, form inputs, etc)
  ↓ YES → Use Zustand / Redux

Decision tree complete ✓

💡 What Junior Devs Do: "I'll use Context for everything" → Performance problems
💡 What Senior Devs Do: "Let me check the change frequency and nesting depth" → Right tool, every time

🎤 The Interview Answer

If someone asks you about prop drilling in an interview, say this:

"Prop drilling is normal and fine for 1-3 levels of nesting. It's explicit and testable. When you have deeper trees or many components passing data they don't use, Context API is better — it lets us skip intermediate components. But Context has gotchas: if state changes frequently, consumers re-render unnecessarily. For that case, we use Zustand or Redux, which let us select specific pieces of state."

If they follow up with "When would you use each?", show your decision tree above. That's the sign of a developer who thinks, not just codes.

// App.js
import { useState } from 'react'

function App() {
  const [username, setUsername] = useState('Alice')

  return <Layout username={username} setUsername={setUsername} />

function MainContent({ username, setUsername }) {

        value={username}

    </form>
  )
}

Example 3: The Performance Problem (Bad Context Usage)


// AppContext.js
import { createContext, useContext, useState } from 'react'

const AppContext = createContext()

export function AppProvider({ children }) {
  // Lots of state packed together
  const [state, setState] = useState({
    user: { name: 'John', email: 'john@example.com' },
    theme: 'light',
    notifications: [],
    sidebarOpen: true,
    // ... many more fields
  })

  // This is the problem:
  // When ANY field changes, all consumers re-render
  const toggleTheme = () => {
    setState({ ...state, theme: state.theme === 'light' ? 'dark' : 'light' })
  }

  return (
    <AppContext.Provider value={{ state, toggleTheme }}>
      {children}
    </AppContext.Provider>
  )
}

export function useApp() {
  return useContext(AppContext)
}

// Component 1: Only cares about theme
function DarkModeToggle() {
  const { state, toggleTheme } = useApp()

  return (
    <button onClick={toggleTheme}>
      Current theme: {state.theme}
    </button>
  )
}

// Component 2: Only cares about user
function UserProfile() {
  const { state } = useApp()

  return <div>Hello {state.user.name}</div>
}

// PROBLEM: When toggleTheme() is called:
// 1. state changes
// 2. AppContext value becomes a new object
// 3. BOTH DarkModeToggle AND UserProfile re-render
// 4. UserProfile didn't care about theme, but it re-renders anyway!

Example 4: The Solution (Zustand for high-frequency state)


// store.js (Zustand)
import create from 'zustand'

const useAppStore = create((set) => ({
  // Separate concerns
  user: { name: 'John', email: 'john@example.com' },
  theme: 'light',
  notifications: [],
  sidebarOpen: true,

  // Actions
  setTheme: (theme) => set({ theme }),
  setUser: (user) => set({ user }),
  addNotification: (notif) => set((state) => ({
    notifications: [...state.notifications, notif]
  }))
}))

// Component 1: Only subscribes to theme
function DarkModeToggle() {
  const theme = useAppStore((state) => state.theme)
  const setTheme = useAppStore((state) => state.setTheme)

  return (
    <button onClick={() => setTheme(theme === 'light' ? 'dark' : 'light')}>
      Current theme: {theme}
    </button>
  )
}

// Component 2: Only subscribes to user
function UserProfile() {
  const user = useAppStore((state) => state.user)

  return <div>Hello {user.name}</div>
}

// SOLUTION: Zustand uses selector pattern
// When theme changes:
// 1. Only DarkModeToggle re-renders (it selected theme)
// 2. UserProfile does NOT re-render (it didn't select theme)
// 3. Performance is optimized automatically ✓

// Usage in app
function App() {
  return (
    <div>
      <DarkModeToggle />
      <UserProfile />
    </div>
  )
}

export default App

Example 5: Real-world app structure


// contexts/ThemeContext.js
// Purpose: Theme is infrequent (changes once per session)
// Decision: Context API is fine
import { createContext, useContext, useState } from 'react'

const ThemeContext = createContext()

export function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light')

  const toggleTheme = () => {
    setTheme(theme === 'light' ? 'dark' : 'light')
  }

  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      {children}
    </ThemeContext.Provider>
  )
}

export function useTheme() {
  return useContext(ThemeContext)
}

// store/userStore.js
// Purpose: User state changes frequently (multiple fields, animations)
// Decision: Zustand for optimal performance
import create from 'zustand'

export const useUserStore = create((set) => ({
  user: null,
  isLoading: false,
  error: null,

  fetchUser: async (id) => {
    set({ isLoading: true })
    try {
      const response = await fetch(`/api/users/${id}`)
      const user = await response.json()
      set({ user, error: null })
    } catch (error) {
      set({ error: error.message })
    } finally {
      set({ isLoading: false })
    }
  },

  updateUser: (updates) => set((state) => ({
    user: { ...state.user, ...updates }
  }))
}))

// App.js
import { ThemeProvider } from './contexts/ThemeContext'

function App() {
  return (
    <ThemeProvider>
      <Layout />
    </ThemeProvider>
  )
}

// pages/UserPage.js
import { useUserStore } from '../store/userStore'
import { useTheme } from '../contexts/ThemeContext'

function UserPage() {
  const { user, isLoading } = useUserStore()
  const { theme } = useTheme()

  const containerStyle = {
    background: theme === 'dark' ? '#1a1a1a' : '#fff'
  }

  if (isLoading) return <p>Loading...</p>
  if (!user) return <p>No user found</p>

  return (
    <div style={containerStyle}>
      <h1>{user.name}</h1>
      <p>{user.email}</p>
    </div>
  )
}

export default UserPage

How to Choose: Decision Tree

Step 1: How deep is the nesting?

function App() {
  <Level1>          ← 1 level
    <Level2>        ← 2 levels
      <Level3>      ← 3 levels
        <Level4>    ← 4 levels
          <Level5>  ← 5 levels (...)

If 1-3 levels: Props are fine. Stop here. ✓

Step 2: How often does the state change?

Rarely (once per session or less):

  • Theme toggle (light/dark)
  • Language switch (EN/ES)
  • User login/logout
  • Feature flags

→ Use Context API. It's simple and works great for this. ✓

Frequently (multiple times per second):

  • Form inputs (onChange events)
  • Mouse position tracking
  • Animation progress
  • Real-time notifications

→ Use Zustand / Redux. You need selector-based subscriptions to avoid unnecessary re-renders. ✓

Step 3: The Complete Decision Matrix

┌─────────────────────────────────────────────────────────────────┐
│                     DECISION MATRIX                              │
├──────────────────────────┬──────────────┬──────────────────────┤
│ Nesting / Change Freq    │ Props        │ Context / Zustand    │
├──────────────────────────┼──────────────┼──────────────────────┤
│ 1-3 levels / Any freq    │ ✓ USE PROPS  │ ✗ Overkill           │
├──────────────────────────┼──────────────┼──────────────────────┤
│ 4+ levels / Rare changes │ ✗ Tedious    │ ✓ USE CONTEXT        │
├──────────────────────────┼──────────────┼──────────────────────┤
│ 4+ levels / Frequent     │ ✗ Tedious    │ ⚠ Context risky      │
│                          │              │ ✓ USE ZUSTAND       │
├──────────────────────────┼──────────────┼──────────────────────┤
│ Multiple stores needed   │ ✗ Not an opt │ ✓ Multiple contexts  │
│                          │              │ ✓ Multiple Zustand   │
└──────────────────────────┴──────────────┴──────────────────────┘

Real-World Example: Building a Dashboard

Scenario: Dashboard with themes, user auth, and real-time data.

1. Theme (light/dark)
   └─ Changes: once per session
   └─ Decision: Context API ✓

2. Language (EN/ES)
   └─ Changes: once per session
   └─ Decision: Context API ✓

3. Current user (logged in)
   └─ Changes: only on login/logout
   └─ Decision: Context API ✓

4. Live notifications (coming in real-time)
   └─ Changes: many times per second
   └─ Decision: Zustand (avoid Context re-render spam) ✓

5. Form input (user typing)
   └─ Changes: on every keystroke
   └─ Decision: Local state (useState) ✓
   └─ Reason: Data is scoped to that component only

6. Chart data (fetched once on load)
   └─ Changes: once when page loads
   └─ Decision: Local state (useState) is fine, or Zustand if shared ✓

Common Mistakes (Junior Devs Make These)

Mistake 1: Context for everything

// ❌ BAD
const AppContext = createContext()

// Put ALL state in one Context:
// - theme (rarely changes)
// - form inputs (change constantly)
// - notifications (arrive rapidly)

// Result: Every keystroke causes everything to re-render

Better approach:

// ✓ GOOD
const ThemeContext = createContext()  // Rare changes
const NotificationStore = Zustand()   // Frequent changes
const formState = useState()           // Local to component

Mistake 2: Props for everything

// ❌ BAD - Prop drilling through 10 levels
<App user={user}>
  <Layout user={user}>
    <Sidebar user={user}>
      <Menu user={user}>
        <UserAvatar user={user}>
          ...nested 5 more levels
        </UserAvatar>
      </Menu>
    </Sidebar>
  </Layout>
</App>

// It's hard to refactor and hard to understand

Better approach:

// ✓ GOOD - Use Context for global state
<UserProvider>
  <App />
</UserProvider>

// Then any component can useUser() without prop chains

Mistake 3: Zustand for simple state

// ❌ OVERKILL
const useFormStore = Zustand((set) => ({
  email: '',
  password: '',
  setEmail: (e) => set({ email: e }),
  setPassword: (p) => set({ password: p })
}))

// This form is used in only ONE component
// useState is simpler and faster

Better approach:

// ✓ GOOD - Keep it local
function LoginForm() {
  const [email, setEmail] = useState('')
  const [password, setPassword] = useState('')

  return (
    <form>
      <input value={email} onChange={(e) => setEmail(e.target.value)} />
      <input value={password} onChange={(e) => setPassword(e.target.value)} />
    </form>
  )
}

Interview Question: "What would you use here?"

Scenario 1: A dashboard that shows a user's profile in the top-right corner.

App
 ├── Header (displays user name in top-right)
 ├── Sidebar
 └── MainContent
      ├── Page1
      ├── Page2
      └── Page3

App has the user data, Header needs it.
Nesting: 1 level (App → Header)

Answer: "I'd use props. It's shallow, clear, and performant."

Scenario 2: A multi-level menu where every item needs the current language (EN/ES).

App
 └── Layout
      └── Sidebar
           └── Menu (level 1)
                └── Submenu (level 2)
                     └── Submenu (level 3)
                          └── MenuItem (level 4)
                               └── ItemLabel (needs language)

Nesting: 5 levels. Language changes once per session.

Answer: "I'd use Context. It's deep, language is infrequent,
and we don't want to pass language through every component."

Scenario 3: A chat app with incoming messages (1 per second), form inputs, and a theme.

For messages: Zustand (frequent updates, selector subscriptions)
For theme: Context (rare updates)
For form input: useState (local only)

Answer: "I'd separate concerns. Messages and theme have
different update patterns, so they need different solutions."
